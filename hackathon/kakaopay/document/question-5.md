## Q1. 무엇을, 누가, 어떤 상황에서 쓰나요?

**무엇을** — 초보 투자자가 코스피·코스닥 투자에서 겪는 매수·매도 불안과 정보 부족을, "감정 인정 → 데이터 사실확인 → 규칙 상기 → 다음 행동 1개"의 설득 구조로 풀어주는 동행 플러그인입니다. 시세·공시·재무·뉴스·거시 5종 실데이터를 읽기 전용 MCP 로 조회하고, 스킬 9종(관리자 하네스 1 + 서브 8)이 상황별 프로토콜로 응대합니다.

핵심 설계는 "종목을 찍어주는 AI"가 아니라 **"지금 이 결정에서 확인하지 않은 것을 짚어주는 AI"** 입니다. 추천을 요구받으면 거절하는 대신, 관심사→재무·악재 스크리닝→후보 비교표→규칙 기준가(매수 밴드·손절·익절)라는 **본인이 재현할 수 있는 절차**로 바꿔 응대합니다.

**누가 · 어떤 상황에서**

- **사용자는 주식을 시작했거나 시작하려는 초보 투자자입니다.** 카카오페이증권 AI 서비스센터가 정의한 문제 상황 그대로 — "떨어지는데 팔아야 하나요?"(불안), "이 뉴스가 악재인가요?"(정보 부족), "뭘 사야 할지 모르겠어요"(결정 마비), "왜 자꾸 잃지?"(반복 실수) — 에서 씁니다.
- 단발 질문은 해당 스킬이 즉답하고, "투자 사이클 시작해줘" 이후에는 관리자 하네스(`invest-cycle`)가 상태 파일 기반으로 **선별→주문계획(dry run)→4주 추적→리포트→회고** 루틴을 세션이 바뀌어도 이어서 운영합니다. 새 세션에서 "지금 뭐 해야 해?" 한마디면 지난 진행 상황을 읽고 다음 한 걸음을 제시합니다.
- 산출물은 대화 응답과 함께 `outputs/result-<날짜>.md` 리포트, `outputs/cycle/` 상태·포트폴리오 대장으로 남습니다.

## Q2. 왜 이 문제를 선택했나요?

1. **출제 기업이 문제를 이미 정의해 줬습니다.** 카카오페이증권 AI 서비스센터장 인터뷰의 핵심은 "초보 투자자들이 매수·매도 시 겪는 불안감이나 정보 부족을 AI 로 해결하고, 안심하고 다음 단계를 실행하도록 돕는 것", 그리고 "사용자를 어떻게 납득시키고 안심시키는지의 논리와 설득 과정"이었습니다. 이 플러그인의 4단계 응대 구조와 신뢰 전달 형식(결론→근거→반대근거→한계·고지)은 그 문장을 그대로 설계로 옮긴 것입니다.
2. **초보 투자자의 진짜 문제는 판단 한 번이 아니라 루틴의 부재라고 봤습니다.** 추격매수, 물타기 가속, 처분효과(이익은 조기 실현, 손실은 방치)는 전부 "규칙 없이 감정으로 반복"할 때 생기는 패턴입니다. 그래서 좋은 답변기 8종에서 멈추지 않고, 그것들을 하나의 관리되는 사이클로 묶는 하네스까지 갔습니다. 사용자가 매주 돌아올 이유가 있는 구조가 증권사 서비스로서도 의미 있다고 판단했습니다.
3. **"확률 90% 종목 추천" 같은 요구를 정직하게 다루는 것 자체가 이 도메인의 승부처입니다.** 그런 숫자는 만들 수 없고(통계적으로), 만들면 안 되며(금소법상 단정적 판단 제공), 검증할 수도 없습니다. 대신 KRX 대형주 66종목·10년 실시세 백테스트로 실측 승률(월 51.2% / 분기 53.6% / 연 55.7%, 최악 5% 구간 연 -42.1%)을 계산해 동봉하고, 모든 추천성 응답이 이 기대치를 병기하게 했습니다. 지어낸 확률 대신 검증 가능한 통계 — 이것이 금융 도메인에서 AI 가 신뢰를 얻는 방식이라고 생각합니다.

## Q3. 플러그인은 어떻게 작동하나요?

**스킬 라우팅** — 발화 유형별로 서브 스킬이 발동하고, 모든 판정 출력은 `trust-explainer` 4단 구조(기준 충족형 결론 → 출처·시점 병기 근거 → 반대 근거 의무 1개+ → 한계·고지)를 거칩니다.

```text
불안 호소 ─────────► anxiety-triage ──┬─ 손실공포 → sell-companion
                                      ├─ FOMO    → buy-companion
"뭐부터 하지?" ────► next-step-guide  ├─ 정보부족 → 뉴스/공시/거시 사실 확인
"뭘 사야 할지…" ──► stock-explorer ──► (후보 좁히기 + 기준가 표) → buy-companion
"3종목이랑 가격" ──► periodic-picks (실측승률 → 규칙 5종 스크리닝 → 밴드/손절/익절)
"내 매매 봐줘" ────► trade-retrospective (행동 패턴 탐지 → 규칙 1~2개 제안)

[하네스] "사이클 시작 / 주간 점검 / 지금 뭐 해?" ► invest-cycle (관리자)
  outputs/cycle/state.json 이 단일 진실 —
  P1 선별(periodic-picks 위임) → P2 dry run → P3 4주 추적 → P4 리포트
  → P5 회고(trade-retrospective 위임) → 포트폴리오 대장 갱신 → 다음 사이클
```

**데이터 축 (읽기 전용 MCP 4서버 19도구)**

| 서버 | 답하는 질문 | 비고 |
|---|---|---|
| 시세 (price) | 지금 얼마인가, 연속 상승 중인가(추격매수 판정), 얼마에 사고팔까(`plan_trade`), 과거 승률은(`backtest_stats`) | 키 불필요. 낡은 심볼 방어(14일 신선도 검증, 낡으면 계산 거부) |
| 공시·재무 (DART) | 흑자인가, 유상증자·감사의견 이슈가 있나 | `dart_corp_search` 의 종목코드가 시세·뉴스와 조인 키 |
| 뉴스 (네이버) | 하락 이유가 실제 보도된 악재인가 — **없으면 없다고 말할 근거** | 본문 미수집(제목·요약·링크만), 악재 키워드 팬아웃 |
| 거시 (ECOS) | 시장 전체 요인인가 종목 요인인가 | 기준금리·국고채·환율·CPI |

**어댑터 경계** — 시세는 데모용 공개 어댑터(Yahoo, 지연 시세 명시)이며, 실서비스 전환 시 `price/client.mjs` 한 파일만 증권사 사내 시세 API 로 교체하면 도구 계약과 스킬이 그대로 유지됩니다. 하네스의 상태 파일(`state.json`/`portfolio.json`) 자리도 같은 원리로 사용자 계정 DB 가 들어가는 지점입니다.

**컴플라이언스 가드** — 지시형("사라/팔아라")·확률/목표수익 수치 생성·레버리지 제안 금지, 매매 계획의 가격은 "예측이 아니라 규칙"임을 매 응답 명시, 응답 말미 필수 고지문. 인용 가능한 유일한 확률 숫자는 재계산 가능한 백테스트 실측치뿐입니다.

## Q4. AI를 어떻게 활용했나요?

**1. 런타임: 결정론 계산과 LLM 종합의 분업**

숫자는 도구가 만들고, 설득은 모델이 만듭니다. 승률·매매 기준가·연속상승일수·재무 수치는 전부 결정론 코드(백테스트 엔진, `plan_trade` 의 KRX 호가단위 계산, streak 판정)가 산출하고, LLM 은 그 수치들을 초보자의 언어로 — 감정을 먼저 인정하고, 반대 근거를 의무적으로 붙이고, 다음 행동을 하나로 좁혀서 — 종합합니다. LLM 이 확률이나 목표가를 지어낼 수 있는 자리를 스킬 규칙과 도구 설계로 원천 제거한 것이 이 분업의 핵심입니다.

**2. 빌드: E2E 가 버그를 찾는 개발 루프**

Codex/Claude 로 스킬·MCP 서버·백테스트 엔진·테스트를 구현하되, 매 라운드를 "실세션 E2E → 발견 → 수정 → 재검증"으로 돌렸습니다. 실제로 E2E 가 잡아낸 것들입니다.

- 플러그인 MCP 가 공식 문서대로 하면 조용히 전멸하는 문제 → 디버그 로그·프로브 서버로 원인 실측(래퍼 형식, `${CLAUDE_PLUGIN_ROOT}` 미치환, env 미상속, 비대화 승인 정책, read-only 샌드박스). 문서와 다른 실측 규격 6건을 표로 정리해 두었습니다(저장소 `codex.md` §4).
- 코스닥 종목이 Yahoo 의 낡은 `.KS` 잔재 심볼에 걸려 2024년 가격이 나오던 버그 → 신선도 검증 + 낡은 시세로는 매매 계획 계산 거부.
- "지금 뭐 해야 해?"가 하네스 대신 next-step-guide 로 새던 트리거 충돌 → 상태 파일 존재 시 하네스로 합류하는 본문 상호 가드.

AI 로 만들고 AI 실세션으로 검증하며, 사람은 판정 기준과 컴플라이언스 경계를 설계하는 역할 분담이었습니다.

## Q5. 어떻게 검증했나요?

**1. 자동 테스트 7스위트** — 단위 3(공용 유틸 / 백테스트 코어: 승률·모멘텀·백분위 산술 / 매매계획: 호가단위 경계·분할 산술·예산부족·레벨 모드) + MCP 스모크 4(DART·ECOS·뉴스·시세 — 프로토콜 왕복과 키 존재 시 라이브 호출까지). 한 명령으로 재현됩니다.

```bash
node src/test/run-all.mjs          # 7스위트 — ALL GREEN 이어야 정상
node src/bin/backtest.mjs          # 승률 통계 재계산 (약 1~2분, 방법론·한계 포함 JSON 산출)
```

**2. 설치 E2E** — 원큐 스크립트(`src/bin/install-codex.ps1`: 마켓플레이스 등록→설치→MCP 승인 병합→키 스캐폴드→스모크)를 실제 codex-cli 0.142.5 에서 멱등 실행으로 검증했습니다.

**3. Codex 실세션 E2E — 대화가 곧 테스트** ([`logs/e2e-*.out.md`](./logs) 전부 동봉)

| 시나리오 | 확인한 것 |
|---|---|
| "삼성전자 5% 빠졌는데 팔아야 하나요? 무서워요" | MCP 4서버 12도구 실호출, 감정인정→사실→원칙/감정 분리→다음 행동 1개 프로토콜 준수 |
| 분기 / 월 / 연 3종목+가격 (3회) | 규칙 스크리닝(도구 42~82회), 실측 승률 병기, 예산 현실성(1주 218만원 종목 자동 제외), 3분할·손절·익절 가격표 |
| "주식 추천해줘" (막연 요청, 2회) | 종목 지목 대신 후보+근거+반대근거, 확률 수치 0개, 고지문 |
| 매매 내역 복기 (정답지 내장) | 합성 데이터에 심어둔 3개 패턴(추격매수 -801,500원 / 물타기 75.5% 집중·평단 17,617 교차검증 / 처분효과)을 수치 그대로 재현 |
| 사이클 시작 → **완전히 새로운 세션**에서 "지금 뭐 해야 해?" | 상태 파일 생성 → 새 세션이 state 를 읽고 P1 수행, watchlist 3종 기록, P2 로 전이 — 하네스의 세션 간 영속 증명 |

**4. 정답지 설계** — 복기 데모의 매매 내역은 개별 거래 한 건만 봐서는 안 보이고 시간순 재구성과 두 파일 교차 대조에서만 드러나도록 패턴을 심었습니다(보유현황의 평단 17,617원이 매매내역 4회 매수에서 재계산되는지까지). "문제없음을 확인한 것도 산출물"이라는 원칙으로, 해당 없는 패턴은 점검했으나 미발견으로 보고하게 했습니다.

**5. 표현 안전성** — 모든 스킬에 단정·보장 표현 금지, 기준 충족형 서술 강제, 필수 고지문이 명시돼 있고, E2E 산출물 전건에서 준수를 확인했습니다. 백테스트 통계에는 생존 편향·배당 제외·거래비용 미반영 한계를 결과 JSON 과 응답 양쪽에 병기합니다.
