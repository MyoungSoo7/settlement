# Musinsa Fashion First — 해커톤 제출 패키지 (AX 인재전쟁 — 무신사)

**"유저가 AI 에이전트에게 옷에 대해 질문할 때, 그 에이전트가 가장 먼저 탐색하는
도구가 무신사가 되기를 지향한다"** — 무신사 전사 테크 리드의 10년 비전.

이 플러그인은 그 비전의 **에이전트 인터페이스 프로토타입**이다. AI 에이전트(Codex) 안에서
패션 질문이 발생하는 순간, 브랜드 검증·가격 판정·트렌드 시계열·옷장 매칭을 수행하는
"첫 번째 패션 도구"로 동작한다. 데모는 공개 데이터로 돌고, 실서비스에서는 어댑터 한 파일
교체로 무신사 내부 API 에 연결된다 (아래 [프로덕션 경로](#프로덕션-경로--무신사-원코어와의-접점)).

## 가설 → 해결책 수미상관

**가설**: 패션 브랜드에 관심 있는 고객의 이탈·후회는 취향 부족이 아니라
**검증 절차의 부재**에서 온다. 마찰은 정확히 3개다.

| # | 문제 (가설) | 이 마찰의 실체 | 해결 스킬 | 데이터 근거 (MCP) |
|---|---|---|---|---|
| 1 | **브랜드 과부하** | 브랜드는 수천 개인데 "뭘 하는 브랜드고 믿을 만한지" 확인할 방법이 없다 | `brand-scout` (브랜드 카드), `trend-check` (유행 수명) | `brand_snapshot` 상품·가격대 실체, `brand_risk_scan` 평판 보도, `buzz_trend` 언급량 시계열 |
| 2 | **가격 불신** | "70% 할인"이 정가 뻥튀기인지 알 수 없다 — 할인율은 판매자의 언어 | `price-navigator` (분포 위치 번역) | `price_band` 시장 가격 분포 + 기준가 퍼센타일 |
| 3 | **실패 구매** | 사놓고 안 입는 옷(충동·중복), 사이즈 반품 루프, 배분 없는 예산 소진 | `style-match` (3-코디 규칙), `size-safety` (실측 대조), `closet-retrospective` (패턴 복기), `budget-shopper` (예산 배분×브랜드 검증) | 옷장·구매 내역 교차 분석 (로컬 데이터) + 위 MCP 오케스트레이션 |

모든 판정은 `honest-stylist` 의 신뢰 구조 — **결론 → 근거(수치·출처·시점) →
반대 근거·한계 → 다음 단계 1개** — 로 전달된다. 과장("무조건 예뻐요")이 기본값인
도메인에서 정직한 판정 구조 자체가 차별화다.

**검증 가능한 설계**: 가설이 맞다면, 플러그인은 (a) 처음 듣는 브랜드를 데이터 카드로
검증해내고, (b) 세일가의 실체를 분포 위치로 판정하고, (c) 옷장 데이터에서 실패 구매
패턴을 스스로 찾아내야 한다. (a)(b)는 라이브 API 스모크로, (c)는 **정답지를 심어둔
합성 옷장 데이터**(`src/data/sample/README-data.md`)로 재현 가능하게 채점된다.

## 비즈니스 임팩트 (무신사 관점)

- **반품률 감소**: `size-safety` 는 "브랜드 표기 사이즈는 브랜드마다 다른 언어, 믿을 것은
  cm 실측" 원칙으로 사이즈 실패(반품 사유 1위)를 구매 전에 차단한다. 반품 1건의 왕복
  물류비가 그대로 마진이다.
- **브랜드 발굴의 고객판**: 무신사가 AI 로 잠재력 있는 브랜드를 발굴하듯(무신사의
  "생존의 문제"), `brand-scout`+`trend-check` 는 고객이 신진 브랜드를 **신뢰 근거와 함께**
  발견하게 한다 — 롱테일 브랜드 노출·전환의 고객측 인프라.
- **신뢰 = 재방문**: 판매자 언어(할인율·품절임박)가 아니라 구매자 언어(분포 위치·착용당
  비용)로 말하는 커머스 도구는 그 자체로 락인이다. "입게 될 옷"만 사게 하는 3-코디
  규칙은 단기 객단가보다 장기 LTV·만족도에 베팅한다.

## 스킬 구성 (8종 + 관리자 하네스 1종)

| 스킬 | 역할 |
|---|---|
| `brand-scout` | 처음 듣는 브랜드의 실체(카테고리×가격대)·평판(리스크 보도)·흐름(버즈)을 브랜드 카드로 |
| `trend-check` | "지금 사면 한 철인가" — 언급량 시계열로 유행 수명 5분류 (상승 초입/본격/정점/식는 중/유보) |
| `price-navigator` | 할인율 대신 시장 분포 위치로 세일 실체 판정 + 정가 앵커 감지 |
| `budget-shopper` | "예산 30만원, 뭐부터?" — 버퍼 확보 → 빈 고리 우선 배분(조합 수 산술) → 브랜드 검증 → 시장가 대조 → 3-코디 통과 아이템만 담는 쇼핑 플랜 |
| `style-match` | 3-코디 규칙 — 가진 옷과 3가지 조합이 안 떠오르면 보류. 옷장 빈 고리 분석 |
| `size-safety` | 브랜드별 핏 시그니처 + 기준 아이템 실측 대조로 반품 방지 |
| `closet-retrospective` | 옷장·구매 내역에서 반복 행동 패턴 5종 탐지 → 다음 구매 규칙 1~2개 제안 |
| `honest-stylist` | 모든 판정에 결론→근거→반대근거→다음 단계 1개 구조 강제, 바디셰이밍·FOMO 화법 금지 |
| `closet-ops` ★ | **주간 옷장 운영 하네스(관리자)** — 위 8개를 sub-skill 로 호출·검수하는 유일한 오케스트레이터 (아래 절 참조) |

### closet-ops — 주간 운영 하네스

단건 Q&A 를 넘어 옷장을 **주간 사이클로 운영**하는 관리자 스킬. 옷장을 자산 원장
(`wardrobe/` — closet.csv·rules.md·tracking.csv·cycle-log.md)으로 유지하며,
매 사이클 단계마다 담당 sub-skill 을 호출하고 산출물을 검수한다:

```
Input(주간 상황) → [PLAN 플랜수립] ⇄ [VAULT 에셋 보관함] → [DRY-RUN 구매 전 게이트]
                → [TRACK 4주 착용 트래킹] → [REPORT 주간 리포트] → [RETRO 회고→규칙 갱신]
                → (52주 루프 — 회고가 다음 사이클의 입력)
```

- **PLAN** 은 style-match(빈 고리)·budget-shopper(배분)·brand-scout(후보 검증)를 호출하고,
  **DRY-RUN** 은 price-navigator·size-safety·리콜/평판·누적 규칙을 전 항목 게이트로 돌린다.
  구매 실행은 항상 사용자의 결재 — 하네스는 승인 근거까지만 만든다.
- **TRACK** 은 구매 후 4주 착용을 추적해 착용당 비용으로 성공/실패를 판정하고,
  **RETRO** 는 closet-retrospective 를 호출해 `rules.md` 를 갱신한다(규칙 5개 상한).
- 하네스의 성공 지표는 구매 횟수가 아니라 **착용당 비용의 하락**이다 — 매주 사게
  만드는 하네스는 실패한 하네스로 정의한다.

## 데이터 축 (읽기 전용 MCP 4종, 17개 도구)

- **상품·가격** (`fashion-first-shop`): 쇼핑 검색·가격 분포 통계·브랜드 스냅샷.
  "이 세일가가 시장 분포의 몇 %인지" 판정의 근거.
- **트렌드** (`fashion-first-trend`): 월별 언급량 시계열(최근3 vs 직전3 방향 판정),
  키워드 비교 랭킹. 수집 커버리지 한계를 결과에 동봉(정직한 데이터).
  네이버 데이터랩 권한 등록 시 쇼핑인사이트 **실클릭 시계열**(`datalab_shopping_trend`)이
  코드 변경 없이 자동 활성 — 두 상태(미등록 안내 에러 / 등록 후 실데이터) 모두 실검증.
- **뉴스·평판** (`fashion-first-news`): 브랜드 리스크 스캔(가품·리콜·불매·논란·소송·폐업),
  기회 스캔(콜라보·팝업·한정판·입점). 본문 미수집(제목·요약·링크만 — 저작권).
  "리스크 보도 없음"도 유효한 결과. 여기에 **공정거래위원회 소비자24 공식 리콜 DB**
  (`brand_recall_check`·`recall_latest` — 화장품·공산품 등 6품목)를 더해 뉴스 프록시를
  정부 처분 기록으로 격상. `CONSUMER24_API_KEY` (없으면 `DATA_GO_KR_API_KEY` 폴백 —
  키 미등록이면 발급 경로를 안내하는 구분 가능한 에러로 강등, 데이터랩과 동일 패턴).
- **날씨** (`fashion-first-weather`): 기상청 단기예보 — 현재 실황 + 3일 코디 브리핑
  (일별 최저/최고기온·강수확률·기온 밴드별 옷차림 가이드, 일교차 8도+ 레이어드 경고,
  강수 60%+ 우천 대비 플래그). "이번 주말 뭐 입지"의 근거. `DATA_GO_KR_API_KEY`.

### API 키 발급·활용신청 안내

키가 없으면 해당 축만 우아하게 강등되고(스킬이 사용자에게 사실을 직접 질문),
옷장 복기 데모와 closet-ops 하네스는 키 없이 완전 동작한다.

| env 키 | 발급처 | 신청할 것 | 담당 축 (없으면 강등) |
|---|---|---|---|
| `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET` | [developers.naver.com](https://developers.naver.com) | 애플리케이션 등록 → **"검색"** API 추가 (즉시 발급) | 상품·가격 + 트렌드 + 뉴스·평판 |
| 〃 (같은 앱) | 〃 | (선택) 같은 앱에 **"데이터랩"** API 추가 등록 | `datalab_shopping_trend` 실클릭 시계열 — 미등록이어도 뉴스 버즈 백엔드로 동작 |
| `DATA_GO_KR_API_KEY` | [공공데이터포털 data.go.kr](https://www.data.go.kr) | **"기상청_단기예보 ((구)_동네예보) 조회서비스"** 활용신청 (자동승인) → 마이페이지 일반 인증키 | 날씨×코디 |
| `CONSUMER24_API_KEY` | [소비자24 consumer.go.kr](https://www.consumer.go.kr) | 회원가입·로그인 → **Open API 활용신청** ("[리콜정보] 화장품"·"[리콜정보] 공산품") → 인증번호 발급 | 공식 리콜 DB (`brand_recall_check`·`recall_latest`) |

> 리콜 API 주의: data.go.kr 에도 "공정거래위원회_화장품/공산품 리콜정보"가 검색되지만
> **LINK 형**이라 data.go.kr 활용신청·공용 키만으로는 호출이 거부될 수 있다(코드 50 — 실측).
> 실제 인증번호는 소비자24에서 직접 발급받아 `CONSUMER24_API_KEY` 로 설정하는 것이 확실하며,
> 플러그인은 이 키를 우선 사용하고 없으면 `DATA_GO_KR_API_KEY` 로 폴백한다.

모든 키는 `.env.example` 을 `~/.codex/.env` 로 복사해 채우면 된다 (발급처·부재 시 동작이
파일 안에도 적혀 있다).

```text

|   |-- skills/                          # 스킬 8종 (위 표)
|   |-- mcp/{shop,trend,news,weather}-server.mjs # 읽기 전용 MCP 서버 (zero-dependency stdio)
|   |-- naver/ kma/ common/              # API 클라이언트 + 공용 모듈 (프로덕션 교체 지점)
|   |-- data/sample/                     # 합성 옷장·구매 내역 (정답지 내장)
|   |-- test/                            # 데이터 정합 + MCP 스모크 4종 + 단일 러너
|   |-- bin/install-codex.ps1            # 원큐 설치 (등록→설치→승인→키→스모크)
|   `-- bin/run-sample.ps1               # 데모 부트스트랩
|-- README.md
`-- logs/                                # E2E 검증 증거
```

## 설치 (Codex) — 원큐

```powershell
powershell -ExecutionPolicy Bypass -File src/bin/install-codex.ps1
```

마켓플레이스 등록 → 플러그인 설치 → MCP 승인 config 병합 → `~/.codex/.env` 스캐폴드 →
스모크 검증까지 한 번에 (멱등 — 재실행 안전).

## E2E 검증 증거 (codex 실세션, 2026-07-10)

3개에서 시나리오별 실검증**을 마쳤다 — 원문과 도구 호출 로그는 `logs/` 에 동봉:

| 시나리오 | 실호출된 MCP 도구 | 결과 (로그) |

## 프로덕션 경로 — 무신사 원코어와의 접점

- 데모의 데이터 축은 공개 유통 데이터(네이버 쇼핑·뉴스)다. 실서비스에서는
  `src/naver/{shop,trend,news}.mjs` **어댑터 3파일만** 무신사 내부 API(상품 검색,
  스타일/스냅 트렌드, 리뷰·브랜드 데이터)로 교체하면 12개 도구 계약과 스킬 7종이
  그대로 동작한다.
- 무신사가 드라이브 중인 **원코어 멀티플랫폼**과 정합: 도구 계약(MCP)을 플랫폼 공용
  코어로 두면 무신사·29CM·솔드아웃 어느 채널의 카탈로그든 같은 에이전트 인터페이스로
  서빙된다 — "AI 에이전트가 가장 먼저 탐색하는 첫 번째 도구"의 기술적 실체.
- 옷장 데이터는 데모에선 CSV, 실서비스에선 구매 이력·위시리스트가 그 자리에 들어간다
  (반품 사유·착용 프록시는 리뷰/재구매 신호로 대체 가능).
- **버티컬 일반화**: "예산 배분 → 브랜드 검증 → 시장가 대조 → 조합 검증"(budget-shopper)의
  절차는 도메인 불문이다 — MCP 어댑터만 바꾸면 식품·리빙·뷰티 등 다른 커머스 버티컬에
  같은 스킬 계약으로 이식된다. 무신사 뷰티 확장과도 같은 결로 맞닿는 구조.

조회 시점을 명시한다. 기사 본문은 수집하지 않으며(제목·요약·링크만), 본 패키지의 출력은
정보 제공이며 특정 브랜드·판매처와 이해관계가 없다.
