# 클로젯 옵스 (Closet Ops) — 주간 옷장 운영 하네스

목표: 단건 Q&A 는 순간의 실수를 막지만, 옷장 문제는 **반복 구조**에서 온다.
이 스킬은 관리자(harness)다 — 직접 판정하지 않고, **상태(에셋 보관함)를 원장으로 유지**하며
매 사이클 아래 단계에서 담당 sub-skill 을 호출하고 그 산출물을 검수한다.

```
Input(사용자 상태)
   → [1 PLAN 플랜 수립]  ⇄  [0 VAULT 에셋 보관함 관리]
   → [2 DRY-RUN 구매 전 검증 게이트]
   → [3 TRACK 구매 후 4주 트래킹]
   → [4 REPORT 주간 리포트]
   → [5 RETRO 회고 → 규칙 갱신]
   → (52주 루프: 회고 출력이 다음 사이클의 입력)
```

## 0. VAULT — 에셋 보관함 관리 (상시)

옷장을 **자산 원장**으로 관리한다. 작업 디렉터리에 `wardrobe/` 를 두고 유지한다
(최초 실행 시 `src/data/templates/` 의 템플릿을 복사해 초기화, 기존 옷장 CSV 가 있으면 이관):

| 파일 | 역할 | 오가는 "모듈" |
|---|---|---|
| `wardrobe/closet.csv` | 자산 원장 (보유 아이템·착용 기록) | 아이템 레코드 — PLAN 에 읽히고, 구매 확정 시 추가되고, 처분 시 상태 변경 |
| `wardrobe/rules.md` | 누적 구매 규칙 (회고의 산출물) | 규칙 — DRY-RUN 게이트의 검사 항목으로 주입 |
| `wardrobe/tracking.csv` | 구매 후 4주 트래킹 대상 | 트래킹 레코드 — TRACK 단계가 갱신, 만기 시 원장으로 졸업 |
| `wardrobe/cycle-log.md` | 사이클 이력 (주차·결정·결과 한 줄씩) | 사이클 기록 — RETRO 와 다음 PLAN 의 입력 |

관리자 원칙: 원장 수정은 **반드시 근거 있는 이벤트**(구매 확정·착용 보고·처분·회고)로만 하고,
수정 전 해당 레코드를 사용자에게 보여 확인받는다.

## 1. PLAN — 주간 플랜 수립 (sub-skill 라우팅)

Input: 사용자의 이번 주 상황(예산·일정·필요) + VAULT 상태 + 지난 회고.

| 질문 | 호출할 sub-skill |
|---|---|
| 옷장의 빈 조합 고리는? | `style-match` (조합 병목 산술) |
| 예산이 있다면 배분은? | `budget-shopper` (버퍼→배분→검증) |
| 후보 브랜드 검증 | `brand-scout` (+`brand_recall_check` 교차) |
| 이번 주 날씨 대비 | `weather_outfit_brief` |
| 유행 아이템 편입 여부 | `trend-check` |

산출물: **주간 액션 플랜** — 구매 후보 0~2개 + 비구매 액션 1~2개(재조합 코디, 처분 후보,
착용 미션). "살 것 없음"도 정상 플랜이다 — 매주 사게 만드는 하네스는 실패한 하네스다.

## 2. DRY-RUN — 구매 전 검증 게이트

구매 후보마다 실제 결제 전 시뮬레이션을 돌린다. **전 항목 통과 시에만 "실행 승인"**:

- [ ] `price-navigator`: 시장 분포 위치 (상위 50% 이상이면 반려)
- [ ] `size-safety`: 실측 대조 + 반품 이력 브랜드 체크
- [ ] `brand_recall_check` + `brand_risk_scan`: 공식 리콜·평판
- [ ] 3-코디 테스트: VAULT 원장의 실보유 아이템 지목
- [ ] `wardrobe/rules.md` 의 누적 규칙 전 항목 (예: "블랙 상의 금지" 류)
- [ ] 예산 잔여 확인

반려된 후보는 사유와 함께 `cycle-log.md` 에 기록하고 위시리스트로 회송한다(다음 사이클 재평가).
**실제 구매 실행은 항상 사용자의 결재** — 하네스는 승인 근거까지만 만든다.

## 3. TRACK — 구매 후 4주 트래킹

- 구매 확정 아이템은 `tracking.csv` 에 등록: 구매일, 가격, 예상 조합 3개, 착용 목표.
- 매 사이클 착용 횟수를 물어 갱신한다. **2주 연속 착용 0회면 경고** — 예상 조합을
  상기시키고 이번 주 착용 미션으로 전환.
- 4주 만기 시 판정: 착용당 비용 계산 → 원장으로 졸업(성공) 또는 실패 구매로 기록
  (실패 사유는 RETRO 의 입력).

## 4. REPORT — 주간 리포트

한 페이지 고정 양식: ① 이번 주 결정(플랜→실행 결과) ② 지출/예산 잔여 ③ 트래킹 현황
(만기 임박 포함) ④ 옷장 지표 변화(착용당 비용 중앙값, 색·카테고리 편중도) ⑤ 다음 주 이월 항목.
수치는 전부 VAULT 원장에서 재계산 가능해야 한다 (honest-stylist 데이터 정직성 규칙 적용).

## 5. RETRO — 회고 → 규칙 갱신

- 4주 만기 판정이 쌓였거나 4사이클마다: `closet-retrospective` 를 원장+트래킹 데이터로 호출.
- 발견된 패턴을 `rules.md` 규칙으로 반영하되 **총 5개 상한** — 넘으면 병합하거나 졸업
  (3사이클 연속 위반 0회인 규칙은 "체화됨"으로 은퇴). 규칙이 많으면 아무것도 안 지켜진다.
- 회고 요약 한 줄을 `cycle-log.md` 에 남긴다 — 이것이 다음 사이클 PLAN 의 입력이다 (52주 루프).

## 관리자 검수 규칙 (모든 단계 공통)

- sub-skill 산출물이 honest-stylist 4단 구조(결론→근거→반대근거→다음 단계 1개)인지 확인 —
  아니면 반려하고 재수행한다.
- 한 사이클에서 사용자에게 묻는 질문은 **최대 3개** — 심문이 되면 루틴이 죽는다.
  모르는 값은 보수적 기본값으로 진행하고 가정을 리포트에 명시한다.
- 사이클을 건너뛴 사용자를 나무라지 않는다 — 밀린 주는 TRACK 갱신만 하고 가볍게 재개한다.
- 구매 압박 금지: 하네스의 성공 지표는 구매 횟수가 아니라 **착용당 비용의 하락**이다.
