## Q1. 무엇을, 누가, 어떤 상황에서 쓰나요?

**무엇을** — 패션 소비의 판단 순간마다 "고객이 스스로 검증할 수 없는 것"을 대신 검증해 주는 Codex 플러그인입니다. 처음 듣는 브랜드의 실체·평판·흐름을 데이터 카드로 만들고, "70% 할인"을 시장 가격 분포에서의 위치로 번역하고, 유행이 한 철인지 시계열로 판정하고, 가진 옷과 3가지 조합이 되는지로 "입게 될 옷"만 사게 하고, 브랜드별 핏 시그니처로 사이즈 반품을 막고, 옷장·구매 내역에서 반복 실패 패턴을 복기합니다.

옷을 추천해서 던지는 도구가 아닙니다. 추천은 이미 넘칩니다 — 부족한 것은 그 추천을 믿어도 되는지 확인하는 절차이고, 이 플러그인은 그 절차를 자동화합니다.

**누가 · 어떤 상황에서**

- **사용자는 패션 브랜드에 관심이 있는 일반 소비자입니다.** 무신사에서 낯선 브랜드를 발견했을 때, 세일가 앞에서 망설일 때, 예산 30만원으로 뭘 먼저 살지 막막할 때, 두 사이즈 사이에서 고민할 때, "옷은 많은데 입을 게 없다"고 느낄 때 씁니다.
- 단건 질문을 넘어서면 관리자 하네스(`closet-ops`)가 옷장을 자산 원장으로 유지하며 주간 사이클(플랜 → 구매 전 검증 게이트 → 4주 착용 트래킹 → 리포트 → 회고)로 운영해 줍니다.
- 무신사 관점에서는 "유저가 AI 에이전트에게 옷을 물을 때 가장 먼저 탐색하는 도구가 무신사"라는 10년 비전의 에이전트 인터페이스 프로토타입입니다. 데모는 공개 데이터로 돌고, 어댑터 파일만 교체하면 무신사 내부 API 로 같은 도구 계약이 유지됩니다.

## Q2. 왜 이 문제를 선택했나요?

세 가지 이유가 맞물립니다.

1. **무신사의 과제를 고객 쪽에서 뒤집었습니다.** 무신사는 트렌드와 잠재력 있는 브랜드를 놓치지 않고 발굴하는 일을 "생존의 문제"로 정의합니다. 같은 문제가 고객 쪽에도 대칭으로 존재합니다 — 브랜드는 수천 개인데 고객에게는 검증 수단이 없습니다. 플랫폼의 브랜드 발굴이 공급이라면, 고객의 브랜드 검증은 수요입니다. 이 플러그인은 그 수요측 인프라입니다.
2. **가설과 해결책의 수미상관을 검증 가능하게 설계할 수 있는 문제였습니다.** 가설은 "패션 소비의 이탈·후회는 취향 부족이 아니라 검증 절차의 부재에서 온다"이고, 마찰을 정확히 3개로 좁혔습니다 — ① 브랜드 과부하 ② 가격 불신 ③ 실패 구매(충동·중복·사이즈). 각 마찰에 스킬과 데이터 축이 1:1 로 매핑되고, 가설이 맞다면 플러그인이 해내야 하는 일(브랜드 검증 카드, 분포 위치 판정, 패턴 재발견)을 전부 재현 가능한 테스트로 만들 수 있었습니다.
3. **비즈니스 임팩트가 플랫폼 지표로 직결됩니다.** 사이즈 실패는 반품률(왕복 물류비가 그대로 마진), 브랜드 검증은 롱테일 브랜드의 발견성과 전환, 정직한 판정 구조는 재방문과 LTV 로 이어집니다. 하네스의 성공 지표를 구매 횟수가 아니라 "착용당 비용의 하락"으로 정의한 것도 같은 이유입니다 — 단기 객단가보다 장기 신뢰에 베팅하는 것이 커머스 도구의 올바른 정렬이라고 판단했습니다.

## Q3. 플러그인은 어떻게 작동하나요?

스킬 9종(실무 8 + 관리자 하네스 1)이 읽기 전용 MCP 4서버 17도구를 근거 축으로 씁니다.

```text
고객 질문
  -> brand-scout          # 브랜드 실체(카테고리×가격대)·평판·흐름 → 브랜드 카드
  -> trend-check          # 언급량 시계열 + 데이터랩 실클릭 → 유행 수명 5분류
  -> price-navigator      # 시장 분포 percentile → 세일 실체 판정, 정가 앵커 감지
  -> budget-shopper       # 버퍼 확보 → 빈 고리 배분 → 브랜드 검증 → 예산 플랜
  -> style-match          # 3-코디 테스트 (+ 날짜 있으면 예보 반영)
  -> size-safety          # 기준 아이템 실측 대조, 반복 반품 브랜드 감지
  -> closet-retrospective # 옷장·구매 내역 행동 패턴 5종 탐지 → 규칙 1~2개
  == honest-stylist       # 모든 출력에 결론→근거→반대근거→다음 단계 1개 강제
  == closet-ops           # 주간 운영 하네스 — 위 스킬들을 사이클 단계별로 호출·검수
```

데이터 축은 네 개입니다. **상품·가격**(쇼핑 검색 — 가격 분포 통계·브랜드 스냅샷), **트렌드**(뉴스 언급량 히스토그램 + 데이터랩 쇼핑인사이트 실클릭 시계열), **뉴스·평판**(리스크/기회 스캔 + 공정거래위원회 소비자24 공식 리콜 DB), **날씨**(기상청 단기예보 — 기온 밴드별 옷차림 가이드). 모든 도구는 읽기 전용이고(`readOnlyHint` 명시), zero-dependency stdio 서버라 심사 환경에서 `npm install` 없이 Node 18+ 만으로 돕니다.

`closet-ops` 하네스는 옷장을 `wardrobe/` 원장 4파일(자산 원장·누적 규칙·트래킹·사이클 로그)로 유지하며, 구매 후보는 실제 결제 전에 6항목 dry-run 게이트(가격 분포·사이즈·리콜·3-코디·누적 규칙·예산)를 전부 통과해야 합니다. 구매 실행은 항상 사용자의 결재이고, 회고가 규칙을 갱신해(5개 상한) 다음 사이클의 입력이 되는 52주 루프 구조입니다.

프로덕션 경로도 설계에 포함했습니다. 데모의 데이터 축은 공개 API 지만, `src/naver/`·`src/kma/`·`src/recall/` 어댑터만 무신사 내부 API(상품 검색·스냅 트렌드·리뷰)로 교체하면 17개 도구 계약과 스킬 9종이 그대로 동작합니다. 도구 계약을 플랫폼 공용 코어로 두면 무신사·29CM·솔드아웃 어느 채널이든 같은 에이전트 인터페이스로 서빙된다는 점에서, 무신사가 드라이브 중인 원코어 멀티플랫폼과 같은 결의 구조입니다.

## Q4. AI를 어떻게 활용했나요?

두 층위에서 활용했습니다.

**1. 런타임: 기계 파생 신호 위에서만 추론하는 에이전트**

순수 규칙 매칭도, 근거 없는 LLM 감상평도 아닙니다. 판정의 원료는 전부 기계적으로 파생됩니다 — 가격은 표본 분포의 percentile 로, 트렌드는 최근 3개월/직전 3개월 비율의 방향 판정으로, 날씨는 격자 예보의 기온 밴드로, 옷장은 착용당 비용 재계산으로. 에이전트는 그 위에서 스킬이 강제하는 프로토콜에 따라 종합합니다.

특히 **데이터 정직성**을 스킬 차원에서 강제했습니다. 수집이 기간을 못 덮으면 `coverageComplete=false` 와 미상(null) 구분을 결과에 동봉하고, 언급량은 수요의 프록시일 뿐임을 명시하고, "뉴스 0건·리콜 이력 없음"도 유효한 결과로 보고합니다. 모든 판정은 반대 근거(이 판정이 틀릴 수 있는 이유)를 최소 1개 포함해야 하며, 바디셰이밍·구매 압박·근거 없는 수식어는 화법 금지 목록으로 차단했습니다. 과장이 기본값인 도메인에서 정직한 판정 구조 자체가 차별화라고 봤습니다.

**2. 빌드: 실측 주도 개발**

Claude Code 로 전체(스킬 문서, MCP 서버 4종, API 어댑터, 정답지 데이터, 테스트, 설치 스크립트, 제출 문서)를 제작하되, 문서보다 실측을 우선했습니다. 그 과정에서 문서만으로는 알 수 없는 사실들을 확정하고 설계에 반영했습니다 — Codex 플러그인 MCP 의 실동작 규격(`cwd:"."`+상대경로, env 미상속), 데이터랩 권한 미등록 시의 에러 코드와 권한 추가 시 무변경 자동 활성, 공공데이터포털 키 끝에 섞인 한글 오타 1자가 전 API 를 401 로 만든 사건(어댑터에 비ASCII 제거 방어 반영), 소비자24 리콜 API 의 자체 인증키 체계와 필수 파라미터(`cntntsId`), PowerShell 5.1 의 한글 스크립트 BOM 요구사항까지. 전부 재발 방지 코드 또는 안내 에러로 승격되어 있습니다.

## Q5. 어떻게 검증했나요?

정답지 기반 재현 검증, 라이브 스모크, 실세션 E2E, 강등 경로 검증을 겹쳐 썼습니다.

**1. 정답지를 심은 합성 데이터와 정합성 테스트**

옷장 16벌·구매 18건(반품 포함)의 합성 데이터에 행동 패턴 3종을 의도적으로 심었습니다. 브랜드는 전부 가상이며(실존 브랜드 평판 보호), 패턴은 한 줄만 봐서는 안 보이고 두 파일 교차와 시간순 재구성에서만 드러나도록 설계했습니다.

| # | 패턴 | 데이터 근거 | 기대 추론 |
|---|---|---|---|
| 1 | 세일 충동구매 → 미착용 | 할인 55%+ 4건(전부 세일알림 트리거) 161,000원, 착용 합계 2회 | 착용당 80,500원 vs 정가 대조군 1,553원 — 숙성·3-코디 규칙 |
| 2 | 같은 옷 또 사기 | 상의 12벌 중 블랙 7벌, 충동 구매 3건도 또 블랙 | 포화 카테고리 반복 투자 = "입을 게 없다"의 산술적 원인 |
| 3 | 사이즈 실패 루프 | 테일러독 L 3연속(반품 2회 "사이즈 작음" + 방치 1벌) | 브랜드 고유의 작은 핏 — 한 치수 업 규칙, 반품률 임팩트 |

`closet-data-test.mjs` 가 이 정답지의 모든 수치를 CSV 에서 재계산해 빌드 시점에 검증합니다(정답지 드리프트 방지). 검증 재구매(착용 31회)를 대조군으로 넣어 "재구매=문제"라는 과잉 일반화도 차단했습니다.

**2. 스모크 스위트 — 프로토콜과 라이브를 함께**

`node src/test/run-all.mjs` 하나로 5개 스위트(데이터 정합 + MCP 4서버)가 돕니다. 키가 없으면 MCP 프로토콜 검증만, 키가 있으면 실 API 라이브 검증까지 수행하며 어느 쪽이든 GREEN 이어야 합니다. 제출 시점 기준 **ALL SUITES GREEN** 입니다.

**3. Codex 실세션 E2E — 7개 시나리오**

설치 스크립트(마켓플레이스 등록 → 설치 → MCP 승인 → 키 스캐폴드 → 스모크, 멱등)로 실제 codex-cli 에 설치한 뒤, 새 스레드에서 시나리오별로 실행하고 원문과 도구 호출 로그를 `logs/` 에 동봉했습니다.

| 시나리오 | 확인된 것 |
|---|---|
| 브랜드 검증 (마르디 메크르디) | MCP 6도구 실호출, 실체→가품 리스크→상승 판정(언급량 496 vs 252)의 4단 구조 응답 |
| 세일 실체 (45,000원) | "하위 25%, 중앙값 대비 -13.3% — 살 만함" + 가품 유통 주의 + 사이즈 확인 연결 |
| 옷장 복기 (정답지) | 패턴 1을 수치까지 완전 재현(161,000원·2회·80,500원/회), 패턴 2·3 재현 + 하의 부족 보너스 인사이트 |
| 예산 플랜 (30만원) | 버퍼 4만원 확보 → 하의 병목 배분 → 실브랜드 2개 검증 후 담기, "안 입은 재고 있는 상의는 제외" |
| 날씨 코디 (주말 데이트) | 실예보(25~33도) 기반으로 옷장 아이템 ID 를 지목한 조합 + 낮/저녁 분기 |
| 트렌드 비교 (발레코어 vs 블록코어) | 데이터랩 실클릭 시계열과 언급량 버즈의 교차 검증, 수집 한계 명시 |
| 운영 하네스 (첫 사이클) | 원장 초기화(이관을 SHA256 으로 자가 검증), 첫 플랜이 "이번 주 구매 없음", CSV 만으로 규칙 4개 도출 |

특히 하네스 첫 사이클이 구매 권유가 아니라 "구매 없음 + 저착용 아이템 착용 미션"을 내놓은 것은, "매주 사게 만드는 하네스는 실패한 하네스"라는 설계 원칙이 실세션에서 작동함을 보여준 결과였습니다.

**4. 강등 경로의 양쪽 상태를 실검증**

키가 없거나 권한이 없을 때의 동작도 검증 대상입니다. 데이터랩은 권한 미등록 상태(구분 가능한 안내 에러 + 버즈 폴백)와 권한 등록 후(코드 변경 없이 실클릭 시계열 자동 활성)를 **모두 실세션에서** 확인했고, 소비자24 리콜 축은 키 미등록 상태의 발급 안내 에러까지 스모크로 고정했습니다. 옷장 복기 데모는 키가 하나도 없어도 완전 동작합니다.

**5. 설치·운영 경로의 실검증**

설치 스크립트는 재실행 멱등성(마켓플레이스 중복·승인 config 블록 단위 병합·env 보존)을 실제 반복 실행으로 확인했고, 그 과정에서 발견한 버그 2건 — 새 MCP 서버 추가 시 승인 블록이 병합되지 않던 문제, PowerShell 5.1 이 BOM 없는 한글 스크립트를 CP949 로 파싱해 깨지던 문제 — 을 수정해 반영했습니다. 검증이 패키지 자체를 개선하는 환류였습니다.

검증 기준은 일관됩니다: 수치의 재계산 가능성, 판정의 근거 정합성, 데이터 한계의 정직한 동봉, 강등 경로의 예측 가능성, 그리고 설치부터 응답까지 심사자가 그대로 재현할 수 있을 것.
