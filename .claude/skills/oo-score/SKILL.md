# oo-score — OO 설계 5축 패널 채점

## 2계층 구조 (이 스킬의 위치)

| 계층 | 무엇 | 어떻게 |
|---|---|---|
| 1. 결정적 게이트 | 9.5점을 만든 기계적 불변식의 회귀 차단 | `guard.mjs` OO-DOMAIN-* 3규칙(쓰기 시점·pre-commit·CI) + `scripts/harness/test/oo-gate.test.mjs`(트리 전수: setter 0·@Setter 0·IAE 0·애그리거트 봉인·전이표 존재) |
| 2. LLM 판정 (이 스킬) | 실제 5축 점수 산출 | 독립 리뷰어 3인 패널 → 축별 중앙값 ≥9.5 판정 |

점수는 LLM 판단이라 훅으로 강제 불가 — 훅은 구조 회귀를 막고, 이 스킬이 주기적으로 점수를 재측정한다.

## 언제 돌리나

- 대규모 도메인 리팩터링·신규 서비스 추가 후
- 분기 점검 / 포트폴리오 갱신 전
- 게이트(1계층)는 GREEN 인데 설계 품질이 의심될 때

## 실행 절차 (검증된 프로토콜 — 2026-07-14 캠페인 정본)

1. **패널 소집**: 읽기 전용 리뷰어 **3인을 병렬** 투입한다. 반드시 **서로 다른 에이전트 계열**로
   (예: `oh-my-claudecode:architect` / `oh-my-claudecode:critic` / `hexagonal-arch-reviewer`), 상호 비공지,
   개선 이력 비공지. 코드 수정 금지·서브에이전트 생성 금지 조항 포함.
2. **공통 루브릭** (각 리뷰어 프롬프트에 동일하게):
   - 대상: settlement/order/loan/investment/account 의 domain/ + application/ (워크트리 사본 `.claude/worktrees/**` 제외)
   - 5축 각각 소수점 1자리 소점수: ① 도메인 모델 풍부함(rich model·상태머신/불변식의 도메인 내 강제·전이 단일 출처)
     ② 캡슐화/불변성(setter·public 생성자 grep 실측·final·write-once) ③ SOLID(god class·정책 데이터화·포트 역할·DIP)
     ④ 응집도/결합도(Money VO 경계 일관성·정당 중복 비감점) ⑤ 예외 설계(도메인 vs 인프라 가드 구분·컨텍스트 보존·HTTP 매핑)
   - 공정성 조항: 실측 없는 추정 감점 금지 / MSA·트랜잭션 경계 필연 설계 비감점 / 주석 명문화된 설계 판단 비감점 /
     서비스 내부 일관 관례는 서비스 간 차이만으로 비감점 / write-once 가드의 IllegalStateException 은 정당 용법
   - **캘리브레이션 조항(필수)**: "실측 가능한 잔여 결함을 찾지 못한 축은 9.6~10.0 에서 채점 — 결함 없이 관성으로
     9.5 미만을 주는 것은 부정확한 채점" (리뷰어 천장 편향 교정. 이것 없이는 '결함 없음+9.3' 이 나온다)
   - 출력 형식 강제: 마지막 줄 `SCORES: [축1, 축2, 축3, 축4, 축5]`
3. **집계**: 축별 **중앙값**(3인). 평균 아님 — 중앙값이 리뷰어 편차(실측 ±0.5)를 흡수한다.
4. **판정**: 전 축 중앙값 ≥9.5 → PASS. 미달 축은 **3인 지적의 합집합에서 파일:라인이 있는 결함만** 수정 스토리로
   등록 후 재패널. 지적이 반복 순환하면(새 유형 없이) 수렴 완료로 보고.

## 함정 (캠페인에서 실측)

- 단일 리뷰어 반복은 라운드마다 다른 흠을 찾아 점수가 진동한다 — 반드시 패널.
- 리뷰어가 fork/서브에이전트를 만들거나 빌드를 돌리면 안 된다(읽기 전용 명시).
- 채점 대상에 `.claude/worktrees/**` 사본이 섞이면 중복 정의로 오감점.
