# TDD Discipline (실패를 목격하지 않은 테스트는 검증력이 없다)

## 철칙

**프로덕션 코드보다 실패하는 테스트가 먼저다.** 실패를 직접 목격하지 않았다면 그 테스트가
올바른 것을 검증하는지 알 수 없다. 구현이 먼저 나와 버렸다면 — 테스트를 쓴 뒤 구현을 잠시
되돌려서라도 RED 를 확인하고 복원한다.

## 순서 (CLAUDE.md 테스트 컨벤션과 결합)

**도메인 → 서비스 → 컨트롤러 → 통합** 순으로 짠다. 도메인은 순수 POJO 라 테스트가 가장 싸다 —
상태머신 전이·금액 계산·정책은 반드시 도메인 단위 테스트부터.

## RED — 실패 테스트 작성

최소 케이스 구성 (돈 경로는 필수):

- **대표값 1** + **경계값**(라운딩 `.5`·0원·1원·최대금액 — 📘`money-safety` 테스트 템플릿)
- **상태머신**은 정상 전이만이 아니라 **비정상 전이 차단**(`canTransitionTo` 거짓 경로) 케이스
- **멱등**이 걸린 경로는 동일 입력 2회 호출 케이스

RED 확인: 실패 사유가 **의도한 어서션**인지 본다. 컴파일 에러·NPE 로 실패하는 건 RED 가 아니다.

## GREEN — 최소 구현

테스트를 통과시키는 최소한만 구현한다. 다음은 게이트 우회이므로 금지:

- 어서션 완화 · `@Disabled` · 기대값을 실제값에 맞춰 수정(스냅샷 갱신식 사고)
- 커버리지 제외 설정 조작 (adapter 제외 범위는 부모 `build.gradle.kts` 가 정본 — 임의 확장 금지)

## 판정 — LLM 판단이 아니라 게이트가 정답

```bash
./gradlew :<module>:test :<module>:jacocoTestCoverageVerification   # LINE 90% / 핵심 도메인 INSTRUCTION 80%
```

통과 출력을 확인하기 전에는 "구현 완료"라고 쓰지 않는다(완료 선언 절차는 📘`verify-before-done`).

## 예외 (사용자 합의 필요)

일회성 스크립트·스파이크·설정 파일은 합의 하에 생략 가능. 단 **돈 경로(정산·지급·원장·대출·
투자·환불)는 예외 없음** — "간단한 변경이라 생략"은 합리화다.
