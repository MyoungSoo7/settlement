# Debugging Discipline (원인 규명 전 수정 금지)

## 철칙

**재현하고 원인을 규명하기 전에는 코드를 고치지 않는다.** "아마 이것 때문일 것" 수정은
증상을 옮길 뿐이고, 돈 경로에서는 회계 손상을 숨긴다. 이 절차는 rigid — 단계 생략 금지.

## 절차

1. **재현 최소화** — 실패를 재현하는 **가장 작은 테스트**를 확보한다(없으면 만든다).
   이 실패 테스트가 나중에 회귀 가드가 된다. 재현 불가면 수정도 불가 — 관찰 로그부터 늘린다.
2. **증거 수집** — 스택트레이스 전문, 실패 어서션의 기대/실제 값, 관련 로그를 그대로 확보한다.
   요약·기억으로 대체 금지. 게이트 출력(`./gradlew :<module>:test`)이 1차 증거다.
3. **가설은 한 번에 하나** — 가설마다 "참이면 X가 보이고, 거짓이면 Y가 보인다"를 **먼저 적고**
   확인한다. 확인 없이 다음 가설로 넘어가지 않는다.
4. **원인 확정 후 최소 수정** — 원인 문장("~여서 ~가 된다")을 한 줄로 쓸 수 있을 때만 수정한다.
   수정은 원인을 제거하는 최소 diff — 리팩터링을 끼워 넣지 않는다.
5. **회귀 확인** — 1의 실패 테스트 green + 해당 모듈 기존 스위트 green. 그다음 같은 원인이
   다른 곳에도 있는지 grep 으로 전수 확인한다.

## 금지 패턴 (증상 은폐)

- try-catch 로 예외 삼키기 · 어서션 완화 · `@Disabled`/테스트 skip · 타임아웃 늘리기
- 원인 모른 채 "재시도 추가" · 라운딩/금액 불일치를 tolerance 로 덮기
- 멱등 위반을 "중복이면 무시" 로 덮기 (원인은 3단 방어 어디가 뚫렸는지 — `idempotency-and-events`)

## 이 코드베이스의 1차 용의자 맵

| 증상 | 먼저 볼 곳 |
|---|---|
| 컨슈머가 이벤트를 못 받음 / 중복 처리 | 📘`idempotency-and-events` (3단 방어 순서) |
| settlement 에서 order 데이터 안 보임 | 📘`projection-view-ops` (뷰 드리프트·백필) |
| 계약 테스트 FAIL / 역직렬화 실패 | 📘`event-contract-change` |
| 코드는 있는데 404/빈 화면/기동 실패 | 📘`msa-service-wiring` (5곳 배선 누락) |
| 시산표 차대 불일치 | 📘`ledger-invariants` + 🤖`gl-ledger-auditor` |

## 중단 조건

가설 3개가 연속 기각되면 멈춘다 — 지금까지의 증거·기각된 가설을 정리해 사용자에게 보고하거나
`tracer`(증거 기반 인과 추적) 에이전트로 위임한다. 같은 실험 반복은 시간 낭비다.
