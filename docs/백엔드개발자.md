# 백엔드 개발자 — 기본 소양과 이 프로젝트의 백엔드 기술

이 문서는 두 부분이다. **Part 1**은 백엔드 개발자라면 갖춰야 할 _보편적 기본 소양_, **Part 2**는 그 소양이 이 프로젝트(Lemuel — 이커머스 + 정산 MSA)에서 _실제 코드로 어떻게 구현됐는지_ 매핑한다. 즉 "무엇을 알아야 하나"와 "여기선 어떻게 했나"를 짝지어 본다.

# Part 1. 백엔드 개발자가 알아야 할 기본 소양

## 1) 언어 · 런타임 (JVM)
- **메모리 구조**(힙/스택/메타스페이스), **GC** 동작과 튜닝 감각, 객체 수명.
- **동시성 모델**: 스레드, 메모리 가시성(`volatile`), 락, `java.util.concurrent`.
- 자바라면 컬렉션·제네릭·오토박싱 비용, 스트림, 레코드, 가상 스레드(Loom) 흐름.

## 2) 네트워크 · HTTP · API 설계
- HTTP 메서드/상태코드 정확한 사용, **멱등성**(GET/PUT/DELETE), `PATCH` vs `PUT`.
- REST 자원 모델링, 페이지네이션(offset vs 커서), 버저닝, 에러 표준화(RFC 7807).
- TLS, CORS, 인증 헤더, 타임아웃·재시도·백오프, **Idempotency-Key**.

## 3) 데이터베이스
- **ACID**, 트랜잭션 격리수준(Read Committed / Repeatable Read / Serializable)과 이상현상(dirty/non-repeatable/phantom).
- **인덱스 원리**(B+Tree), 실행계획(EXPLAIN), 풀스캔 회피, 복합 인덱스 선두 컬럼.
- 정규화 vs 비정규화, **N+1 문제**, 커넥션 풀, 락(낙관/비관), 데드락.

## 4) 동시성 · 정합성
- **낙관적 락**(`@Version`) vs **비관적 락**(`SELECT ... FOR UPDATE`), 적용 기준.
- **멱등성**: 재시도·중복 메시지에도 안전하게(유니크 제약, 멱등 키, 멱등 컨슈머).
- 원자적 연산, 경쟁 상태(race condition), `SKIP LOCKED`로 작업 분배.

## 5) 아키텍처 · 설계
- 계층형 / **헥사고날(포트&어댑터)** / 클린 아키텍처, 의존성 방향(안 → 밖 금지).
- **SOLID**, 응집도↑·결합도↓, DDD(바운디드 컨텍스트, 애그리거트), DTO 경계.
- 트레이드오프로 말하기(모놀리스 vs MSA, 동기 vs 비동기).

## 6) 분산 시스템 · 메시징
- **CAP**, 결과적 일관성, 분산 트랜잭션의 어려움 → **Saga / Outbox** 패턴.
- 메시지 보장: at-least-once vs exactly-once, **멱등 컨슈머**, 순서, DLQ.
- 서비스 간 경계: 동기(REST) vs 비동기(이벤트), 데이터 복제(읽기 프로젝션).

## 7) 캐싱
- 계층(로컬 L1 / 분산 L2), 무효화 전략, TTL, **캐시 스탬피드/penetration** 방지.
- 정합성 vs 성능 트레이드오프, 쓰기 정책(write-through/around/back).

## 8) 보안
- 인증(JWT/세션) vs 인가(RBAC), **비밀번호 해싱**(BCrypt/Argon2, salt).
- 입력 검증, **OWASP Top 10**(SQLi/XSS/CSRF), 비밀관리, 최소 권한, 민감정보 마스킹.
- Rate Limiting, 감사 로그.

## 9) 관측성(Observability)
- 3축: **로그**(구조화), **메트릭**(Micrometer/Prometheus), **트레이스**(분산 추적).
- 상관관계 ID(MDC/traceId), 비동기 경계에서 컨텍스트 전파, SLO/알람.

## 10) 테스트
- 테스트 피라미드(단위→통합→E2E), TDD, 경계/엣지 케이스.
- **Testcontainers**로 실제 인프라 통합 테스트, 계약 테스트, 동시성 테스트.

## 11) 빌드 · 배포 · 운영
- 빌드 도구(Gradle/Maven), **CI/CD**, 컨테이너(Docker)·오케스트레이션(K8s).
- **DB 마이그레이션**(Flyway/Liquibase, 무중단/롤백), 블루-그린/카나리, 12-Factor.

## 12) 자료구조 · 알고리즘 기초
- 복잡도 감각(Big-O), 해시/트리/힙/그래프, 적합한 구조 선택 → [`자료구조.md`](./자료구조.md).

# Part 2. 이 프로젝트에서의 백엔드 기술

Lemuel은 **주문·결제·정산을 2개 마이크로서비스 + API Gateway로 분리한 헥사고날 백엔드**다. 위 소양이 어디에 구현돼 있는지 매핑한다.

## 한눈에 — 소양 ↔ 구현 매핑

| 기본 소양 | 이 프로젝트의 구현 |
|-----------|--------------------|
| 아키텍처(헥사고날) | `{domain}/domain · application/port · adapter/{in,out}`, ArchUnit 강제 |
| MSA 경계 | order-service ↔ settlement-service **코드 의존 0** (Read-model + Kafka) |
| 분산 트랜잭션 | **Transactional Outbox + Kafka**, 3단 멱등 방어 |
| DB/인덱스 | PostgreSQL 17, Flyway V1~, B+Tree 인덱스 풀스캔 방지 |
| 동시성 | `@Version` 낙관락, 비관락+Idempotency-Key, `SKIP LOCKED` 배치 발행 |
| 캐싱 | Caffeine(L1) + Redis(L2) 2-tier + Pub/Sub 무효화 |
| 검색/집계 | Elasticsearch 비정규화 프로젝션(Nori) |
| 보안 | JWT(HS256), BCrypt(cost 12), Bucket4j RateLimit, 감사·PII 마스킹 |
| 관측성 | Micrometer+Prometheus, MDC traceId, OTel 트레이싱, **AOP 로깅/성능/트랜잭션 추적** |
| 회복탄력성 | Resilience4j(서킷브레이커), DB 재시도 큐, ShedLock |
| 테스트 | Testcontainers 통합·동시성 IT, ArchUnit |
| 배포 | Gradle 멀티모듈, Docker, 파라미터화 Dockerfile |

## 2-1. 헥사고날 아키텍처 (포트 & 어댑터)
각 도메인이 `domain`(순수 POJO) → `application/port/{in,out}` + `service` → `adapter/{in/web, in/kafka, out/persistence, out/event, out/search}` 로 분리. **도메인·애플리케이션은 프레임워크/인프라를 모른다** — 예: 정산 색인 로직(`IndexSettlementService`)은 ES가 아니라 `SettlementSearchIndexPort` 에만 의존. 의존성 방향은 **ArchUnit 테스트로 강제**.

## 2-2. MSA 분리 + Read-only Projection (핵심 설계)
- `settlement-service` 는 `order-service` 를 **import 하지 않으면서** Order/Payment/User/Product 데이터를 읽어야 한다.
- 해법: settlement-service 안에 `@Immutable` JPA 엔티티(`Settlement*ReadModel`)를 정의해 **같은 테이블을 read-only 매핑** → 코드 의존성 0.
- `settlement-service/build.gradle.kts` 에 `implementation(project(":order-service"))` **없음** → MSA 코드 경계 100% 확립.
- 서비스 간 상태 변경은 **Kafka 이벤트로만** 전파.

## 2-3. 분산 트랜잭션 — Transactional Outbox + 멱등
직접 분산 트랜잭션 대신 **Outbox 패턴**:
```
[order-service] Payment.capture() (DB tx)
   ├─ payments.status = CAPTURED
   └─ outbox_events INSERT (PaymentCaptured)   ← 같은 트랜잭션 (원자성)
                 ↓ poller(claim 기반 다중 워커 배치 발행)
             Kafka: lemuel.payment.captured
                 ↓
[settlement-service] consumer → Settlement.createFromPayment()
```
- **3단 멱등 방어**: `outbox_events.event_id`(UUID UNIQUE) → `processed_events`(consumer_group+event_id PK) → `settlements.payment_id`(UNIQUE). 메시지 중복·재처리에도 정산 1건 보장.
- 발행기: 단일 인스턴스 직렬 폴링 → **claim(`FOR UPDATE SKIP LOCKED`) 기반 다중 워커 배치 비동기 발행**으로 처리량 확보(리스 만료 자동 회수). Kafka 대기는 트랜잭션 밖에서.

## 2-4. 데이터베이스 · 인덱스
- PostgreSQL 17, **Flyway** 마이그레이션(V1~, 이후 timestamp 명명)으로 스키마 버전 관리.
- **인덱스(B+Tree)로 풀스캔 방지**: 리포지토리 쿼리 패턴↔인덱스 전수 대조해 누락 보강(`(status, created_at)`, 부분 인덱스 등). 복합 인덱스로 `WHERE a=? ORDER BY b` 정렬 제거.
- 복식부기 원장(`ledger_entries` 차변/대변), 홀드백, PG 대사(reconciliation) 등 금융 도메인 스키마.

## 2-5. 동시성 · 정합성
| 기법 | 적용 |
|------|------|
| **낙관적 락 `@Version`** | `settlements`, `payouts` — 동시 확정/지급 충돌 감지·재시도 |
| **비관적 락 + Idempotency-Key** | 환불 — `refunds.idempotency_key` 유니크 + 행 잠금으로 중복 환불 차단 |
| **원자적 조건부 UPDATE** | 재고 차감 — `stock >= qty` 가드를 UPDATE에 넣어 초과판매 방지(동시 100스레드 IT로 검증) |
| **`FOR UPDATE SKIP LOCKED`** | Outbox 멀티워커 claim — 워커 간 겹침 없는 작업 분배 |

## 2-6. 캐싱
- **2-tier**: Caffeine(L1, Window-TinyLFU) + Redis(L2, opt-in), 각 캐시가 전용 L1 + 공유 L2.
- **Pub/Sub 무효화**: `CacheInvalidationPublisher` 가 인스턴스 로컬 L1을 전파 무효화 → 다중 인스턴스 일관성.

## 2-7. 검색 · 집계 (Elasticsearch)
- `settlement-service` 가 Settlement+Order+Payment+Refund 를 **비정규화 통합 문서**로 ES 색인(Nori 한글 분석, 금액 `scaled_float`).
- **이벤트 기반 `@Async` 색인 파이프라인** + 실패 시 `settlement_index_queue` DB 재시도 큐.
- 정합성=PostgreSQL, 검색·집계=ES (CQRS 성격 읽기 분리). 상세 → [`일라스틱서치.md`](./일라스틱서치.md).

## 2-8. 보안
| 항목 | 구현 |
|------|------|
| 인증 | **JWT(HS256)** — `shared-common.common.config.jwt` |
| 비밀번호 | **BCrypt(cost=12)** |
| Rate Limiting | **Bucket4j 토큰버킷** — `RateLimitFilter` + `RateLimitRegistry`(`ConcurrentHashMap` + `computeIfAbsent`) |
| 감사·PII | 감사 로그 + **PII 마스킹**(`shared-common.common.audit`, AOP `AuditAspect`) |
| 경계 | API Gateway(Spring Cloud Gateway, reactive)에서 라우팅·인증 필터 |
| CORS | 환경변수 화이트리스트 |

## 2-9. 관측성(Observability)
- **메트릭**: Micrometer → Prometheus(결제·정산·발행 카운터/타이머).
- **로그**: 구조화(JSON, logstash-logback), `TraceIdFilter` 가 요청별 `traceId` 를 MDC 주입 → 결제→정산→환불 체인 grep.
- **분산 트레이싱**: Micrometer Tracing + OTel, Outbox가 `traceparent` 를 영속화해 **비동기 경계에서도 trace 단절 없음**.
- **AOP 통합 추적**: `MethodTraceAspect`(진입/종료 로깅 + 실행시간 + Micrometer 타이머 + 느린 메서드 WARN), `TransactionTraceAspect`(`@Transactional` 경계 + `txId` MDC) — 헥사고날 레이어(web/service/kafka)에 횡단 적용.

## 2-10. 회복탄력성(Resilience)
- **Resilience4j 서킷브레이커**: Toss 등 PG 연동(`TossPaymentService`, PG 어댑터) — 외부 장애 격리, `TossPaymentResilienceTest` 로 검증.
- **DB 기반 재시도 큐**: 색인 실패(`settlement_index_queue`), Outbox 재시도 — 일시 장애에도 유실 없음.
- **ShedLock**: `@Scheduled` 분산 락 — N개 레플리카 중 1개만 실행 보장(중복 배치 방지).
- PG 라우팅 fallback 체인: unhealthy PG → 다음 후보 자동 전환.

## 2-11. 배치 · 스케줄링
- **Spring Batch** + 스케줄러: 일별 정산 생성, 지급(`PayoutScheduler`), 홀드백 해제, 원장 발행(`LedgerOutboxPoller`), 인덱스 재시도.
- ShedLock으로 다중 인스턴스 안전.

## 2-12. 테스트 전략
- **Testcontainers**(PostgreSQL/ES/Kafka)로 실제 인프라 통합 테스트.
- **동시성 IT**: 재고 차감(100스레드), Outbox claim 멀티워커(`SKIP LOCKED` disjoint 분배) — 커밋된 데이터 + 독립 트랜잭션으로 실제 경합 검증.
- **ArchUnit**: 헥사고날 의존 방향·MSA 경계 위반 컴파일타임급 차단.
- 도메인 단위 → 서비스 → 컨트롤러 → 통합 순. JaCoCo 커버리지 게이트.

## 2-13. 빌드 · 배포
- **Gradle 멀티모듈**(Kotlin DSL): `shared-common`(java-library) + 2 서비스 + gateway.
- **Docker**: `MODULE` 빌드 인자로 서비스 선택하는 파라미터화 Dockerfile, Docker Compose(PG+ES+Redpanda+서비스), 운영은 Kubernetes.

## 정리 — 이 프로젝트가 보여주는 백엔드 역량

1. **설계**: 헥사고날 + MSA 분리 + Read-model 프로젝션으로 _경계와 의존성_ 을 코드로 강제(ArchUnit).
2. **분산/정합성**: Outbox + Kafka + 3단 멱등으로 _분산 트랜잭션 없이 신뢰성_, claim 기반 병렬 발행으로 _처리량_.
3. **데이터**: B+Tree 인덱스·복식부기 원장·PG 대사 등 _금융 도메인의 정확성_, 풀스캔 방지.
4. **동시성**: 낙관/비관 락·멱등 키·원자 UPDATE·SKIP LOCKED를 _상황에 맞게_ 선택.
5. **성능**: 2-tier 캐시, ES 검색 프로젝션, JDBC 배치.
6. **운영**: 관측성 3축(메트릭/로그/트레이스) + AOP 횡단 추적, Resilience4j·ShedLock·재시도 큐로 _장애에 강하게_.
7. **품질**: Testcontainers 통합·동시성 테스트 + ArchUnit + 커버리지 게이트.

> 한 줄로: **"트랜잭션 정합성이 중요한 금융 도메인을, 경계를 명확히 한 헥사고날 MSA 위에서, 멱등·관측·회복탄력성을 갖춰 구현한 백엔드."**
