# Elasticsearch — 유용성 · 사용법 · 이 프로젝트에서의 사용

이커머스 + 정산 MSA 플랫폼(Lemuel)에서 **정산 검색·집계용 읽기 프로젝션**으로 Elasticsearch(ES 8.17)를 사용한다. 이 문서는 (1) ES가 왜 유용한가 → (2) 핵심 개념·기본 사용법 → (3) **이 프로젝트가 실제로 ES를 쓰는 방식** 순으로 정리한다.

---

## 1) Elasticsearch란 & 왜 쓰는가

**Elasticsearch** = Apache Lucene 기반의 분산 검색·분석 엔진. 데이터를 **역색인(Inverted Index)** 으로 저장해 "수억 건에서 특정 단어/조건을 ms 단위로" 찾는다.

### 핵심 유용성

| 강점 | 설명 | RDB 대비 |
|------|------|----------|
| **전문 검색(Full-text)** | 형태소 분석 + 역색인으로 "포함/유사/오타 허용" 검색 | `LIKE '%키워드%'` 는 인덱스 못 타고 풀스캔 → ES는 O(1)급 |
| **빠른 집계(Aggregation)** | 합계·평균·버킷·통계를 분산 병렬로 | 대용량 `GROUP BY` 보다 빠르고 유연 |
| **분산·확장(Scale-out)** | 샤드(shard)로 수평 분할, 레플리카로 가용성 | 단일 RDB 스케일업 한계 |
| **준실시간(NRT)** | 색인 후 ~1초 내 검색 가능 | — |
| **스키마 유연성** | JSON 문서, 동적 매핑 | 정규화 join 불필요(비정규화) |
| **관측성 스택** | Kibana 시각화, 로그/메트릭(ELK) | — |

### 언제 ES, 언제 RDB?

| 요구 | 적합 |
|------|------|
| 트랜잭션·정합성·단건 CRUD·외래키 | **RDB**(PostgreSQL) |
| 키워드/형태소 검색, 자동완성, 유사도 | **ES** |
| 대시보드 집계·통계·드릴다운 | **ES** |
| "진실의 원천(System of Record)" | **RDB** |
| "읽기 최적화 사본(Read Projection)" | **ES** |

> **원칙**: ES는 보통 **진실의 원천이 아니라 읽기 프로젝션**이다. RDB에 정본을 쓰고, 검색/집계용으로 ES에 비정규화해 복제한다. (이 프로젝트도 이 구조)

## 2) 핵심 개념

| 개념 | 의미 | RDB 유추 |
|------|------|----------|
| **Index** | 문서의 모음 | 테이블 |
| **Document** | JSON 한 건 | 행(row) |
| **Field** | 문서의 속성 | 컬럼 |
| **Mapping** | 필드 타입·분석 정의 | 스키마 |
| **Shard** | 인덱스의 물리 분할 단위(Lucene 인덱스) | 파티션 |
| **Replica** | 샤드 복제본(가용성·읽기 분산) | 레플리카 |
| **Analyzer** | 텍스트 → 토큰 분해(검색 정확도 핵심) | — |
| **Inverted Index** | 토큰 → 문서ID 목록 | (역방향 인덱스) |

### 필드 타입 — `text` vs `keyword` (가장 중요)
- **`text`**: 분석기로 토큰화 → **전문 검색**용. 정렬·집계 부적합.
- **`keyword`**: 분석 없이 통째로 → **정확 일치·정렬·집계**용(상태값, ID, enum).
- 둘 다 필요하면 멀티필드(`field` + `field.keyword`).

### Analyzer(분석기)
텍스트를 어떻게 토큰으로 쪼개는지 결정. 한국어는 **Nori**(은전한닢 기반) 형태소 분석기를 쓴다 — "정산을"→"정산"+"을"(조사 제거) 처럼 어근만 색인해야 검색이 맞는다.

## 3) 기본 사용법 (REST API)

```bash
# 1) 문서 색인 (PUT /인덱스/_doc/id)
PUT settlement_search/_doc/1001
{ "settlementId": 1001, "settlementStatus": "DONE", "settlementAmount": 50000,
  "paymentMethod": "신한카드", "settlementDate": "2026-06-11" }

# 2) 검색 (Query DSL) — 상태 필터 + 한글 전문검색
GET settlement_search/_search
{
  "query": {
    "bool": {
      "filter": [ { "term": { "settlementStatus": "DONE" } } ],
      "must":   [ { "match": { "paymentMethod": "카드" } } ]
    }
  }
}

# 3) 집계 — 일자별 정산금 합계
GET settlement_search/_search
{
  "size": 0,
  "aggs": {
    "by_date": {
      "date_histogram": { "field": "settlementDate", "calendar_interval": "day" },
      "aggs": { "total": { "sum": { "field": "settlementAmount" } } }
    }
  }
}
```

**쿼리 절 구분**:
- `match`(분석·점수 매김, 전문검색) vs `term`(정확 일치, keyword/숫자)
- `bool`의 `filter`(점수 無·캐시 O, 빠름) vs `must`(점수 산정)
- 집계는 `size:0`(문서 본문 불필요)으로 성능 ↑

### Spring Data Elasticsearch (자바)
```java
// 문서 = @Document, 필드 = @Field, 인덱스 설정 = @Setting
@Document(indexName = "settlement_search")
class Doc { @Id String id; @Field(type = FieldType.Keyword) String status; }

// 저장/조회
elasticsearchOperations.save(doc);
elasticsearchOperations.get(id, Doc.class);
// 또는 ElasticsearchRepository<Doc, String> 파생 쿼리
```

## 4) 이 프로젝트에서의 Elasticsearch

> 위치: `settlement-service` (정산 서비스 전용). `order-service` 는 ES 미사용(거래 정본은 PostgreSQL).
> 의존성: `spring-boot-starter-data-elasticsearch`, ES 8.17. 접속: `spring.elasticsearch.uris`(환경변수).

### 4-1. 피처 플래그로 격리 — `app.search.enabled`
ES 관련 빈은 전부 `@ConditionalOnProperty(name = "app.search.enabled", havingValue = "true")`. **기본 off** — ES 없이도 서비스가 뜬다(검색은 부가 기능, 핵심 정산 흐름과 디커플). 켜질 때만 색인 어댑터·이벤트 리스너·ES 설정이 활성화.

### 4-2. 비정규화 통합 문서 — `SettlementSearchDocument`
인덱스 `settlement_search` 하나에 **Settlement + Order + Payment + Refund 를 한 문서로 합쳐(비정규화)** 저장한다. 검색 시 join이 필요 없도록 읽기 최적화한 **검색 프로젝션**.

```
SettlementSearchDocument (index: settlement_search)
├─ Settlement: settlementId, settlementStatus(keyword), settlementAmount, settlementDate, 승인/반려 정보
├─ Order:      orderId, userId, orderStatus(keyword), orderAmount, orderCreatedAt
├─ Payment:    paymentId, paymentStatus, paymentAmount, refundedAmount, paymentMethod(text/nori), pgTransactionId(keyword)
└─ Refund:     hasRefund, refundCount, latestRefundStatus, refundReason(text/nori), 일시
```

**필드 타입 설계 포인트**:
- **금액** → `@Field(type = Scaled_Float, scalingFactor = 100)`: ES가 내부적으로 `값×100`을 정수로 저장 → **부동소수 오차 없이** 통화 집계(`BigDecimal` 매핑).
- **상태값·ID·PG거래ID** → `keyword`: 정확 일치·집계·정렬용(`settlementStatus`, `orderStatus`, `pgTransactionId`).
- **한글 텍스트**(반려사유·결제수단·환불사유) → `text` + `analyzer = "nori_analyzer"`: 형태소 검색.

### 4-3. 한국어 형태소 분석 — `settlement-index-settings.json`
`@Setting(settingPath = "elasticsearch/settlement-index-settings.json")` 로 커스텀 분석기 적용:
- `nori_tokenizer` (`decompound_mode: mixed` — 복합어 분해, `discard_punctuation`)
- `nori_part_of_speech` 필터로 **조사·어미·기호 등 불용 품사(E, J, MM, SP …) 제거**
- `lowercase` + `nori_readingform`(한자→한글 독음)
- → "정산을/결제했다" 같은 입력에서 어근만 색인돼 한글 검색 정확도 확보.
- `@Document(createIndex = false)`: 인덱스는 외부(설정 파일·운영 도구)에서 생성 — 매핑/분석기를 명시적으로 통제.

### 4-4. 이벤트 기반 비동기 색인 파이프라인 (헥사고날)

```
[정산 도메인 변경] 생성/확정/환불/승인/반려
        │  (Spring ApplicationEvent)
        ▼
SettlementIndexEvent  ── eventType: BATCH_CREATED/BATCH_CONFIRMED/REFUND_PROCESSED → 벌크
                          SINGLE_UPDATED/APPROVED/REJECTED → 단건
        │
        ▼  @Async @EventListener
SettlementIndexEventListener        (in/event, 얇은 인바운드 어댑터)
        │
        ▼  포트 호출 (ES 비의존 순수 로직)
IndexSettlementService  ── IndexSettlementUseCase 구현
        │  loadSettlementPort 로 정본 조회 → 포트로 색인 위임
        ▼  SettlementSearchIndexPort (out 포트)
SettlementSearchAdapter ── ElasticsearchOperations.save(document)   (out/search 어댑터)
        │
        └─(실패 시)→ EnqueueFailedIndexPort → settlement_index_queue (DB 재시도 큐)
```

**핵심 설계**:
- **헥사고날 격리**: 애플리케이션 레이어(`IndexSettlementService`)는 ES API를 모른다 — `SettlementSearchIndexPort` 인터페이스에만 의존. ES 교체·비활성화가 자유롭고 ArchUnit으로 경계 강제.
- **비동기**(`@Async`): 색인이 정산 트랜잭션을 막지 않는다(검색은 NRT라도 충분).
- **벌크 vs 단건**: 대량 생성/확정은 `bulkIndexSettlements`, 개별 승인/반려는 단건.

### 4-5. 실패 복원력 — DB 기반 재시도 큐
색인 실패 시 예외를 삼키지 않고 **`settlement_index_queue` 테이블**(`SettlementIndexQueueAdapter`)에 재시도 작업을 적재한다. `nextRetryAt` 백오프로 재처리 → **ES 일시 장애에도 색인 유실 없음**. (5장 Outbox와 동일 사상의 DB-backed 큐. 마이그레이션 `V5__settlement_index_queue.sql`.) 검색 비활성 환경에선 `NoOpSettlementIndexQueueAdapter` 가 대체.

### 4-6. 읽기 경로 구분 — ES vs PostgreSQL
- **ES**: 정산 **검색·전문검색·집계** 용 비정규화 프로젝션.
- **PostgreSQL + QueryDSL**(`SettlementQueryController` / `SettlementQueryAdapter`): 관리자 **트랜잭션 조회·커서 페이지네이션**(정합성 필요한 정본 조회)은 RDB에서 처리.
- → 정합성은 RDB, 검색/집계는 ES로 **CQRS 성격의 읽기 분리**.

## 5) 운영·설계 시 주의

| 포인트 | 내용 |
|--------|------|
| **정합성 한계** | ES는 NRT·결과적 일관성. "방금 바뀐 값이 즉시 정확" 이 필요하면 RDB 조회 |
| **매핑은 사후 변경 어려움** | 필드 타입 바꾸려면 재색인(reindex). 초기 매핑 설계 중요 |
| **`text`/`keyword` 혼동** | 집계·정렬을 `text` 에 하면 실패/비효율 → `keyword` 사용 |
| **금액·정밀도** | float 직접 저장 금지 → `scaled_float`(이 프로젝트 방식) 또는 정수화 |
| **재색인 전략** | 매핑 변경 시 alias 스위칭(blue-green reindex)로 무중단 |
| **유실 방지** | 색인은 비동기라도 **실패 재시도 큐**(이 프로젝트)나 CDC/Outbox로 보장 |
| **보안** | `spring.elasticsearch.username/password`, 운영은 TLS·인증 필수 |

## 정리

- **ES 유용성**: 역색인 기반 전문검색 + 분산 집계 + 준실시간. RDB의 `LIKE` 풀스캔·무거운 `GROUP BY` 를 대체하는 **읽기 프로젝션**.
- **이 프로젝트**: `settlement-service` 가 `app.search.enabled` 플래그 하에, Settlement·Order·Payment·Refund 를 **하나의 비정규화 문서**(`settlement_search`)로 합쳐 **Nori 한글 분석 + scaled_float 금액**으로 색인.
- **파이프라인**: 도메인 이벤트 → `@Async` 리스너 → 헥사고날 포트(`SettlementSearchIndexPort`) → `ElasticsearchOperations.save`, 실패 시 **`settlement_index_queue` DB 재시도 큐**로 유실 방지.
- **경계**: 정본·정합성=PostgreSQL, 검색·집계=Elasticsearch (CQRS 성격의 읽기 분리). 애플리케이션 레이어는 ES에 직접 의존하지 않음(포트로 격리).
