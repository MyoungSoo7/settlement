# 05. 문서형 하네스 설계도 (2003-4-2 산출물)

> 입력: `2003-4-1-workspace/00-source-meeting-note.md`, `2003-4-1-workspace/05-2003-4-1-output.md`. 회의록을 1page 기획문서로 바꾸는 흐름을 입력 분해 → 출력 맵핑 → 확인 질문으로 설계한다. 회의록에 없는 사실은 추가하지 않는다.

# 문서형 하네스 설계도

## 목적
- 사내 교육 신청 페이지 개편 회의록을 1page 기획문서 초안으로 바꾼다.
- 확인되지 않은 내용은 결론으로 쓰지 않고 확인 필요 항목 칸으로 남긴다.
- 2003-4-1에서 정리한 권한·제약·순서·검수 포인트를 본 설계도의 각 단계 옆에 붙여 둔다.

## 하네스 구조 (아스키)

네 층으로 본다.

(0) 상단 메타 스킬 — AI가 매 호출 직전에 읽는 행동 양식(역할·금지·순서·hook 협업 규약). Superpowers `using-superpowers/SKILL.md` 자리.

(1) 세로축 파이프라인 — 입력 → 6단계 처리 → 출력.

(2) 가로축 레일 — 권한·제약·순서가 왼쪽에서, 금지사항이 단계마다 가드를 댐.

(3) 하단 게이트 — Write/Edit 직후 PostToolUse hook이 결과를 검사하고, 실패 시 Claude 재작성 루프로 되돌려 보냄.

## 금지사항
- 회의록에 없는 수치·일정·담당자·외부 사례를 본문 결론으로 쓰지 않는다.
- "CS 문의 60%"처럼 단서가 붙은 수치는 배경 본문이 아니라 확인 필요 항목으로 옮긴다.
- 페이지 전체 리뉴얼, 결제 모듈 모달 위치 변경, 환불 기준 자체 명문화는 해결 방향에 넣지 않는다.
- 미결정 항목을 "~로 한다 / ~할 예정이다" 같은 결론 톤으로 단정하지 않는다.

## 처리 순서

1. 원문 보존 — 회의록 원문을 그대로 첨부하거나 인용 블록으로 유지한다.
2. 입력 항목 분해 — 회의에서 확정된 내용과 결정되지 않은 내용을 먼저 두 묶음으로 나눈다.
3. 출력 항목 맵핑 — 확정 내용만 제목·배경·문제·사용자·해결 방향·성공 기준 6칸으로 옮긴다.
4. 확인 질문 분리 — 결정되지 않은 내용은 확인 필요 항목 한 칸으로 모은다.
5. 보존 표현 인용 검사 — 세 줄 보존 표현이 큰따옴표로 남아 있는지 마지막에 한 번 더 확인한다.
6. 1page 기획문서 초안 작성 — 한 페이지 안에 7칸이 들어가는지 확인하고 마감한다.

## 4-1에서 가져온 검수 포인트

- 7칸이 한 페이지 안에 들어간다.
- 확인 필요 항목과 결론 칸이 서로 다른 자리에 분리되어 있다.
- 보존 표현이 의역되지 않고 큰따옴표 인용으로 유지된다.
- 회의록에 없는 수치·일정·담당자가 결론 칸에 들어가지 않았다.
- 범위 밖 항목(전체 리뉴얼·결제 모달·환불 기준 명문화)이 해결 방향에 섞이지 않았다.

이 중 ②③④ 세 줄은 `2003-4-1-workspace/.claude/hooks/check-1page-output.sh` PostToolUse hook으로 자동 검사된다. md 검수 점검과 hook이 같은 기준을 이중으로 본다.

## 확인 질문으로 남긴 항목

- 최종 담당자·일정: 월요일 정기 미팅에서 확정 예정. 미팅 이후 설계도에 회신을 한 번 더 반영한다.
- CS 문의 60% 수치 출처: 한CS가 "정확한 수치는 다시 뽑아 와야 한다"고 단서를 달았다. 출처 확인 전까지 본문 결론으로 쓰지 않는다.
- 환불 기준 명문화: 운영팀 별도 확인 사안. 이번 페이지 개편 범위 밖이라 해결 방향에 넣지 않고 확인 필요 항목으로만 표시한다.
- 시안·메일 템플릿 작업 일정: 김디자이너(다음 주 화요일 시안), 정BE(시안 확정 후 3일) 발언이 있으나, 박PM이 "오늘은 방향만 합의한다"고 했으므로 월요일 정기 미팅 확정 전까지 결론 일정으로 쓰지 않는다.
