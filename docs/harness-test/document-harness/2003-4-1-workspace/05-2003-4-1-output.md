# 하네스 작성 프랙티스

## 관찰 대상
- case study: Superpowers
- 설치/실행 검증 여부: 하지 않음
- 관찰 목적: 문서가 권한·제약·순서·검수 포인트를 어떻게 나누는지 확인

## Superpowers 관찰표

| 관찰 경로 | 관찰할 문서형 역할 | 프로젝트 1에 옮길 해석 |
|---|---|---|
| README.md | 전체 흐름을 찾는 목차 | md 템플릿 v1의 첫 단락에 입력 → 처리 원칙 → 출력 형식 순서를 한눈에 보이게 둔다 |
| AGENTS.md / CLAUDE.md / GEMINI.md | 초기 지침과 금지선 | "회의록에 없는 사실을 만들지 않는다"를 템플릿 상단 역할·금지 섹션에 먼저 적는다 |
| commands/ (write-plan, execute-plan) | 호출 가능한 순서 | 원문 보존 → 확정/미결정 분리 → 7칸 옮김 → 보존 표현 인용 검사 순서를 처리 원칙으로 정의한다 |
| skills/ (writing-skills, subagent-driven-development, using-superpowers) | 반복 행동의 분리 단위 | 처리 원칙·금지·출력 형식·예시·복사용 빈칸을 한 문단에 섞지 않고 섹션을 나눠 적는다 |
| hooks/ / tests/ | 검수와 개입 타이밍 | 보존 표현 누락 검사·확인 필요 항목 분리·미확인 수치 단정 사용을 출력 전·후 두 칸으로 나눈다. **이 workspace에서는 실제로 `.claude/settings.json` + `PostToolUse` hook으로 옮겼다 (아래 "Superpowers에서 옮긴 것 / 못 옮긴 것" 표 참조).** |

## 프로젝트 1에 옮길 규칙

### 권한
- AI의 역할은 회의록 원문과 확인된 요구사항 안에서 1page 기획안 초안을 7칸으로 구조화하는 보조자다.
- "반드시 보존할 표현"은 출력에서 큰따옴표 인용으로 유지한다.
- 비어 있는 입력(결정권자·일정·수치 출처)은 임의로 채우지 않고 "확인 필요 항목"으로 옮긴다.

### 제약
- 회의록에 없는 수치·일정·담당자·외부 사례·환불 기준은 결론처럼 쓰지 않는다.
- "정확한 출처는 다시 뽑아 와야 한다" 같은 단서가 붙은 수치(예: CS 문의 60%)는 본문 결론이 아니라 확인 필요 항목에만 적는다.
- 페이지 전체 리뉴얼, 결제 모듈 모달 위치 변경, 환불 기준 자체 명문화는 회의에서 범위 밖으로 합의됐다. 해결 방향에 넣지 않는다.
- 미결정 항목을 결론 톤("~로 한다", "~할 예정이다")으로 단정하지 않는다.

### 순서
1. 회의록 원문을 그대로 보존한다.
2. "회의에서 확정된 내용"과 "회의에서 결정되지 않은 내용"을 먼저 분리한다.
3. 확정 내용만 제목·배경·문제·사용자 또는 대상·해결 방향·성공 기준 6칸으로 옮긴다.
4. 결정되지 않은 내용은 확인 필요 항목 한 칸으로 모은다.
5. "반드시 보존할 표현" 세 줄("신청 직후 환불·일정 문의를 줄인다" / "페이지 전체 리뉴얼은 이번 범위가 아니다" / "환불 기준 자체가 명문화돼 있지 않다")이 인용 부호로 남아 있는지 마지막에 한 번 더 확인한다.

### 검수 포인트
- 7칸이 한 페이지 안에 들어간다.
- 확인 필요 항목과 결론 칸이 서로 다른 자리에 분리되어 있다.
- 보존 표현이 의역되지 않고 큰따옴표 인용으로 유지된다.
- 회의록에 없는 수치·일정·담당자가 결론 칸에 들어가지 않았다.
- 범위 밖 항목(전체 리뉴얼·결제 모달·환불 기준 명문화)이 해결 방향에 섞이지 않았다.

## 다음 설계도로 넘길 항목
- 회의록 입력에서 먼저 분리할 항목: 확정된 내용, 결정되지 않은 내용, 반드시 보존할 표현, 비어 있으면 질문할 항목
- 1page 기획안 출력으로 보낼 항목: 제목, 배경, 문제, 사용자 또는 대상, 해결 방향, 성공 기준
- 확인 질문으로 남길 항목: 최종 담당자·일정(월요일 정기 미팅), CS 문의 60% 수치 출처, 환불 기준 명문화 여부(운영팀 별도 확인)
- md 템플릿 v1에 바로 넣을 항목: 역할, 금지, 처리 원칙 5단계 순서, 출력 형식 7칸, 보존 표현 인용 규칙, 검수 점검 5줄
- 2003-4-2 분업형 설계도에서 다시 다룰 후보: 확인 필요 항목 회차별 답변 추적(월요일 미팅 → 운영팀 회신 → CS 수치 회신), 보존 표현 누락 시 재시도, 외부 회신 단계 순서 강제

## Superpowers에서 옮긴 것 / 못 옮긴 것 / 직접 구현한 것

Superpowers 관찰의 진짜 산출물은 "어디까지가 md로 되고 어디부터는 안 되는지를 그은 선"이다. 이 workspace에서는 그 선을 다음과 같이 분명히 그어 둔다.

| Superpowers 구성 | 회의록 → 1page 적용 방식 | 구현 자리 |
|---|---|---|
| `AGENTS.md` / `CLAUDE.md` 류 초기 지침 | md 텍스트로 그대로 옮김 | `02-2003-2-3-output.md` 역할·금지 섹션 |
| `skills/*.md` 반복 행동 분리 | 섹션 분리 규칙으로 사용 | `02-2003-2-3-output.md` 각 섹션 |
| `skills/using-superpowers/SKILL.md` 메타 사용법 | **이 workspace에서 실제로 옮겼다** | `.claude/skills/using-1page-harness/SKILL.md` |
| `commands/` 호출 순서 | md로는 안내만 가능. 강제 실행은 4장 분업형으로 미룬다 | 본 산출물 "다음 설계도로 넘길 항목" |
| `hooks/` 개입 타이밍 | **이 workspace에서 실제로 옮겼다** | `.claude/settings.json` + `.claude/hooks/check-1page-output.sh` |
| `tests/` 자동 검수 | 위 hook의 검사 항목 3종이 이 자리를 채운다 (보존 표현 누락·확인 필요 항목 섹션 존재·미확인 수치 단정 사용) | `.claude/hooks/check-1page-output.sh` |

### 이 workspace의 hook 동작 명세
- 트리거: `PostToolUse(matcher: "Write|Edit")` — 1page 기획안 산출물 파일을 쓰거나 편집한 직후.
- 대상 필터: 파일 경로에 `1page`, `one-page`, `2003-4-3` 중 하나가 포함될 때만 검사한다. 회의록·중간 산출물은 건드리지 않는다.
- 검사 1 (보존 표현): "신청 직후 환불·일정 문의를 줄인다" / "페이지 전체 리뉴얼은 이번 범위가 아니다" / "환불 기준 자체가 명문화돼 있지 않다" 세 줄이 큰따옴표 인용 형태로 모두 있는지 확인.
- 검사 2 (확인 필요 항목 섹션): `## 확인 필요 항목` 헤딩이 존재하는지 확인.
- 검사 3 (미확인 수치 단정): 확인 필요 항목 섹션 위쪽 본문에서 "60%" 단정 사용을 막는다.
- 실패 처리: 위반 항목을 stderr로 적고 exit code 2를 반환한다. Claude Code는 이 메시지를 다음 턴 입력으로 받아 산출물을 재작성한다.

### hook으로도 못 잡는 자리 (4-2 → 4장으로 미룬 후보)
- `commands/` 호출 순서 강제: hook은 결과물을 사후 검사하지만 순서를 강제하지는 못한다. 분업형 하네스 후보.
- 확인 필요 항목 답변의 회차 추적(월요일 미팅 → 운영팀 회신 → CS 수치 회신): 상태 저장이 필요하다. MCP형 하네스 후보.
- 보존 표현 누락 시 재시도는 hook이 신호만 준다. 자동 재실행 흐름은 분업형 후보.

## 확인 질문으로 남긴 항목
- Superpowers `commands/` 호출 순서가 실제로 강제되는지는 문서 관찰만으로 단정할 수 없다. 4-2 설계도에서 분업형 후보로만 표시한다.
- `hooks/`는 이 workspace에 실제로 걸어 두었지만, 검사 항목 3종이 충분한지는 1회 실행으로는 알 수 없다. 4-3 산출물에서 한 번 더 검수한다.
