# 04. Superpowers 관찰 메모

> 4-1 작성을 위한 사전 관찰 자료. 설치·실행 검증은 하지 않고, 관찰 가능한 문서 경로만 기록한다.

## 관찰 원칙
- repo 설치·실행·성능을 주장하지 않는다.
- 인용은 경로 이름과 그 경로가 보여 주는 문서형 역할까지만.
- 같은 경로가 여러 역할을 겸하면 "이 실습에서 옮길 해석"만 한 줄로 적는다.

## 경로별 관찰

| 관찰 경로 | 보여 주는 문서형 역할 | 회의록 → 1page 기획안에 옮길 해석 |
|---|---|---|
| `README.md` | 전체 흐름을 찾는 목차 | md 템플릿 v1의 첫 단락에 입력 → 처리 → 출력 순서를 한눈에 보이게 둔다 |
| `AGENTS.md` / `CLAUDE.md` / `GEMINI.md` | 초기 지침과 금지선 | "회의록에 없는 사실을 만들지 않는다"를 템플릿 상단 역할·금지에 먼저 적는다 |
| `commands/brainstorm.md` | 발산 단계 호출 | 사용하지 않음. 회의록은 이미 입력 자체가 모인 자료다 |
| `commands/write-plan.md` | 계획 단계 호출 | "확정 내용 분리 → 7칸으로 옮김" 순서를 처리 원칙에 둔다 |
| `commands/execute-plan.md` | 실행 단계 호출 | 보존 표현 인용 검사와 확인 필요 항목 회수는 사람 검수로 남긴다 |
| `skills/writing-skills/SKILL.md` | 출력 톤 규칙 단위 | 출력 형식과 예시 기준을 한 섹션에 묶지 않고 분리해서 적는다 |
| `skills/subagent-driven-development/SKILL.md` | 큰 작업을 분업으로 쪼개는 단위 | 답변 추적·재질문은 문서 밖 분업형 후보로 표시한다 |
| `skills/using-superpowers/SKILL.md` | 메타 사용법 단위 | "다른 업무에 복사할 때 바꿀 빈칸"을 템플릿 마지막에 정리한다 |
| `hooks/hooks.json` | 개입 타이밍 선언 | 보존 표현 누락 시 재시도는 문서 안에서 강제하지 않고 검수 점검으로만 둔다 |
| `hooks/session-start` | 시작 시 환경 고정 | "비어 있으면 질문할 항목"을 입력 섹션 맨 위에 둔다 |
| `tests/` | 검수 케이스 모음 | 예시 입력·예시 출력 한 쌍을 템플릿 끝에 고정한다 |

## 결정되지 않은 관찰
- Superpowers의 `commands/` 호출 순서가 강제되는지 여부는 문서로 단정하지 않는다. 4-2 설계도에서 분업형으로 옮길 후보로만 표시.
- `hooks/` 가 실제 실행 시 어떤 검사를 거는지 여부는 관찰 범위 밖. 문서형 한계로 분류한다.
