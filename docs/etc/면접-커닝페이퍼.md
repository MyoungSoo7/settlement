## ① 헥사고날 아키텍처
> *"비즈니스 로직을 보호하고 외부 의존성을 어댑터로 분리"*

**열어서 보여줄 것**
- 디렉토리: `settlement-service/src/main/java/.../settlement/`
  - `domain/` (POJO, 프레임워크 의존 0) — `Settlement.java`
  - `application/port/in/` (UseCase 인터페이스)
  - `application/port/out/` (영속성/외부 포트)
  - `adapter/in/` (web, kafka), `adapter/out/` (persistence, readmodel)

**한마디**
> "도메인이 포트(인터페이스)만 알고 어댑터 구현은 모릅니다. 그래서 DB·Kafka를 목으로 바꿔도 도메인 테스트가 그대로 돕니다. ArchUnit으로 의존 방향(adapter→application→domain)도 강제합니다."

**꼬리질문: "헥사고날 써서 실제로 뭐가 좋았나?"**
> "정산 금액 계산 로직을 인프라 없이 순수 단위 테스트할 수 있었고, read-model 기반 조회를 나중에 이벤트 프로젝션으로 바꿀 때 도메인은 안 건드렸습니다."

## ② MSA — 주문·대출·기사예약 독립 서비스 분리
> *"서비스별 확장성과 독립 배포, 장애 전파 최소화"*

**열어서 보여줄 것**
- `settings.gradle.kts` — 5개 모듈(order/settlement/reservation/loan/gateway)
- 각 서비스 자체 부트 클래스: `SettlementServiceApplication`, `LoanServiceApplication`, `ReservationServiceApplication` 등 → **각각 독립 실행/배포**
- loan = 자체 DB `lemuel_loan`, reservation = 자체 DB `reservations_db`

**한마디**
> "5개 서비스가 각자 @SpringBootApplication을 갖고 독립 배포됩니다. loan과 reservation은 DB까지 완전 분리(DB-per-service)하고 Kafka 이벤트로만 통신합니다."

**⚠️ 지뢰 꼬리질문: "주문도 DB 분리했나? 장애 격리 되나?"**
> "네, **물리 분리까지 완료**했습니다(ADR 0020 Phase 4). 원래 order↔settlement는 opslab를 공유하고 settlement가 `@Immutable` Read-only Projection으로 order 테이블을 직접 매핑했는데, **Strangler 패턴 6단계**로 점진 전환했습니다. 정산 생성에 필요한 데이터는 **이벤트에 동봉(Event-Carried State Transfer)**하고, 조회는 order 이벤트로 채우는 **로컬 CQRS 프로젝션(settlement_*_view)**으로 바꾼 뒤, settlement를 자체 `settlement_db`로 물리 분리했습니다. 그래서 **order DB가 죽어도 정산 생성·조회가 지속**되고 이벤트 큐만 적체됩니다."
> — 추가 한 방: "대사(일일/기간/PG)도 cross-DB 직독을 없앴습니다. order가 자기 합계를 내부 API(`/internal/recon/*`)로 노출하고 settlement는 `OrderReconClient`로 받아 자기 숫자와 비교합니다 — **양측 모두 자기 DB만 읽어 cross-DB 연결 0**. settlement의 `opslab.` 참조는 0이고 가드 테스트가 재유입을 빌드 실패로 막습니다. cross-DB JOIN(refunds×adjustments)은 settlement가 refund_id 집합을 만들고 order가 그 합계를 돌려주는 2단계로 분해했습니다."
> — 핵심: "Read-only Projection(공유 DB) → 이벤트 기반 CQRS(DB 분리)로 **진화시킨 과정**을 설명" = 최고 가점

## ③ Outbox 패턴 + Kafka, 비동기 이벤트 처리
> *"데이터 저장과 이벤트 발행의 정합성 확보 + 비동기 처리 구조"*

**열어서 보여줄 것**
- `shared-common/.../outbox/application/service/OutboxPublisherScheduler.java`
  - `@Scheduled(fixedDelay 2s)` 폴링 → `BATCH_SIZE=100` 배치 발행
  - **`FOR UPDATE SKIP LOCKED`** 로 claim → 멀티워커 수평 확장 (ShedLock 없이)
  - claim 리스(`claimed_at`, 1분) — 워커 죽으면 다른 워커가 회수
- `OutboxClaimConcurrencyIT.java` — 동시 claim 겹침 없음 검증

**한마디**
> "DB 트랜잭션 안에서 비즈니스 데이터와 outbox 레코드를 같이 커밋해 원자성을 보장하고, 별도 폴러가 SKIP LOCKED로 PENDING을 집어 배치 발행합니다. 폴러가 여러 인스턴스여도 같은 행을 안 가져가서 발행량이 인스턴스 수만큼 늘어납니다."

**꼬리질문: "왜 Outbox? @TransactionalEventListener나 직접 발행은?"**
> "DB 커밋과 Kafka 발행이 다른 시스템이라 둘 사이에 장애가 나면 정합성이 깨집니다(dual write 문제). Outbox는 발행 의도까지 같은 DB 트랜잭션에 남겨서, 커밋되면 반드시 언젠가 발행됨(at-least-once)을 보장합니다."

## ④ 3단계 멱등성 검증 (중복 정산 방지)
> *"요청·이벤트·정산 단계의 3단계 멱등성 체계"*

**열어서 보여줄 것 — 실제 3단의 정체**
- `PaymentEventKafkaConsumer.java` 주석(29~33줄)에 3단 명시:
  1. **`outbox_events.event_id UUID UNIQUE`** — 프로듀서 측 중복 발행 방지
  2. **`processed_events (consumer_group, event_id)` PK** — 컨슈머 측 재수신 방지 (82줄 `existsById` 체크)
  3. **`settlements.payment_id UNIQUE`** — 스키마 수준 최종 방어
- `SettlementIdempotencyIntegrationTest.java` — 같은 이벤트 2번 보내도 정산 1건 검증

**한마디 (3단을 정확히)**
> "발행 측은 event_id UNIQUE, 소비 측은 processed_events PK로 같은 이벤트 재수신을 거르고, 마지막으로 settlements.payment_id UNIQUE로 어떤 경로로 와도 한 결제당 정산 1건만 생기게 DB가 막습니다. 컨슈머에서 먼저 existsById로 체크하고, 경합으로 뚫려도 UNIQUE 제약이 최종 방어합니다."

**⚠️ 주의**: 이력서엔 "요청·이벤트·정산 단계"로 개념화했지만, 실제 구현은 위 **DB UNIQUE 3개 + 컨슈머 체크**. 물으면 위 3개로 답할 것.

**꼬리질문: "그래도 중복 뚫리는 경우는?"**
> "DLT(Dead Letter Topic)로 보냅니다. 독성 메시지(JSON 파싱 실패 등)는 재시도 없이 즉시 DLT, 일시적 예외는 ExponentialBackOff(2s×2, 3회) 후 DLT. 같은 파티션 후속 메시지 stall을 막기 위해서입니다."

## ⑤ 수수료·정산주기·Holdback 정산시점 스냅샷
> *"정책 변경 이후에도 재현성과 감사 추적성 확보"*

**열어서 보여줄 것**
- `Settlement.java` 필드: `commissionRate`, `holdbackRate`, `holdbackReleaseDate` — **정산 시점 값을 박제**(주석 "이력 보존")
- `SellerTier.java` — NORMAL 3.5%/T+7, VIP 2.5%/T+3, STRATEGIC 2.0%/T+1
- `HoldbackPolicy.java` — NORMAL 30%/30일, VIP 10%/14일, STRATEGIC 0%
- 마이그레이션 `V32__seller_tier_and_commission_rate.sql`, `V42__settlement_holdback.sql`

**한마디**
> "수수료율·정산주기·보류정책을 정산 생성 시점에 settlement 행에 스냅샷으로 저장합니다. 나중에 회사가 요율을 바꿔도 과거 정산은 그때 요율로 재계산되므로, 감사 시 '이 정산은 왜 이 금액?'을 그 시점 데이터로 그대로 재현할 수 있습니다."

**꼬리질문: "왜 등급을 참조 안 하고 스냅샷?"**
> "참조(FK)만 하면 등급/요율이 바뀌는 순간 과거 정산 금액 근거가 사라집니다. 돈·감사 도메인이라 '계산 당시의 입력값'을 불변으로 남기는 게 정확성과 추적성에 필수입니다."

## 🎯 면접 30초 요약 (엘리베이터 피치)
> "이커머스 정산을 헥사고날 + MSA로 설계한 프로젝트입니다. 결제 완료 이벤트를 Outbox+Kafka로 받아 정산을 자동 생성하는데, 돈이 틀리면 안 되니까 **멱등성 3단 방어로 중복 정산을 막고**, 수수료·보류 정책을 **정산 시점 스냅샷으로 박제**해 감사 추적성을 확보했습니다. order/settlement도 Strangler 패턴으로 이벤트 기반 CQRS + DB 물리 분리까지 끝냈고, 남은 opslab 직독은 대사 도구로 한정해 `@AuditCrossRead`로 명시했습니다 — **서빙은 완전 분리, 감사만 의도적으로 원천을 읽는다**가 핵심입니다."

## 가장 잘 받아칠 3대 지뢰
1. **"DB 공유면 MSA 아니잖아요?"** → 이제 **공유 아님**. Read-only Projection(공유 DB)에서 출발해 Strangler 6단계로 이벤트 기반 CQRS + `settlement_db` 물리 분리까지 완료. 서빙은 로컬 프로젝션, 대사는 self-totals(order 내부 API 비교)로 전환해 **settlement의 opslab 직독 = 0**(가드 테스트 강제). 잔여는 대사 배치의 settlement→order 동기 HTTP(가용성 결합)뿐, 핫패스는 무의존
2. **"3단계 멱등성이 뭔데요?"** → outbox event_id UNIQUE / processed_events PK / settlements payment_id UNIQUE
3. **"스냅샷 왜요?"** → 정책 변경 후에도 과거 정산 재현·감사 추적성
