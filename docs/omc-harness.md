# OMC 하네스 엔지니어링 — 기대효과와 측정 (KPI 5)

> oh-my-claudecode(OMC, 전역 플러그인)가 이 저장소에서 수행하는 하네스 엔지니어링의
> **기대효과를 5개 KPI로 정의하고, 무엇이 어디까지 측정 가능한지**를 못박는 정본 문서.
> 측정 도구는 전부 저장소 네이티브(`scripts/harness/*.mjs`, 플러그인 0 의존)로 배선되어
> OMC 미설치 환경에서도 우아하게 동작한다. 하네스 구성 자체의 정본은 [`HARNESS.md`](../HARNESS.md).

**Last updated:** 2026-07-22 (베이스라인 측정일)

## 1. 이중 하네스 구조

이 저장소에는 성격이 다른 두 하네스가 병존한다.

| 계층 | 소유 | 하는 일 | 강제 수단 |
|---|---|---|---|
| **전역 OMC** | `~/.claude/plugins/…/oh-my-claudecode` | 위임·모델 라우팅, 워크플로 루프(ralph/autopilot/ulw/team), 상태 영속(`.omc/`), 검증 분리(자기승인 금지) | UserPromptSubmit/PreToolUse 훅 주입(권고 계층) |
| **저장소 하네스** | `scripts/harness/` | 돈·경계·이력 불변식 차단, 도메인 규칙 스킬 라우팅, 문서 드리프트 게이트 | PreToolUse exit 2 + pre-commit + CI 3중(차단 계층) |

역할 경계는 HARNESS.md "이중 라우팅 경계" 조항이 정본이다: **워크플로 모드는 OMC 소유,
도메인·절차 규율은 저장소 하네스가 정본**(지침 우선순위: 프로젝트 문서 > 플러그인 주입).

## 2. 측정의 구조적 한계 2가지 (KPI 설계의 전제)

- **예방의 역설** — 가드가 완벽할수록 차단 건수는 0에 수렴하므로, 건수만으로는
  "죽은 규칙"과 "완전 예방"을 구분할 수 없다. → **카나리아**(규칙별 최소 위반 픽스처를
  주입해 차단되는지 검사)로 0의 의미를 분리한다(KPI-1).
- **반사실 부재** — "OMC 덕분에 품질이 좋아졌다"는 OMC 없는 동일 작업과의 비교로만
  성립한다. `DISABLE_OMC=1` 킬 스위치로 A/B 세션 설계가 가능하지만, 표본이 작고 태스크
  이질성이 커서 **유의성 검정이 아닌 방향성 확인**이 상한이다. KPI-3/4는 인과가 아니라
  추이를 본다.

### KPI별 기대효과 매핑 (OMC 하네스 엔지니어링 축 → 지표)

- **KPI-1 · 가드 유효성** ← "결정적인 것은 기계로 강제" 축. OMC 권고 계층이 아닌 저장소
  차단 계층의 지표지만, OMC 권고(스킬 리마인더)가 작동할수록 차단 발화는 0에 수렴해야
  한다 — 카나리아 PASS + 차단 0회 조합이 이상 상태다.
- **KPI-2 · 라우터 순응률** ← "권장의 기계화" 축. 라우터가 주입한 규칙 스킬 리마인더를
  모델이 실제로 소비하는 비율. 낮아지면 리마인더 문구·타이밍을 조정한다.
- **KPI-3 · 워크플로 완주율** ← OMC 위임 오케스트레이션(워커 분배·태스크 완료) 축.
  `.omc/state/mission-state.json` 의 미션 종결 상태를 읽는다(읽기 전용 관측 —
  `.omc` 는 OMC 소유이므로 이 지표는 OMC 미설치 시 N/A 로 강등된다).
- **KPI-4 · 재작업률** ← 검증 분리(자기승인 금지·verifier) 축의 **대리지표**. 검증 분리가
  작동하면 결함이 커밋 전에 걸러져 후속 fix 커밋이 줄어야 한다. 인과 분리는 불가 —
  추이와 A/B 로만 해석한다.
- **KPI-5 · 상주 컨텍스트 비용** ← 컨텍스트 예산 엔지니어링(온디맨드 스킬 분리) 축.
  상주 CLAUDE.md 가 비대해지면 OMC·저장소 하네스가 공유하는 "필요할 때만 로드" 설계가
  무너지고 있다는 신호다.

## 4. 측정 배선 (저장소 네이티브 3종)

| 도구 | 커버 KPI | 소스 | 성격 |
|---|---|---|---|
| `telemetry-report.mjs` | 1, 2 | `.claude/harness/logs/*.jsonl` + 규칙별 카나리아(in-process `scanText`) | 리포트 — 항상 exit 0 |
| `session-metrics.mjs` | 3, 4 | `.omc/state/sessions`·`mission-state.json`(읽기 전용) + `git log` | 리포트 — 항상 exit 0, `.omc` 부재 시 N/A |
| `harness-audit.mjs` | 5 | git 추적 파일(CLAUDE.md·`.claude/skills/**/SKILL.md`) 바이트 | 게이트에 얹힌 info 라인 — KPI-5 는 실패 사유 아님 |

설계 불변식: **측정은 게이트를 깨뜨리지 않는다.** 리포트 2종은 어떤 실패에도 exit 0,
audit 의 KPI-5 는 정보 라인이다. 세션↔커밋 1:1 귀속은 하지 않는다(세션이 병렬로 겹침 —
일 단위 병렬 추이만 제시).

## 5. 운용

- **주기**: 주 1회 리포트 2종 실행 + 수치가 목표를 이탈하면 원인 조사(순응률 하락 →
  리마인더 문구, 재작업률 상승 → 검증 분리 우회 여부). audit 은 pre-commit·CI 에서 상시.
- **A/B 실험(선택)**: 정형 작업 클래스(예: P0 시드류)를 `DISABLE_OMC=1` 세션과 교차 배정,
  게이트 첫 통과 여부·후속 fix 수를 비교. 방향성 확인용 — 결론 과장 금지.
- **베이스라인 갱신**: 본 문서의 수치는 측정일 스냅샷이다. 재측정 시 표의 베이스라인과
  Last updated 를 함께 갱신한다.
