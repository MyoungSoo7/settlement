# 디자인 패턴 — 개념 정리 + Lemuel 프로젝트 적용 사례

이 문서는 두 부분으로 구성된다.
1. **디자인 패턴 개념 정리** — 디자인 패턴이 무엇이고 왜 쓰는지, GoF 분류
2. **이 프로젝트에 실제로 사용된 패턴** — 코드 근거와 함께 카탈로그화

> 근거: 헥사고날 포트 인터페이스(in/out) 100여 개, `PgRouter`, `PaymentGatewayAdapter`, `TwoTierCache`, `CacheInvalidationPublisher`, `Settlement` 상태 머신, Outbox 리포지토리 등 핵심 클래스 정독.

---

## 1부. 디자인 패턴이란

**디자인 패턴(Design Pattern)** 은 소프트웨어 설계에서 반복적으로 등장하는 문제에 대한 검증된 해결 템플릿이다. 코드를 그대로 복사하는 라이브러리가 아니라, "이런 상황엔 이렇게 구조를 잡는다"는 **설계 어휘(vocabulary)** 다.

**왜 쓰는가**
- 검증된 구조 재사용 → 시행착오 감소
- 공통 어휘로 팀 커뮤니케이션 비용 감소 ("여기 Strategy 로 빼자")
- 변경에 강한 구조(OCP·DIP) → 유지보수성 향상

**GoF(Gang of Four) 3대 분류**

| 분류 | 목적 | 대표 패턴 |
|------|------|-----------|
| **생성(Creational)** | 객체 생성 방식 캡슐화 | Factory Method, Builder, Singleton, Abstract Factory |
| **구조(Structural)** | 객체·클래스 조합 구조 | Adapter, Decorator, Proxy, Composite, Facade |
| **행위(Behavioral)** | 객체 간 책임·협력 | Strategy, Observer, State, Template Method, Command, Chain of Responsibility |

> 주의: 패턴은 목적이 아니라 수단이다. 단순한 문제에 패턴을 억지로 끼우면 오히려 복잡도가 증가한다(over-engineering). 이 프로젝트는 **헥사고날 아키텍처를 떠받치는 데 필요한 패턴 위주**로 사용한다.

## 2부. 이 프로젝트에 사용된 디자인 패턴

### 아키텍처 패턴 (가장 상위)

#### ◆ Ports & Adapters (헥사고날 아키텍처)
프로젝트 전체의 뼈대. 도메인을 중심에 두고, 외부와의 모든 접점을 **포트(인터페이스)** 로 추상화하고 **어댑터** 로 구현한다.

- **인바운드 포트** (`application/port/in`): `CreateOrderUseCase`, `RefundPaymentPort`, `ExecutePayoutUseCase` … 애플리케이션이 제공하는 기능 계약.
- **아웃바운드 포트** (`application/port/out`): `SaveOrderPort`, `PgClientPort`, `PublishEventPort`, `FirmBankingPort` … 애플리케이션이 외부에 요구하는 계약.
- **어댑터** (`adapter/in/web`, `adapter/out/persistence` 등): 포트의 실제 구현(REST, JPA, Kafka, PDF).
- 효과: 도메인은 프레임워크/DB/PG 를 **모른다**. 의존성 역전(DIP)으로 테스트·교체가 쉬움. ArchUnit 으로 방향 강제.

#### ◆ Transactional Outbox
**위치**: `shared-common/.../outbox`, `payment/adapter/out/event/OutboxBackedEventPublisher`

DB 트랜잭션과 메시지 발행의 원자성 문제를 해결. 도메인 변경 + `outbox_events` INSERT 를 같은 트랜잭션으로 묶고, 별도 poller 가 Kafka 로 발행. (이벤트 유실 0)

#### ◆ CQRS 성격의 Read-Model Projection
**위치**: `settlement-service/.../adapter/out/readmodel`

settlement-service 가 order-service 코드를 import 하지 않고 `@Immutable` JPA 엔티티(`SettlementPaymentReadModel` 등)로 읽기 전용 조회. 쓰기 모델(order)과 읽기 모델(settlement) 분리.

---

### 생성 패턴 (Creational)

#### ◆ Factory Method (정적 팩토리)
- `Settlement.createFromPayment(...)` — 생성과 동시에 검증 + 수수료/순액 계산.
- `EcommerceCategory.createRoot()/createChild()` — 깊이 규칙·순환 참조 검증 포함.
- `ReconciliationDiscrepancy.newDiscrepancy(...)`, `HoldbackPolicy.forTier(tier)`.
- 효과: 생성 로직과 불변식을 한곳에 모아 잘못된 객체 생성 차단.

#### ◆ Builder
- `Caffeine.newBuilder()`, `Bucket.builder()`, `Bandwidth.builder()`, `Counter.builder()` (`RateLimitRegistry`, `PgRouter`).
- 효과: 선택적 파라미터가 많은 객체를 가독성 있게 조립.

#### ◆ Singleton (Spring 관리 빈)
- `@Component`/`@Service`/`@Configuration` 빈은 기본 싱글톤 스코프 — `PgRouter`, `RateLimitRegistry`, 각종 UseCase.

---

### 구조 패턴 (Structural)

#### ◆ Adapter
- `adapter/out/external`·`adapter/out/pg` 의 PG 클라이언트, JPA 리포지토리 어댑터가 외부 API/프레임워크를 도메인 포트 형태로 변환.
- `PaymentGatewayAdapter` 구현체들이 각 PG 의 상이한 API 를 공통 컨트랙트로 흡수.

#### ◆ Decorator / Proxy + Near-Cache
**위치**: `TwoTierCache extends AbstractValueAdaptingCache`

Spring `Cache` 인터페이스를 구현하면서 L1(Caffeine) + L2(Redis) 2계층을 감싼다. 조회 시 L1 miss → L2 조회 후 L1 승격(near-cache), L2 장애는 try/catch 로 graceful degrade. 기존 캐시 추상화에 동작을 덧씌우는 전형적 데코레이터.

#### ◆ Composite (트리)
- `EcommerceCategory` 가 `List<EcommerceCategory> children` 를 갖는 재귀 트리 구조(부분-전체 계층).

---

### 행위 패턴 (Behavioral)

#### ◆ Strategy
- `PaymentGatewayAdapter` — PG별 구현이 곧 전략. `PgRouter` 가 런타임에 적절한 전략 선택.
- 수수료 전략: `SellerTier.rate()` 주입으로 정산 수수료율을 티어별로 교체.
- 헥사고날 아웃바운드 포트 자체가 "교체 가능한 전략" 으로 동작(예: `FirmBankingPort`, `SendEmailPort`).

#### ◆ Chain of Responsibility (Fallback 체인)
**위치**: `PgRouter.selectFor()`

고액 우선 PG → 결제수단별 1순위 → fallback 체인 순으로 후보를 넘기며 처리 가능한(healthy·지원) 첫 어댑터가 책임을 맡음. Kafka 에러 핸들러의 재시도→DLT 흐름도 유사.

#### ◆ State (상태 머신)
**위치**: `Settlement`, `Order`, `Payment`, `Payout`, `Chargeback` 도메인

enum 상태 + 전이 메서드(`startProcessing()`, `complete()`, `fail()`, `retry()`, `cancel()`)로 허용된 전이만 수행, 위반 시 `IllegalStateException`. (예: DONE 정산은 불변)

#### ◆ Observer / Publish-Subscribe
- `CacheInvalidationPublisher` — 한 인스턴스가 캐시를 변경하면 Pub/Sub 으로 타 인스턴스 L1 무효화 전파.
- Kafka 이벤트(payment.captured 등) 발행/구독 — 서비스 간 느슨한 결합.

#### ◆ Command
- 각 `*UseCase` + `*Command`(예: `CreatePaymentCommand`) — 요청을 객체로 캡슐화해 실행. 인바운드 포트가 커맨드 핸들러 역할.

#### ◆ Template Method
- Spring Batch `ItemReader → ItemProcessor → ItemWriter` 청크 처리 골격, `AbstractValueAdaptingCache` 의 `lookup`/`toStoreValue` 훅 재정의.
- 참고: `docs/design-pattern/teplate-pattern.md`.

## 3) 패턴 요약 표

| 분류 | 패턴 | 대표 적용처 |
|------|------|-------------|
| 아키텍처 | Ports & Adapters | 전 모듈 (`port/in`, `port/out`, `adapter`) |
| 아키텍처 | Transactional Outbox | `outbox_events` + poller |
| 아키텍처 | CQRS Read-Model | `adapter/out/readmodel` |
| 생성 | Factory Method | `Settlement.createFromPayment`, `EcommerceCategory.createChild` |
| 생성 | Builder | Bucket4j / Caffeine / Micrometer builder |
| 생성 | Singleton | Spring 빈 |
| 구조 | Adapter | PG·JPA·Kafka 어댑터 |
| 구조 | Decorator/Proxy | `TwoTierCache` |
| 구조 | Composite | `EcommerceCategory` 트리 |
| 행위 | Strategy | `PaymentGatewayAdapter`, `SellerTier` 수수료 |
| 행위 | Chain of Responsibility | `PgRouter` fallback 체인 |
| 행위 | State | `Settlement`/`Payment`/`Order` 상태 머신 |
| 행위 | Observer/Pub-Sub | `CacheInvalidationPublisher`, Kafka 이벤트 |
| 행위 | Command | `*UseCase` + `*Command` |
| 행위 | Template Method | Spring Batch, `AbstractValueAdaptingCache` |

## 정리

- 이 프로젝트의 패턴 사용은 **헥사고날 아키텍처(Ports & Adapters)** 를 중심으로 자연스럽게 파생된다 — 포트는 Strategy/DIP, UseCase 는 Command, 어댑터는 Adapter.
- MSA 경계 유지를 위해 **Transactional Outbox + Read-Model Projection** 을 핵심 아키텍처 패턴으로 채택.
- 도메인 불변식은 **Factory Method(생성 검증) + State(전이 검증)** 로 보호.
- 인프라 관심사는 **Decorator(2-tier 캐시) + Chain of Responsibility(PG fallback) + Observer(캐시 무효화/이벤트)** 로 처리.
- 관련 문서: `docs/design-pattern/teplate-pattern.md`, `docs/design-pattern/callback.md`, `docs/adr/0001-hexagonal-architecture.md`, `docs/adr/0003-transactional-outbox-pattern.md`.
