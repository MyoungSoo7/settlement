# 동기 연동(Synchronous Integration) 분석

이 문서는 Lemuel 플랫폼의 **동기 연동**(요청-응답이 같은 호출 스택에서 블로킹으로 완결되는 연동)을
코드 근거로 정리한다. Kafka/Outbox 기반 **비동기** 연동(`PaymentCaptured` → Settlement 생성 등)은
이 문서의 대상이 아니며, [`docs/카프카.md`](./카프카.md) 및 CLAUDE.md 의 "이벤트 흐름" 절을 참조한다.

> **한눈에 보는 결론**
> 이 프로젝트의 동기 연동은 4갈래다. ① 외부 PG(Toss) **HTTP 호출**, ② API Gateway **HTTP 라우팅**,
> ③ settlement→order 의 **DB Read-only Projection**(같은 DB를 SQL/JPA로 직접 읽음),
> ④ Read/Write **DataSource 라우팅**(opt-in). 반면 **두 서비스 간 직접 REST 호출은 존재하지 않는다** —
> 서비스 경계는 동기 DB 읽기(read-model) + 비동기 이벤트로만 넘는다.

---

## 0. 동기 vs 비동기 연동 지도

| # | 연동 | 방식 | 프로토콜 | 위치(모듈) | 동기? |
|---|------|------|----------|-----------|-------|
| 1 | 주문서비스 → Toss Payments | 결제 승인 confirm | HTTPS REST | order-service | **동기** |
| 2 | 클라이언트 → 백엔드 서비스 | Gateway 라우팅 | HTTP (WebFlux) | gateway-service | **동기**(프록시) |
| 3 | settlement → order DB | 정산 대상 결제 조회 | JPA `@Immutable` / JdbcTemplate | settlement-service | **동기**(DB read) |
| 4 | 읽기/쓰기 분리 | Read Replica 라우팅 | JDBC | shared-common | **동기**(opt-in) |
| 5 | order → settlement | 결제완료 → 정산생성 | Kafka + Outbox | 양 서비스 | 비동기 |
| 6 | PG 정산파일 대사 | 운영자 업로드 파일 파싱 | CSV(파일) | settlement-service | 동기(외부 API 아님) |

> 핵심: settlement-service 의 `build.gradle.kts` 에 `implementation(project(":order-service"))` 가
> **없다.** 즉 두 서비스는 컴파일타임 코드 의존이 0이고, 런타임에 settlement 가 order 의 데이터를
> 얻는 유일한 동기 경로는 **같은 물리 DB 테이블을 read-only 로 직접 읽는 것**(아래 3절)이다.

## 1. 외부 PG 연동 — Toss Payments (동기 HTTP)

가장 전형적인 동기 연동. 사용자가 토스 결제창에서 결제를 마치면 프런트가 `paymentKey` 를 받아
백엔드에 confirm 을 요청하고, 백엔드는 **Toss 서버에 HTTP POST 로 최종 승인을 검증**한다.
이 호출이 성공해야만 결제 레코드가 CAPTURED 가 되고 정산이 트리거된다.

### 1.1 진입점 — `PaymentController`

`order-service/.../payment/adapter/in/api/PaymentController.java`

```java
@PostMapping("/toss/confirm")                       // POST /payments/toss/confirm
public ResponseEntity<PaymentResponse> confirmTossPayment(@Valid @RequestBody TossPaymentConfirmRequest request) {
    PaymentDomain paymentDomain = tossPaymentService.confirmTossPayment(
            request.getDbOrderId(), request.getPaymentKey(),
            request.getTossOrderId(), request.getAmount());
    return ResponseEntity.ok(new PaymentResponse(paymentDomain));
}
```

- 단건: `POST /payments/toss/confirm`
- 장바구니 일괄: `POST /payments/toss/cart/confirm`

### 1.2 실제 외부 호출 — `TossPaymentService`

`order-service/.../payment/application/TossPaymentService.java`

HTTP 클라이언트는 **Spring `RestTemplate`** 이며, **Spring Boot 4 에서 `RestTemplateBuilder` 가
제거**되어 `SimpleClientHttpRequestFactory` 로 타임아웃을 직접 구성한다.

```java
SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
factory.setConnectTimeout(Duration.ofSeconds(3));   // 연결 타임아웃 3s
factory.setReadTimeout(Duration.ofSeconds(5));      // 응답 타임아웃 5s
this.restTemplate = new RestTemplate(factory);
```

> **타임아웃이 동기 연동의 1차 방어선.** read timeout 이 없으면 Toss 응답 지연 시 톰캣 워커 스레드가
> 무한 점유되어 풀이 고갈된다. connect 3s / read 5s 로 상한을 둔다.

승인 호출 본체 — Basic 인증(`secretKey:` Base64) + JSON 바디:

```java
@CircuitBreaker(name = "tossPg", fallbackMethod = "tossFallback")
@Retry(name = "tossPg")
public void callTossConfirmApi(String paymentKey, String tossOrderId, Long amount) {
    String encoded = Base64.getEncoder().encodeToString((secretKey + ":").getBytes(UTF_8));
    HttpHeaders headers = new HttpHeaders();
    headers.setContentType(MediaType.APPLICATION_JSON);
    headers.set(HttpHeaders.AUTHORIZATION, "Basic " + encoded);
    // body: { paymentKey, orderId, amount }
    ResponseEntity<Map> response = restTemplate.postForEntity(tossApiUrl, entity, Map.class);
    ...
}
```

설정값 (`order-service/src/main/resources/application.yml`):

```yaml
toss:
  secret-key: ${TOSS_SECRET_KEY}                              # 환경변수 주입(시크릿)
  api-url: https://api.tosspayments.com/v1/payments/confirm
```

### 1.3 회복탄력성 (Resilience4j)

동기 외부 호출의 장애 격리는 **Resilience4j CircuitBreaker + Retry** 로 한다. 핵심 설계는
**"4xx(PG 비즈니스 오류)는 서킷·재시도 양쪽에서 제외"** 한다는 점이다 — 결제키 만료 같은 오류는
재시도해도 똑같이 실패하고 서킷만 잘못 열리기 때문이다.

`application.yml` 발췌:

```yaml
resilience4j:
  circuitbreaker:
    configs:
      pgDefault:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 20                 # 최근 20건
        minimumNumberOfCalls: 10
        failureRateThreshold: 50              # 실패 50%↑ → OPEN
        waitDurationInOpenState: 30s          # OPEN → HALF_OPEN
        ignoreExceptions:
          - org.springframework.web.client.HttpClientErrorException   # 4xx 제외
    instances: { tossPg, kcpPg, nicePg, inicisPg }    # PG 별 독립 인스턴스(격벽)
  retry:
    configs:
      pgDefault:
        maxAttempts: 3
        waitDuration: 500ms
        enableExponentialBackoff: true        # 500ms → 1s → 2s
        retryExceptions:                      # 5xx / I/O 만 재시도
          - org.springframework.web.client.ResourceAccessException
          - org.springframework.web.client.HttpServerErrorException
          - java.io.IOException
```

- **서킷 OPEN / 재시도 소진** 시 `tossFallback(...)` 으로 위임 → 사용자에겐 "일시 장애" 메시지,
  운영에선 Alertmanager/Slack 연계 지점.
- PG 별로 `tossPg / kcpPg / nicePg / inicisPg` 독립 인스턴스를 둬 **한 PG 장애가 다른 PG 로
  전이되지 않도록 격벽(bulkhead)** 처리.

> 관련 ADR: [`docs/adr/0006-resilience4j-tosspg.md`](./adr/0006-resilience4j-tosspg.md)

### 1.4 다중 PG 라우팅 (헥사고날 + 동기 위임)

도메인은 PG가 무엇인지 모르고 `PgClientPort` 단일 포트만 본다. 그 아래에서 라우팅이 일어난다.

```
PaymentDomain / UseCase
      │  (PgClientPort)
      ▼
RoutingPgClientAdapter(@Primary)          ← order-service/.../payment/adapter/out/external
      │  router.selectFor(amount, method) / resolveByTransactionId(txnId)
      ▼
PgRouter                                   ← order-service/.../payment/adapter/out/pg
      │  선택: ①고액거래 우선PG ②결제수단별 1순위 ③fallback 체인 (unhealthy 회피)
      ▼
PaymentGatewayAdapter 구현체 (TOSS / KCP / NICE / INICIS)
```

- `authorize` → 금액·결제수단·health 로 PG 선택. healthy 한 어댑터만 후보(`isHealthy()` = 서킷 ≠ OPEN).
- `capture` / `refund` → **거래 ID prefix 로 원래 처리한 PG 를 식별**(`PaymentGateway.fromTransactionId`)
  → DB 컬럼 추가 없이 동일 PG 로 정확히 라우팅.
- 라우팅 결정마다 Prometheus 카운터 `pg.routing.requests{provider,reason,method}` 증가.

> 주의: 현재 각 `*PgAdapter`(예: `TossPgAdapter`)의 `authorize/capture/refund` 는 **Mock 동작**이며,
> 실제 Toss 외부 호출은 `TossPaymentService.callTossConfirmApi` 가 별도로 담당한다. 운영 확장 시
> 각 어댑터 내부에서 RestTemplate/WebClient 를 호출하도록 합치는 구조다(`TossPgAdapter` 주석 참조).

## 2. API Gateway — 동기 HTTP 라우팅

`gateway-service` 는 **Spring Cloud Gateway(WebFlux/reactive)** 기반 단일 진입점이다. 클라이언트의
모든 요청을 받아 경로 패턴으로 뒤쪽 서비스에 **동기 프록시**한다. Java 코드는 부트스트랩 클래스
(`GatewayServiceApplication.java`) 하나뿐이고, 라우팅은 전부 **선언적 YAML**로 정의된다.

`gateway-service/src/main/resources/application.yml`

```yaml
spring:
  main:
    web-application-type: reactive            # WebFlux 고정
  cloud:
    gateway:
      server:
        webflux:
          routes:
            - id: order-service-orders
              uri: ${ORDER_SERVICE_URI:http://localhost:8088}
              predicates:
                - Path=/api/orders/**,/api/payments/**,/api/refunds/**,/api/products/**,
                       /api/categories/**,/api/coupons/**,/api/reviews/**,/api/users/**,
                       /api/auth/**,/reservations/**,/admin/categories/**,/admin/pg/**,...
            - id: settlement-service
              uri: ${SETTLEMENT_SERVICE_URI:http://localhost:8082}
              predicates:
                - Path=/api/settlements/**,/api/reconciliation/**,/api/reports/**,
                       /api/ledger/**,/admin/payouts/**,/admin/chargebacks/**,
                       /admin/pg-reconciliation/**,/admin/dlq/**,/admin/reconciliation/**
```

| 라우트 | 대상 | 포트 | 대표 경로 |
|--------|------|------|-----------|
| `order-service-orders` | order-service | 8088 | `/api/orders`, `/api/payments`, `/api/products`, `/api/users`, `/reservations` ... |
| `settlement-service` | settlement-service | 8082 | `/api/settlements`, `/api/reconciliation`, `/api/reports`, `/api/ledger`, `/admin/payouts` ... |

- 대상 URI 는 `ORDER_SERVICE_URI` / `SETTLEMENT_SERVICE_URI` 환경변수로 주입(쿠버네티스 서비스명 등).
- Gateway 포트는 8080, `shutdown: graceful`, actuator health/prometheus 노출.
- **인증(JWT)** 자체는 각 서비스의 `shared-common.common.config.jwt` 필터에서 검증한다. 게이트웨이는
  현재 순수 라우팅 역할(별도 커스텀 인증 GlobalFilter 자바 코드는 없음).

## 3. Read-only Projection — settlement → order DB 동기 조회 (핵심)

settlement-service 가 order-service 의 **코드를 import 하지 않으면서** 정산 대상 결제 데이터를 얻는
동기 경로. 두 서비스가 **물리적으로 같은 PostgreSQL DB** 를 공유하되, settlement 쪽에 동일 테이블을
가리키는 **`@Immutable` 읽기 전용 엔티티**를 따로 정의해 읽는다(쓰기는 절대 안 함).

> 이것이 "두 서비스 간 직접 REST 호출이 없는" 이유다. settlement 는 REST 로 order 에 물어보는 대신,
> 같은 DB 의 `payments` 테이블을 자기 read-model 로 직접 SELECT 한다. → 모듈 코드 의존성 0.

### 3.1 read-model 엔티티들

`settlement-service/.../settlement/adapter/out/readmodel/`

| read-model | 매핑 테이블 | 읽는 컬럼 |
|------------|------------|----------|
| `SettlementPaymentReadModel` | `payments` | id, order_id, amount, refunded_amount, status, captured_at ... |
| `SettlementOrderReadModel` | `orders` | 주문 정보 |
| `SettlementUserReadModel` | `users` | email 만 |
| `SettlementProductReadModel` | `products` | name 만 |

```java
@Entity
@Immutable                          // Hibernate: UPDATE/DELETE 금지, read-only 보장
@Table(name = "payments")
@Getter
public class SettlementPaymentReadModel {
    @Id private Long id;
    @Column(name = "order_id") private Long orderId;
    private BigDecimal amount;
    @Column(name = "refunded_amount") private BigDecimal refundedAmount;
    private String status;
    @Column(name = "captured_at") private LocalDateTime capturedAt;
    ...
}
```

### 3.2 조회 어댑터 — `CapturedPaymentsAdapter`

정산 배치/서비스가 "특정 영업일에 CAPTURED 된 결제"를 가져오는 동기 호출.

```java
@Component
@RequiredArgsConstructor
public class CapturedPaymentsAdapter implements LoadCapturedPaymentsPort {
    private final SettlementPaymentReadModelRepository paymentReadRepository;   // Spring Data JPA

    @Override
    public List<CapturedPaymentInfo> findCapturedPaymentsByDate(LocalDate settlementDate) {
        var payments = paymentReadRepository.findByCapturedAtBetweenAndStatus(
                settlementDate.atStartOfDay(),
                settlementDate.plusDays(1).atStartOfDay(), "CAPTURED");
        return payments.stream().map(p -> new CapturedPaymentInfo(...)).toList();
    }
}
```

### 3.3 SQL 직접 조회 변형 — `InternalPaymentsForReconJdbcAdapter`

PG 대사 등 집계성 조회는 JPA 대신 **`JdbcTemplate`** 으로 `payments` 테이블을 직접 SELECT 한다
(같은 read-model 철학, 더 가벼운 SQL).

```java
@Repository
public class InternalPaymentsForReconJdbcAdapter implements LoadInternalPaymentsForReconciliationPort {
    private final JdbcTemplate jdbcTemplate;
    public List<InternalPaymentRow> loadByCapturedDate(LocalDate date) {
        String sql = """
            SELECT id, pg_transaction_id, amount, refunded_amount, captured_at::date AS captured_date
              FROM opslab.payments
             WHERE pg_transaction_id IS NOT NULL
               AND status IN ('CAPTURED','REFUNDED')
               AND captured_at::date = ?
            """;
        return jdbcTemplate.query(sql, (rs, n) -> new InternalPaymentRow(...), date);
    }
}
```

> **트레이드오프**: 같은 DB 를 공유하므로 진정한 "Database per Service" 는 아니다(모듈러 모놀리스 →
> 물리 MSA 전환 중). 운영 확장 경로는 Outbox+Kafka 로 settlement 자체 테이블에 복제하는 것
> (`SettlementPaymentReadModel` 클래스 주석에 명시). 자세한 맥락은 [`docs/msa.md`](./msa.md).

## 4. Read/Write 분리 — DataSource 라우팅 (opt-in, 동기 JDBC)

읽기 TPS 를 레플리카로 오프로드하는 **선택적** 동기 연동. 기본 비활성이며
`app.datasource.read-replica.enabled=true` 일 때만 켜진다.

`shared-common/.../common/config/ReadReplicaDataSourceConfig.java`

```java
@Configuration
@ConditionalOnProperty(name = "app.datasource.read-replica.enabled", havingValue = "true")
public class ReadReplicaDataSourceConfig {
    @Bean @Primary
    public DataSource dataSource(HikariDataSource write, HikariDataSource read) {
        RoutingDataSource routing = new RoutingDataSource();
        routing.setTargetDataSources(Map.of(Route.WRITE, write, Route.READ, read));
        routing.setDefaultTargetDataSource(write);
        return new LazyConnectionDataSourceProxy(routing);   // ★ 커넥션 획득 지연이 핵심
    }
    static class RoutingDataSource extends AbstractRoutingDataSource {
        @Override protected Object determineCurrentLookupKey() {
            return TransactionSynchronizationManager.isCurrentTransactionReadOnly()
                    ? Route.READ : Route.WRITE;
        }
    }
}
```

- **라우팅 키**: 현재 트랜잭션의 `readOnly` 플래그. `@Transactional(readOnly = true)` → READ(레플리카),
  그 외 → WRITE(프라이머리).
- **`LazyConnectionDataSourceProxy` 가 필수**: 실제 커넥션 획득을 첫 쿼리까지 지연시켜야 readOnly
  플래그가 결정된 뒤 라우팅 키가 평가된다. 없으면 트랜잭션 시작 시점에 커넥션을 잡아 항상 프라이머리로 간다.
- 코드베이스 전반의 조회 서비스들이 이미 `@Transactional(readOnly = true)` 를 달고 있어
  (예: `GetOrderService`, `GetProductService`, `SettlementQueryService`, `GetLedgerService` ...)
  플래그만으로 레플리카 라우팅이 적용된다.

> 상세 TPS 개선 맥락: [`docs/tps.md`](./tps.md) (PgBouncer, JDBC 배치, Outbox 비동기 배치 등과 함께).

## 5. PG 정산파일 대사 — 파일 기반 동기 처리

settlement-service 의 `pgreconciliation` 패키지. **외부 API 동기 호출이 아니라**, 운영자가 PG사에서
내려받은 정산 CSV 파일을 업로드하면 그 `InputStream` 을 동기로 파싱·대조하는 흐름이다. (PG사 조회 API
를 직접 호출하지 않으므로 외부 동기 연동으로 분류하지 않되, 동기 처리 파이프라인으로 함께 기록.)

`ReconcilePgFileService.reconcile(...)`:

1. `ParsePgFilePort` 로 업로드 파일 파싱 (`CsvPgFileParserAdapter`)
2. `LoadInternalPaymentsForReconciliationPort` 로 같은 영업일 내부 결제 조회 (3.3 의 JDBC read-model)
3. `PgReconciliationMatcher` 로 `pg_transaction_id` 1:1 매칭 → ROUNDING_DIFF 자동 보정, 그 외 PENDING 적재
4. Run + Discrepancy 단일 트랜잭션 저장
5. 차이 종류별 Prometheus 카운터(`pg.reconciliation.discrepancies`)

엔드포인트: `/admin/pg-reconciliation/**`, `/admin/reconciliation/**` (게이트웨이 → settlement 라우팅).

## 6. 동기 연동 신뢰성 체크리스트

| 관심사 | 적용 위치 | 메커니즘 |
|--------|----------|----------|
| 타임아웃 | Toss 호출 | connect 3s / read 5s (`SimpleClientHttpRequestFactory`) |
| 재시도 | Toss 호출 | Resilience4j `@Retry` (3회, exp backoff, 5xx·I/O 만) |
| 회로 차단 | PG 호출 | Resilience4j `@CircuitBreaker` (실패 50% → 30s OPEN) |
| 격벽(bulkhead) | 다중 PG | PG 별 독립 CB 인스턴스(tossPg/kcpPg/...) |
| 오류 분류 | Toss 호출 | 4xx(비즈니스)는 서킷·재시도 제외, 즉시 전파 |
| 읽기 부하 분산 | 전 서비스 | Read Replica 라우팅(opt-in) + `readOnly` 트랜잭션 |
| 모듈 격리 | settlement | Read-only Projection(`@Immutable`)으로 코드 의존 0 |
| 관측성 | PG/대사 | Micrometer 카운터 → Prometheus → Grafana 알람 |

## 7. 참고 — 비동기와의 경계

동기 연동과 비동기 연동이 만나는 지점은 **결제 CAPTURED 직후**다.

```
[order-service]  Toss confirm(동기 HTTP) → payments=CAPTURED (DB tx)
                                          └─ outbox_events INSERT (PaymentCaptured)
                                                   ↓ (비동기: 폴러 → Kafka)
[settlement-service] PaymentEventKafkaConsumer → Settlement 생성 (비동기)
                     이후 정산 배치가 read-model(동기 DB 읽기)로 CAPTURED 결제 집계
```

즉 **서비스 간 상태 전이는 비동기 이벤트로**, **정산 계산에 필요한 원천 데이터 조회는 동기 DB 읽기로**
역할이 나뉜다. 직접 REST 동기 호출(order ↔ settlement)은 의도적으로 두지 않았다 — 강결합·장애전파를
피하기 위함이며, 이는 MSA 설계 원칙(2.x in `docs/msa.md`)과 일치한다.
