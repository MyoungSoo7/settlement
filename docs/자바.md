# Java — 본질·버전별 쟁점 + Lemuel(Java 25) 분석

본문(자바의 특징/버전 쟁점/이 프로젝트의 Java 25 이점) + **부록 A. JVM** + **부록 B. GC** 로 구성된다.

> 근거: `build.gradle.kts`(toolchain 25), `Dockerfile`(temurin:25), `application.yml`(virtual threads + JEP 491 주석), records 69개·text block 10개 파일, JAVA_OPTS GC 옵션 정독.

---

## 1) 자바의 특징과 본질

**Java** 는 "Write Once, Run Anywhere(WORA)"를 목표로 한 객체지향 언어 + 런타임 플랫폼이다. 본질은 **언어 + JVM + 표준 라이브러리** 세 층의 결합.

### 핵심 특징

| 특징 | 의미 |
|------|------|
| **플랫폼 독립성** | 소스 → **바이트코드(.class)** 컴파일 → 어떤 OS의 JVM에서도 실행. JVM이 추상화 계층 |
| **JIT 컴파일** | 바이트코드를 실행 중 핫스팟을 네이티브 코드로 변환(C1/C2) → 인터프리터+컴파일 하이브리드 |
| **자동 메모리 관리(GC)** | 개발자가 free 하지 않음. GC가 도달 불가능 객체 회수 → 메모리 안전 |
| **강한 정적 타입** | 컴파일 타임 타입 검사 → 런타임 오류 감소 |
| **객체지향 + 점진적 다중 패러다임** | 클래스 기반 OOP에 람다/스트림(함수형), record(데이터 지향) 추가 |
| **풍부한 생태계** | Spring, 방대한 라이브러리, 성숙한 도구(프로파일러·APM) |
| **하위 호환성** | 오래된 바이트코드도 신버전 JVM에서 대부분 동작 — 엔터프라이즈 채택의 근간 |

### 본질적 트레이드오프
- **장점**: 안정성·생산성·성숙한 런타임·관측성·거대 생태계 → **장수명 엔터프라이즈/금융 시스템에 강함**(이 프로젝트가 정산 도메인에 Java를 쓴 이유와 직결).
- **비용**: JVM 워밍업(startup·JIT warm-up), 객체 헤더/GC로 인한 메모리 오버헤드 → 최신 버전들이 이 비용을 줄이는 방향으로 진화.

## 2) Java 11 / 17 / 21 / 25 핵심 쟁점 분석

모두 **LTS(Long-Term Support)** 버전. 6개월 주기 릴리스 중 2~3년마다 LTS가 나오고, 엔터프라이즈는 LTS를 채택한다.

### Java 11 (2018, LTS) — "모듈 이후의 첫 실용 LTS"
- **쟁점**: Oracle JDK 라이선스 변경(상용 유료화) → OpenJDK 배포판(Temurin/Corretto/Zulu) 대전환.
- 표준 HTTP Client(`java.net.http`), `var`(지역변수 추론, 10에서), 단일 파일 소스 실행, String 편의 메서드.
- Java EE/CORBA 모듈 제거. ZGC 실험적 도입. Applet/Web Start 폐기 수순.
- **의미**: "Java 8 → 11" 이 사실상 현대 자바의 출발선.

### Java 17 (2021, LTS) — "현대 문법의 완성"
- **Records**(불변 데이터 캐리어), **Sealed Classes**(상속 제한), **Pattern Matching for instanceof**, **Switch Expressions**, **Text Blocks** 정식화.
- JDK 내부 API 강한 캡슐화, ZGC/Shenandoah 프로덕션 등급.
- **의미**: 보일러플레이트 대폭 감소. 도메인 모델을 record/sealed로 안전하게 표현 → DDD 친화.

### Java 21 (2023, LTS) — "동시성 혁명"
- ★ **Virtual Threads(가상 스레드, JEP 444 정식)**: 수백만 개의 경량 스레드. blocking I/O 코드를 그대로 쓰면서 처리량을 끌어올림 — **스레드-per-요청 모델의 부활**.
- **Pattern Matching for switch**, **Record Patterns**(구조 분해), **Sequenced Collections**.
- **Generational ZGC**: 세대별 ZGC로 처리량/효율 개선.
- **의미**: 리액티브의 복잡성 없이 높은 동시성. 이 프로젝트의 핵심 성능 기반.

### Java 25 (2025, LTS) — "성숙·효율·시작 속도"
- 가상 스레드 성숙 + ★ **synchronized 핀(pinning) 제거(JEP 491, JDK 24+)**: 가상 스레드가 `synchronized` 블록에서 캐리어 스레드에 묶이던 문제 해결 → 레거시 코드/라이브러리도 가상 스레드 이점을 온전히 누림.
- **Compact Object Headers**: 객체 헤더 축소 → 힙 메모리 절감·캐시 효율↑(대량 객체 워크로드에 유리).
- **Scoped Values / Structured Concurrency** 정식화 방향 — 가상 스레드 시대의 컨텍스트 전파·작업 묶음 관리.
- **AOT(Project Leyden 계열) 시작 가속**: 클래스 로딩/링킹 사전 처리로 JVM 워밍업 단축.
- 패턴 매칭·record 지속 확장, 최신 GC(G1/ZGC) 개선.
- **의미**: "동시성은 21에서 열고, 25에서 핀 문제·메모리·시작비용까지 다듬어 **프로덕션 LTS로 완성**".

## 3) 이 프로젝트(Java 25)의 성능적 이점 분석

`build.gradle.kts` 의 `JavaLanguageVersion.of(25)`, 런타임 `eclipse-temurin:25-jre-alpine`. 이 프로젝트가 Java 25에서 실제로 얻는 이점을 코드 근거로 분석한다.

### 3-1. 가상 스레드 + JEP 491 (가장 큰 이점) ★
`application.yml`:
```yaml
spring.threads.virtual.enabled: true
# 주석: "Java 21+ 가상 스레드 (Tomcat 요청 처리 + @Scheduled). JEP 491(JDK24+)로 synchronized 핀 없음"
```
- **Tomcat 요청 처리를 가상 스레드로** 수행 → 요청당 스레드를 쓰면서도 OS 스레드 고갈 없이 높은 동시성.
- 이 프로젝트는 **blocking 스택**(JDBC/HikariCP, RestClient로 Toss PG 동기 호출, JPA)이다. 가상 스레드는 이런 blocking I/O 대기 동안 캐리어 스레드를 반납 → **리액티브로 재작성하지 않고도** 처리량 확보.
- **JEP 491 효과**: 정산·결제 코드의 `synchronized`/락 경로에서도 가상 스레드가 캐리어에 핀되지 않음 → JDBC 드라이버·레거시 라이브러리까지 이점 온전.
- **설계 정합**: `application.yml` 주석대로 "가상 스레드가 켜져 있어 요청 스레드 상한이 없으므로, 실질 처리량 천장은 **HikariCP 풀(20)**". 즉 동시성은 가상 스레드로 열되, 진짜 자원(DB 커넥션)은 풀로 스로틀 — Java 21+/25의 권장 패턴을 그대로 구현.

### 3-2. Record — 도메인/값 객체 (69개 파일)
- `MatchResult`, `HoldbackPolicy`, `PgTransactionRow`, `InternalPaymentRow`, 각종 Command/DTO가 record.
- 불변성 보장 + `equals/hashCode/toString` 자동 → 정산 금액/대사 행 같은 **값 객체의 안전성**과 보일러플레이트 제거. 컴팩트 생성자로 검증 내장(`HoldbackPolicy` rate 0~1 검증).

### 3-3. Switch Expression / Pattern Matching
- `SellerTier`·`SettlementCycle`의 등급/주기 분기, `HoldbackPolicy.forTier` 가 `switch` 식으로 표현 → **빠짐 없는(exhaustive) 분기** + 가독성. enum 상태 머신과 잘 맞음.

### 3-4. Text Block — SQL 가독성 (10개 파일)
- `SpringDataOutboxEventRepository` 의 `FOR UPDATE SKIP LOCKED` 네이티브 쿼리 등 멀티라인 SQL을 `"""..."""` 로 작성 → 이스케이프 없이 정확한 SQL.

### 3-5. 메모리/시작 비용 절감 (런타임 효과)
- **Compact Object Headers**(25): 정산 집계·대사처럼 다수의 작은 객체(레코드 행)를 다루는 워크로드에서 힙 footprint·캐시 효율 개선.
- **AOT 시작 가속**: K8s 롤링 배포/스케일아웃 시 Pod 기동 시간 단축 → 탄력성↑.
- `eclipse-temurin:25-jre-alpine` 경량 런타임 + 컨테이너 인식 메모리(`MaxRAMPercentage`)와 결합.

> 정리: 이 프로젝트가 Java 25에서 얻는 최대 이점은 **"blocking 코드를 유지한 채 가상 스레드로 동시성 확보(JEP 491로 핀 문제까지 해소)"**, 그리고 record/switch/text block 로 인한 **정산 도메인 코드의 안전성·표현력**, 보조적으로 메모리·시작 비용 절감이다.

# 부록 A. JVM (Java Virtual Machine)

## A-1. JVM 이란
바이트코드(.class)를 실행하는 추상 머신. OS·하드웨어 차이를 흡수해 WORA를 실현. 대표 구현은 **HotSpot**(OpenJDK).

## A-2. 실행 흐름
```
.java ──javac──▶ .class(바이트코드) ──클래스로더──▶ JVM 메모리
   ──인터프리터로 실행 + JIT(C1/C2)로 핫스팟 네이티브 컴파일──▶ CPU
```
- **클래스 로더**: 부트스트랩 → 플랫폼 → 애플리케이션 로더 계층, 위임 모델.
- **JIT**: 자주 실행되는 코드(핫스팟)를 C1(빠른 컴파일)→C2(고도 최적화)로 단계 컴파일. 그래서 워밍업이 필요(서버는 워밍업 후 빨라짐).
- **AOT(Leyden, 25 방향)**: 일부를 사전 컴파일/링크해 워밍업 단축.

## A-3. 런타임 메모리 구조
| 영역 | 내용 | 공유 |
|------|------|------|
| **Heap** | 객체·배열. GC 대상. Young(Eden+Survivor)/Old 세대 | 스레드 공유 |
| **Metaspace** | 클래스 메타데이터(네이티브 메모리, PermGen 후신) | 공유 |
| **Stack** | 메서드 프레임·지역변수 | 스레드별 |
| **PC Register** | 현재 실행 위치 | 스레드별 |
| **Native/Direct** | JNI, NIO 다이렉트 버퍼 | 공유 |

- 이 프로젝트: 힙은 `MaxRAMPercentage=75%`(컨테이너 비율), 나머지 25%가 Metaspace·스택·다이렉트 버퍼 몫([memory.md](memory.md) 참조).
- **가상 스레드**: 스택을 힙에 저장하는 경량 구조라 OS 스레드보다 훨씬 가볍다 → 수만 동시성 가능.

# 부록 B. GC (Garbage Collection) — 이 프로젝트 분석

## B-1. GC 기본
도달 불가능(unreachable) 객체를 자동 회수. **세대 가설**(대부분 객체는 금방 죽는다)에 기반해 힙을 Young/Old로 나눠 Young을 자주(Minor GC), Old를 가끔(Major/Full GC) 수집.

핵심 지표: **처리량(throughput)** vs **지연(latency, STW=Stop-The-World 시간)** vs **메모리(footprint)** — 삼각 트레이드오프.

## B-2. 주요 GC 알고리즘
| GC | 특성 | 적합 |
|----|------|------|
| **Serial** | 단일 스레드, 작은 힙 | 소규모/CLI |
| **Parallel** | 처리량 우선, STW 김 | 배치 |
| **G1 (기본)** | region 기반, 예측 가능한 STW 목표, 균형형 | **범용 서버(기본값)** |
| **ZGC / Gen ZGC** | 초저지연(<1ms), 대용량 힙, 동시 수집 | 지연 민감·대형 힙 |
| **Shenandoah** | 저지연, 동시 압축 | 지연 민감 |

## B-3. 이 프로젝트의 GC 처리 분석
**현재 설정** (`Dockerfile`/`docker-compose.yml`):
```
JAVA_OPTS="-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0 -XX:InitialRAMPercentage=50.0"
```
분석:
- **명시적 GC 미지정 → 기본 G1GC 사용**. 정산/이커머스 같은 **균형형 워크로드(중간 힙, 처리량+적당한 지연)** 에 G1이 합리적 기본 선택.
- **컨테이너 인식**(`UseContainerSupport`) + **비율 기반 힙**(고정 `-Xmx` 대신 `MaxRAMPercentage=75`) → K8s Pod 메모리 limit만 조정하면 힙·GC가 자동 적응. 컨테이너 환경의 GC 베스트프랙티스.
- `InitialRAMPercentage=50` → 초기 힙을 충분히 확보해 기동 직후 잦은 GC/힙 확장 방지.
- **GC 친화적 애플리케이션 설계**(GC 압력 자체를 줄이는 코드):
  - 캐시 경계(Caffeine `maximumSize=500`) → 장수명 객체 누적 억제 → Old 영역 압박↓.
  - 커서 페이지네이션(최대 100행)·배치 LIMIT 적재 → 대량 단명 객체 폭발 방지(Young GC 부담↓).
  - `open-in-view: false` → 영속성 컨텍스트(엔티티) 조기 해제 → 단명 객체 회수 빠름.
  - record/값 객체는 단명·불변이라 Young 세대에서 값싸게 회수.
- **모니터링 연계**: `HighJVMMemoryUsage`(heap>85%) Prometheus 알림 + `jvm_memory_used/max_bytes` 메트릭으로 GC/힙 상태 감시([프로메테우스.md](프로메테우스.md)).

## B-4. 개선 여지 (옵션)
- **지연이 더 중요해지면**: Java 25의 **Generational ZGC**(`-XX:+UseZGC -XX:+ZGenerational`)로 전환해 STW를 ms 이하로. 단, 처리량/메모리 트레이드오프와 벤치마크 선행 필요.
- **현재는 G1 기본이 합리적** — 워크로드가 초저지연을 요구하지 않고, 운영 단순성/예측 가능성이 더 가치 있음. 섣부른 GC 튜닝보다 **측정(GC 로그·메트릭) 후 결정**이 원칙.

## 정리
- **본질**: Java = 언어 + JVM + 라이브러리. 안정성·생산성·성숙 런타임 → 정산/금융 장수명 시스템에 적합.
- **버전 흐름**: 11(현대 자바 출발) → 17(record/sealed 문법 완성) → 21(가상 스레드 동시성 혁명) → **25(핀 제거·메모리·시작 비용까지 다듬은 LTS)**.
- **이 프로젝트의 Java 25 이점**: ① 가상 스레드+JEP 491로 blocking 스택 그대로 고동시성, ② record/switch/text block로 정산 도메인 안전·표현력, ③ Compact Object Headers·AOT로 메모리·기동 비용 절감.
- **JVM/GC**: 컨테이너 인식 비율 힙 + **기본 G1GC**, GC 압력을 줄이는 애플리케이션 설계(캐시 경계·커서·OSIV off), Prometheus로 힙 감시. 초저지연 필요 시 Gen ZGC 옵션.
- 관련 문서: [memory.md](memory.md), [프로메테우스.md](프로메테우스.md), [network.md](network.md).
