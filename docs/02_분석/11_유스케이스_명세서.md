## 3. 유스케이스 상세 명세

### UC-001: 회원가입

| 항목 | 내용 |
|------|------|
| UC ID | UC-001 |
| 유스케이스명 | 회원가입 |
| 주 액터 | 구매자 |
| 설명 | 이메일과 비밀번호로 신규 회원을 등록한다 |
| 사전 조건 | 미가입 사용자 |
| 사후 조건 | 회원 정보 DB 저장, USER 역할 부여 |

**기본 흐름:**
1. 사용자가 회원가입 페이지에 접근한다.
2. 이메일, 비밀번호, 이름 등 필수 정보를 입력한다.
3. 시스템이 이메일 중복 여부를 확인한다.
4. 시스템이 비밀번호를 BCrypt로 해싱하여 저장한다.
5. 회원 등록 완료 메시지를 표시한다.

**대안 흐름:**
- 3a. 이메일이 이미 존재하는 경우: 중복 안내 메시지 표시
- 2a. 비밀번호 정책 미충족: 비밀번호 규칙 안내

**예외 흐름:**
- E1. 서버 오류 발생: 에러 메시지 표시, 재시도 안내

---

### UC-004: 주문 생성

| 항목 | 내용 |
|------|------|
| UC ID | UC-004 |
| 유스케이스명 | 주문 생성 |
| 주 액터 | 구매자 |
| 설명 | 장바구니 상품을 기반으로 주문을 생성한다 |
| 사전 조건 | 로그인 상태, 장바구니에 상품 존재 |
| 사후 조건 | 주문 레코드 생성 (상태: PENDING) |

**기본 흐름:**
1. 구매자가 장바구니에서 주문하기를 클릭한다.
2. 시스템이 배송 정보 입력 화면을 표시한다.
3. 구매자가 배송 정보를 입력한다.
4. (선택) 구매자가 쿠폰을 적용한다. → UC-010
5. 시스템이 주문 금액을 계산한다.
6. 구매자가 주문을 확인하고 결제를 진행한다. → UC-005
7. 시스템이 주문을 PENDING 상태로 생성한다.

**대안 흐름:**
- 4a. 쿠폰이 만료된 경우: 쿠폰 사용 불가 안내
- 7a. 재고 부족: 재고 부족 안내, 주문 생성 불가

---

### UC-005: 결제 처리

| 항목 | 내용 |
|------|------|
| UC ID | UC-005 |
| 유스케이스명 | 결제 처리 |
| 주 액터 | 구매자 |
| 보조 액터 | Toss PG |
| 설명 | Toss PG를 통해 결제를 승인하고 매입한다 |
| 사전 조건 | 주문 생성 완료 (PENDING) |
| 사후 조건 | 결제 완료 (CAPTURED), 주문 상태 PAID |

**기본 흐름:**
1. 시스템이 Toss 결제 페이지를 호출한다.
2. 구매자가 결제 수단을 선택하고 결제한다.
3. Toss PG가 승인 결과를 콜백으로 전달한다.
4. 시스템이 결제 승인을 처리한다.
5. 시스템이 결제를 매입(CAPTURED) 처리한다.
6. 주문 상태를 PAID로 변경한다.

**대안 흐름:**
- 3a. 결제 실패: 실패 사유 표시, 재결제 안내
- 3b. 사용자 취소: 주문 상태 유지 (PENDING)

**예외 흐름:**
- E1. PG 통신 오류: 재시도 로직 실행 (최대 3회)
- E2. 타임아웃: 결제 상태 조회 후 결과 처리

### UC-006: 환불 요청

| 항목 | 내용 |
|------|------|
| UC ID | UC-006 |
| 유스케이스명 | 환불 요청 |
| 주 액터 | 구매자 |
| 보조 액터 | Toss PG |
| 설명 | 결제된 주문에 대해 환불을 요청한다 |
| 사전 조건 | 결제 완료 상태 |
| 사후 조건 | 환불 완료, 역정산 처리 |

**기본 흐름:**
1. 구매자가 환불 요청을 한다.
2. 시스템이 Idempotency-Key를 확인한다.
3. 시스템이 PESSIMISTIC_WRITE 락을 획득한다.
4. Toss PG에 환불 요청을 전송한다.
5. 환불 결과를 저장한다.
6. 주문 상태를 REFUNDED로 변경한다.
7. 정산 조정(역정산) 레코드를 생성한다.

**대안 흐름:**
- 2a. 동일 Idempotency-Key 존재: 이전 결과 반환 (멱등성)
- 4a. 부분 환불: 환불 금액만큼 처리

---

### UC-009: 자동 정산 처리

| 항목 | 내용 |
|------|------|
| UC ID | UC-009 |
| 유스케이스명 | 자동 정산 처리 |
| 주 액터 | 시스템 (Spring Batch) |
| 설명 | 전날 CAPTURED 결제를 대상으로 자동 정산을 처리한다 |
| 사전 조건 | CAPTURED 상태 결제 존재 |
| 사후 조건 | 정산 레코드 CONFIRMED |

**기본 흐름:**
1. [02:00] 배치가 전날 CAPTURED 결제를 조회한다.
2. 판매자별 PENDING 정산 레코드를 생성한다.
3. [03:00] 배치가 PENDING 정산을 CONFIRMED로 확정한다.
4. [03:10] 배치가 PENDING 조정 건을 CONFIRMED로 확정한다.

**대안 흐름:**
- 1a. CAPTURED 결제 없음: 배치 정상 종료 (0건 처리)
- 2a. 중복 실행 시도: JobRepository가 중복 방지

**예외 흐름:**
- E1. 배치 실패: 알림 발송, 재실행 가능

### UC-012: 정산 모니터링

| 항목 | 내용 |
|------|------|
| UC ID | UC-012 |
| 유스케이스명 | 정산 모니터링 |
| 주 액터 | 관리자 |
| 설명 | 관리자가 정산 현황을 모니터링하고 검색한다 |
| 사전 조건 | ADMIN 권한으로 로그인 |
| 사후 조건 | 정산 현황 데이터 조회 완료 |

**기본 흐름:**
1. 관리자가 정산 관리 대시보드에 접근한다.
2. 기간, 판매자, 상태 등 검색 조건을 입력한다.
3. 시스템이 Elasticsearch에서 정산 데이터를 검색한다.
4. 검색 결과를 목록으로 표시한다.
5. 관리자가 상세 정산 내역을 확인한다.
