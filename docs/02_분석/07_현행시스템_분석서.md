## 1. 분석 개요

### 1.1 분석 목적
현행 전자상거래 시스템의 구조, 기능, 데이터, 인프라 현황을 분석하여 신규 Lemuel 시스템 구축 시 반영할 요구사항과 개선 방향을 도출한다.

### 1.2 분석 범위
- 주문 관리 시스템
- 결제 처리 시스템
- 정산 처리 프로세스 (수작업 포함)
- 상품/카테고리 관리 시스템
- 사용자 인증 시스템

### 1.3 분석 방법

| 방법 | 대상 | 내용 |
|------|------|------|
| 인터뷰 | 업무 담당자 | 현행 업무 프로세스 파악 |
| 문서 분석 | 기존 설계 문서 | 시스템 구조/기능 분석 |
| 시스템 분석 | 운영 시스템 | 실제 동작 확인 |
| 데이터 분석 | 운영 DB | 데이터 구조/양/품질 분석 |

## 2. 현행 시스템 구성

### 2.1 시스템 구성도

```
[사용자] → [웹 서버] → [WAS] → [DB]
                              → [PG사]
         [관리자] → [관리 도구(수작업)]
```

### 2.2 시스템 현황

| 구분 | 현행 | 비고 |
|------|------|------|

## 3. 현행 업무 프로세스

### 3.1 주문 프로세스

```
상품 조회 → 장바구니 → 주문 생성 → 결제 요청 → 결제 승인 → 주문 확정
                                                           ↓
                                          주문 취소 ← 취소 요청
```

### 3.2 결제 프로세스

```
결제 요청 → PG 승인 요청 → 승인 결과 수신 → 결제 완료
                                            ↓
            PG 취소 요청 ← 환불 요청 ← 환불 처리
```

### 3.3 정산 프로세스 (현행 - 수작업)

```
결제 내역 엑셀 다운로드 → 수동 분류 → 수수료 계산 → 정산 금액 산출 → 이체 요청
     (매일)            (수작업)    (수작업)      (수작업)        (수작업)
```

## 7. 문제점 및 개선 방향

### 7.1 문제점 분석

| No | 영역 | 문제점 | 심각도 | 영향 |
|----|------|--------|--------|------|
| 1 | 정산 | 수작업 정산으로 오류 빈발 | 높음 | 정산 지연, 금액 오류 |
| 2 | 결제 | 단일 PG 의존 | 높음 | 장애 시 결제 불가 |
| 3 | 검색 | DB 직접 쿼리 검색 | 중간 | 응답 지연, 한글 검색 미흡 |
| 4 | 주문 | 상태 추적 불완전 | 중간 | 고객 민원 |
| 5 | 인증 | 세션 기반 인증 | 중간 | 확장성 제한 |
| 6 | 아키텍처 | 모놀리식 구조 | 중간 | 유지보수 어려움 |

### 7.2 개선 방향

| 문제점 | 개선 방향 | 적용 기술 |
|--------|----------|-----------|
| 수작업 정산 | 자동 배치 정산 | Spring Batch, 일별/주별/월별 |
| 단일 PG | Toss PG 연동 + 멱등성 | Idempotency-Key, PESSIMISTIC_WRITE |
| DB 직접 검색 | Elasticsearch 도입 | Nori 한글 분석기 |
| 상태 추적 미흡 | 상태 머신 기반 주문 관리 | 도메인 이벤트 |
| 세션 인증 | JWT 기반 인증/인가 | Spring Security + JWT |
| 모놀리식 | 헥사고날 아키텍처 | Ports & Adapters |
