# 면접용 자기소개 — Lemuel(이커머스 + 정산 MSA) 프로젝트 기반

이 프로젝트를 면접에서 효과적으로 어필하기 위한 자기소개·말하기 가이드. 길이별 스크립트 + 기술 어필 포인트(STAR) + 예상 Q&A로 구성했습니다.

> ⚠️ 원칙: **거짓 수치 금지.** 측정한 적 없는 TPS/지연 숫자를 지어내지 마세요. 대신 "왜 그렇게 설계했고, 어떤 트레이드오프를 고려했는지" — 기술 의사결정의 깊이로 승부합니다. 면접관은 숫자보다 사고 과정을 봅니다.

---

## 1) 길이별 자기소개 스크립트

### 🎯 30초 엘리베이터 피치
> "이커머스와 **정산(Settlement) 도메인**을 헥사고날 아키텍처 기반 MSA로 설계한 백엔드 프로젝트를 진행했습니다. 핵심은 **실제 돈이 오가는 정산 시스템**이라 정합성과 멱등성을 최우선으로 두고, 결제→정산 흐름을 **Transactional Outbox + Kafka**로 비동기 연결하면서도 **3단 멱등 방어**로 중복 정산을 원천 차단한 점입니다. Java 25의 가상 스레드, Resilience4j, Prometheus 관측성까지 운영 관점을 함께 고려했습니다."

### 🎯 1분 버전
> "저는 **정산 도메인**처럼 정합성이 까다로운 백엔드 설계에 관심이 많습니다. 이번 프로젝트는 주문·결제·정산을 **order-service / settlement-service / API Gateway** 3개로 분리한 MSA인데, 단순히 나누는 데 그치지 않고 **서비스 간 코드 의존을 0으로** 만드는 데 집중했습니다.
>
> 정산 서비스가 주문 데이터를 봐야 하는데, order-service를 import하지 않고 **Read-only Projection 패턴**(@Immutable 엔티티로 같은 테이블만 읽기)으로 해결했고, 상태 전파는 **Kafka 이벤트**로만 했습니다.
>
> 가장 신경 쓴 건 **'실제 돈'이라는 점**입니다. 결제 완료 이벤트가 중복 전달돼도 정산이 두 번 생기면 안 되므로, Outbox event_id, processed_events PK, settlements.payment_id UNIQUE의 **3단 멱등 방어**를 설계했습니다. 환불·차지백 같은 역정산은 확정된 정산을 수정하지 않고 별도 Adjustment 레코드로 상쇄해 **복식부기 원장 정합성**을 지켰습니다."

### 🎯 3분 버전 (기술 깊이 강조)
> (1분 버전에 이어)
> "운영 관점도 함께 봤습니다. PG 연동은 **Resilience4j로 PG별 독립 Circuit Breaker**를 둬서 한 PG 장애가 다른 PG로 전이되지 않게 격벽을 쳤고, unhealthy하면 fallback 체인으로 자동 전환되는 **PG 라우터**를 만들었습니다.
>
> Kafka 컨슈머는 독성 메시지가 정산 SLA를 무너뜨리지 않도록 **재시도 후 DLT 격리 + 운영자 replay** 경로를 설계했고, 모든 지표를 **Prometheus 커스텀 메트릭**으로 노출해 '결제−환불=정산' 불변식이 깨지면 즉시 알림이 가도록 했습니다. 알림에는 runbook 대응 절차까지 넣었습니다.
>
> Java 25의 **가상 스레드**를 켜서 JDBC 같은 blocking 코드를 리액티브로 재작성하지 않고도 동시성을 확보하되, 실질 처리량 천장은 **HikariCP 커넥션 풀로 스로틀**하는 식으로 자원을 통제했습니다. 이 모든 구조적 결정을 **ADR 18건**으로 문서화하고 **ArchUnit으로 헥사고날 의존 방향을 강제**한 게 제가 가장 자부하는 부분입니다."

## 2) 프로젝트 한 줄 요약 & 키워드

**"정합성이 생명인 정산 도메인을, 코드 의존 0의 MSA로 분리하고 이벤트 기반으로 안전하게 연결한 백엔드"**

| 분류 | 키워드 |
|------|--------|
| 아키텍처 | 헥사고날(Ports & Adapters), MSA, Bounded Context, Read-only Projection, CQRS 성격 |
| 이벤트 | Transactional Outbox, Kafka, 3단 멱등, DLT + replay, 분산 추적(traceparent) |
| 정산 도메인 | 차등 수수료(SellerTier), 홀드백, 정산주기(T+N), 역정산(Adjustment), 복식부기 원장, PG 대사, 지급(payout), 차지백 |
| 동시성/정합성 | 낙관·비관 락, ShedLock, BigDecimal 정확연산, 가상 스레드 |
| 회복탄력성 | Resilience4j(CB/Retry/Bulkhead), PG 라우팅 fallback |
| 운영/관측 | Prometheus 커스텀 메트릭, Grafana, 알림+runbook, 2-tier 캐시, PgBouncer |
| 품질 | ArchUnit, ADR 18건, JaCoCo 커버리지 게이트 |

## 3) 기술 어필 포인트 (STAR 구조)

면접에서 "어떤 어려움을 어떻게 해결했나요?"에 바로 쓸 수 있는 형태.

### ⭐ 포인트 1 — MSA 코드 의존 0
- **상황(S)**: 모놀리스를 정산/거래 컨텍스트로 분리하는데, 정산이 주문·결제 데이터를 조회해야 함.
- **문제(T)**: 단순히 order-service를 라이브러리로 의존하면 결합이 그대로 남아 MSA 의미가 없음.
- **행동(A)**: settlement-service에 `@Immutable` JPA 엔티티(Read-Model)를 정의해 같은 테이블을 **읽기 전용**으로만 매핑. 상태 전파는 Kafka로. build.gradle에 order-service 의존을 **넣지 않음**.
- **결과(R)**: 코드 경계 100% 확립. ArchUnit으로 의존 방향을 자동 검증해 회귀 방지.

### ⭐ 포인트 2 — 중복 정산 방지 (실제 돈)
- **S**: 결제 완료를 Kafka로 받아 정산 생성. Kafka는 at-least-once라 **중복 소비 가능**.
- **T**: 정산이 두 번 생기면 셀러에게 이중 지급 — 금전 사고.
- **A**: ① Outbox `event_id` UUID UNIQUE(발행 중복 차단) ② `processed_events(group, event_id)` PK(수신 중복 차단) ③ `settlements.payment_id` UNIQUE(스키마 최종 방어) — **3단 멱등**. 컨슈머는 수동 ack로 처리 완료 후에만 커밋.
- **R**: 파티션 병렬 소비에도 중복 정산 0 보장.

### ⭐ 포인트 3 — 이중 쓰기(dual write) 문제 해결
- **S**: "DB 커밋"과 "Kafka 발행"이 원자적이지 않아, 커밋 후 발행 실패 시 이벤트 유실.
- **A**: **Transactional Outbox** — 도메인 변경과 outbox INSERT를 같은 트랜잭션으로 묶고, 별도 폴러가 `FOR UPDATE SKIP LOCKED`로 멀티워커 경합 없이 발행.
- **R**: 이벤트 유실 0 + 다중 워커 수평 확장 가능(분산 작업 큐).

### ⭐ 포인트 4 — 확정 정산 불변성 + 역정산
- **S**: 이미 지급된(DONE) 정산에 환불이 발생.
- **T**: DONE 정산 금액을 직접 고치면 원장이 깨짐.
- **A**: DONE은 **불변**(DB 트리거로도 강제), 환불은 음수 `SettlementAdjustment` 별도 레코드로 상쇄 + 복식부기 **역분개**. 홀드백이 있으면 거기서 우선 차감.
- **R**: 회계 정합성 유지 + 환불 이력 완전 추적.

### ⭐ 포인트 5 — 장애 격리 (PG / Kafka)
- **A**: PG는 **Resilience4j 독립 Circuit Breaker**(벌크헤드) + fallback 체인. Kafka 컨슈머는 일시 오류는 재시도, **독성 메시지는 즉시 DLT 격리** 후 운영자 replay.
- **R**: 한 외부 시스템 장애가 전체로 전이되지 않고, 실패 메시지도 유실 없이 복구 가능.

### ⭐ 포인트 6 — 운영/관측성을 코드와 함께 설계
- **A**: 환불 실패율·Outbox 적체·DLT 발행·**'결제−환불=정산' 대사 불일치** 등을 Prometheus 커스텀 메트릭으로 노출, 8개 알림 그룹 + 알림에 runbook 절차 포함. 분산 추적은 traceparent를 Outbox에 실어 비동기 경계까지 연결.
- **R**: "장애를 코드 배포 후가 아니라 설계 단계에서 관측 가능하게" 만든 점.

## 4) 예상 질문 & 답변 가이드

**Q. 왜 MSA로 나눴나요? 오버엔지니어링 아닌가요?**
> A. 솔직히 트래픽 규모만 보면 모놀리스로도 충분합니다. 다만 이 프로젝트의 목적은 **"정산이라는 다른 생명주기·다른 정합성 요구를 가진 컨텍스트를 거래와 분리"**하는 경험이었습니다. 그래서 무작정 나누기보다 **코드 의존 0**이라는 명확한 기준을 세우고, 비용이 큰 동기 통신 대신 이벤트로 느슨하게 연결한 점을 강조하고 싶습니다.

**Q. 왜 동기 REST가 아니라 Kafka인가요?**
> A. 정산 생성은 결제 완료에 **즉시 강결합될 필요가 없고**, order가 죽어도 settlement가 큐로 따라잡는 시간적 디커플링이 더 중요했습니다. 대신 비동기의 대가인 중복·순서 문제를 멱등 3단과 payment_id 파티션 키로 해결했습니다.

**Q. 가상 스레드를 왜 썼고 효과가 있었나요?**
> A. 이 시스템은 JDBC·PG 동기 호출 등 **blocking 스택**입니다. 리액티브로 재작성하는 대신 가상 스레드로 동시성을 얻는 게 ROI가 좋다고 판단했습니다. 다만 가상 스레드는 무한정이라 실질 천장은 DB 커넥션 풀이라, 풀 크기로 의도적으로 스로틀했습니다. (수치는 측정 전이라 단정하지 않겠습니다.)

**Q. 정합성이 깨질 수 있는 시나리오와 방어는?**
> A. ① 중복 이벤트 → 멱등 3단 ② 동시 환불 lost update → 비관적 락 ③ 동시 확정/지급 → 낙관적 락(@Version) ④ 다중 노드 배치 중복 실행 → ShedLock ⑤ DB커밋·발행 비원자성 → Outbox. 각 위험에 대응 장치를 매핑해 설계했습니다.

**Q. 가장 어려웠던 점은?**
> A. "이미 지급된 정산에 환불이 오면?"이었습니다. 금액을 직접 수정하면 원장이 깨지니, **DONE 불변 + Adjustment 상쇄 + 역분개**라는 회계 원칙을 코드로 옮기는 과정에서 도메인 규칙을 깊이 고민했습니다.

**Q. 아쉬운 점 / 더 했다면?**
> A. ① 실제 부하 테스트로 가상 스레드·풀 크기를 **측정 기반으로 튜닝**하지 못한 점 ② GC를 기본 G1로 뒀는데 지연 요구가 커지면 Gen ZGC 검토 ③ 정산 마이그레이션을 별도 DB로 물리 분리하는 후속 작업. 측정 없이 단정하지 않는 게 제 원칙이라, 다음 단계는 벤치마크라고 생각합니다.

## 5) 면접관에게 남길 인상 (핵심 메시지)

1. **"정합성을 타협하지 않는다"** — 실제 돈을 다루는 도메인 감각.
2. **"기준을 세우고 분리한다"** — 코드 의존 0, 멱등 3단처럼 명확한 원칙.
3. **"운영을 설계에 포함한다"** — 관측성·장애 격리·runbook까지.
4. **"측정 없이 단정하지 않는다"** — 과장 대신 트레이드오프로 말한다.
5. **"결정을 문서화한다"** — ADR 18건, ArchUnit 강제.
