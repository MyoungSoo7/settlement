## 2. 사업 이해

### 2.1 사업 목적 이해
Lemuel 프로젝트는 전자상거래 환경에서 주문·결제·정산의 End-to-End 자동화를 목표로 하는 통합 시스템입니다.

### 2.2 핵심 요구사항 분석

| 요구 영역 | 핵심 요구사항 | 제안 대응 방안 |
|-----------|-------------|---------------|
| 주문 관리 | 주문 생성/취소/환불 | 상태 머신 기반 주문 생명주기 관리 |
| 결제 관리 | Toss PG 연동 | Toss Payments API 연동, 멱등키 적용 |
| 정산 관리 | 자동 배치 정산 | Spring Batch 기반 일별 정산 자동화 |
| 검색 | 한글 전문 검색 | Elasticsearch + Nori 분석기 적용 |

### 2.3 기대 효과
- 정산 처리 시간 90% 단축 (수작업 → 자동 배치)
- 결제 오류율 감소 (멱등성 보장, 동시성 제어)
- 판매자 정산 투명성 확보 (실시간 대시보드)

## 3. 기술 제안

### 3.1 시스템 아키텍처

#### 3.1.1 전체 아키텍처
- **아키텍처 패턴**: 헥사고날 (Ports & Adapters)
- **의존성 규칙**: domain ← application ← adapter
- **도메인 분리**: 사용자, 상품, 주문, 결제, 정산, 카테고리, 쿠폰, 리뷰

#### 3.1.2 기술 스택

| 계층 | 기술 | 버전 | 선정 사유 |
|------|------|------|-----------|
| Backend | Spring Boot | 3.x | 엔터프라이즈 검증, 대규모 생태계 |
| Language | Java | 21 LTS | 장기 지원, 가상 스레드 |
| Frontend | React | 18.x | SPA, 컴포넌트 재사용성 |
| Build Tool | Vite | Latest | 빠른 HMR, 최적 번들링 |
| Database | PostgreSQL | 17 | ACID 보장, JSON 지원 |
| Search | Elasticsearch | 8.x | 전문 검색, Nori 한글 분석 |
| Container | Docker | Latest | 표준 컨테이너화 |
| CI/CD | GitHub Actions | - | 소스 저장소 통합 |

#### 3.1.3 도메인 패키지 구조

```
{domain}/
├── adapter/
│   ├── in/web/              # REST Controller, DTO
│   └── out/persistence/     # JPA Entity, Repository, Mapper
├── application/
│   ├── port/in/             # UseCase 인터페이스
│   ├── port/out/            # Port 인터페이스
│   └── service/             # UseCase 구현체
└── domain/                  # 순수 도메인 객체
```

### 3.2 핵심 기술 설계

#### 3.2.1 결제 동시성 제어
- `PESSIMISTIC_WRITE` 락 적용
- `Idempotency-Key` 헤더 기반 멱등성 보장

#### 3.2.2 정산 배치 스케줄

| 시간 | 배치명 | 내용 |
|------|--------|------|
| 매일 02:00 | 정산 생성 | 전날 CAPTURED 결제 → PENDING 정산 |
| 매일 03:00 | 정산 확정 | PENDING → CONFIRMED |
| 매일 03:10 | 정산 조정 확정 | PENDING 조정 → CONFIRMED |

#### 3.2.3 보안 설계
- JWT 기반 인증/인가
- OWASP Top 10 대응
- 민감 데이터 암호화
- SonarCloud / Snyk 연동 자동 스캔

### 3.3 데이터베이스 설계 방향
- PostgreSQL 17, 스키마: `opslab`
- Flyway 마이그레이션 관리 (V1 ~ V21)
- MapStruct 기반 도메인 ↔ 엔티티 변환

### 3.4 프론트엔드 설계 방향

```
frontend/src/
├── api/           # API 서비스 (axios + JWT 인터셉터)
├── components/    # 재사용 UI 컴포넌트
├── contexts/      # React Context (Toast, Cart)
├── pages/         # 페이지 컴포넌트
├── types/         # TypeScript 타입 정의
└── __tests__/     # Vitest 테스트
```

## 4. 관리 제안

### 4.1 개발 방법론
- 반복·점진적 개발 방법론 (Iterative & Incremental)
- 2주 단위 스프린트 기반 개발

### 4.2 프로젝트 관리 방안

| 관리 영역 | 관리 방안 |
|-----------|-----------|
| 일정 관리 | WBS 기반 주간 진척 관리 |
| 품질 관리 | 코드 리뷰, JaCoCo 70% 커버리지 |
| 위험 관리 | 위험 등록부 운영, 주간 리뷰 |
| 변경 관리 | 변경 요청서 기반 CCB 운영 |
| 형상 관리 | Git + GitHub Actions CI/CD |

### 4.3 품질 보증

| 테스트 유형 | 도구 | 기준 |
|------------|------|------|
| 단위 테스트 | JUnit 5, Vitest | 커버리지 70% 이상 |
| 통합 테스트 | Spring Boot Test | E2E 시나리오 통과 |
| 성능 테스트 | JMeter / k6 | TPS 목표치 달성 |
| 보안 스캔 | Snyk, SonarCloud | Critical 0건 |

### 4.4 일정 계획

| 단계 | 기간 | 주요 산출물 |
|------|------|------------|
| 제안/착수 | 1개월 | RFP, PMP, WBS |
| 분석 | 1.5개월 | SRS, RTM, BPD |
| 설계 | 1.5개월 | 아키텍처설계서, ERD, API명세서 |
| 구현 | 2개월 | 소스코드, 단위테스트 결과 |
| 테스트 | 0.5개월 | 통합테스트, 성능테스트 결과 |
| 이행 | 0.5개월 | 운영 이관, 교육 |
