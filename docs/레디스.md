# 1부. Redis 소개와 사용법

## 1-1. Redis 란

**Redis(REmote DICtionary Server)** 는 메모리 기반의 **인메모리 키-값 데이터 스토어**다. 데이터를
주로 RAM 에 두어 **마이크로초 단위 응답**을 내고, 단일 스레드 이벤트 루프로 명령을 처리한다.
캐시·세션·큐·분산락·실시간 순위표 등 *"빠르게 읽고 쓰는 휘발성·준영속 데이터"* 에 널리 쓰인다.

핵심 특징:
- **빠름**: 메모리 기반 → 디스크 DB 대비 수십~수백 배 낮은 지연.
- **풍부한 자료구조**: String, Hash, List, Set, Sorted Set, Stream, Pub/Sub 등.
- **TTL(만료)**: 키마다 수명 설정 → 캐시·세션 자동 정리.
- **선택적 영속성**: RDB 스냅샷 / AOF(append-only file) 로 재시작 시 복구 가능.
- **Pub/Sub**: 채널 기반 메시지 발행/구독(가벼운 이벤트 전파).

## 1-2. 주요 자료구조와 용도

| 자료구조 | 용도 예 | 이 프로젝트 |
|----------|---------|-------------|
| **String** | 캐시값, JSON 한 덩어리, 카운터 | 장바구니 JSON, 캐시 엔트리 |
| Hash | 객체 필드 묶음 | — |
| List | 큐/스택 | — |
| Set / Sorted Set | 태그, 순위표 | — |
| **Pub/Sub** | 인스턴스 간 이벤트 전파 | **캐시 무효화 브로드캐스트** |
| Stream | 이벤트 로그 | — (이벤트는 Kafka 사용) |

## 1-3. 캐시 패턴 (일반)

- **Cache-Aside(Look-aside)**: 앱이 캐시를 먼저 보고, 없으면 DB 조회 후 캐시에 채움. 가장 흔함.
- **Read-through / Write-through**: 캐시 계층이 로딩/쓰기를 대신 위임받음.
- **TTL + 무효화(invalidation)**: 만료 시간으로 자연 정리 + 변경 시 명시적 evict 로 정합성 유지.

## 1-4. Redis 를 캐시로 쓸 때의 주의점

- **정합성**: DB 와 캐시의 값이 어긋날 수 있음 → 변경 시 evict, TTL 로 수렴.
- **장애 대응**: Redis 가 죽어도 앱이 멈추면 안 됨 → **fail-soft**(캐시 미스로 간주하고 DB 폴백).
- **직렬화**: 객체를 어떻게 바이트로 바꿔 저장할지(타입 안전성·gadget 역직렬화 보안).
- **메모리 한계**: maxmemory + eviction 정책(LRU/LFU) 필요.

## 1-5. Spring 에서 쓰는 법 (일반)

1. `spring-boot-starter-data-redis` 추가 → `RedisTemplate`/`StringRedisTemplate` 자동 구성.
2. `spring.data.redis.host/port` 설정.
3. 단순 값은 `redisTemplate.opsForValue().set/get(key, value, ttl)`.
4. 캐시 추상화는 `@Cacheable`/`@CacheEvict` + `CacheManager` 구현.
5. Pub/Sub 은 `convertAndSend(channel, msg)` + `MessageListener`/`RedisMessageListenerContainer`.

# 2부. 이 프로젝트의 Redis 적용 분석

이 프로젝트에서 Redis 는 **두 가지 독립된 용도**로 쓰이며, **둘 다 토글(opt-in)** 이다. 기본값에서는
Redis 없이도 동작한다(장바구니는 PostgreSQL, 캐시는 로컬 Caffeine).

| 용도 | 컴포넌트 | 토글 | 기본값 |
|------|----------|------|--------|
| ① 장바구니 저장소 | `RedisCartAdapter` | `cart.store=redis` | `jpa`(PostgreSQL) |
| ② 2-tier 캐시 L2 | `TwoTierCache` 외 | `app.cache.two-tier.enabled=true` | `false`(로컬 Caffeine) |

> compose 의 order-service 는 둘 다 켠다(`CART_STORE: redis`, `APP_CACHE_TWO_TIER_ENABLED: "true"`).
> settlement-service 는 Redis 스타터가 없어 자동 스킵(`@ConditionalOnClass(RedisConnectionFactory)`).

## 2-1. ① 장바구니 저장소 — `RedisCartAdapter`

장바구니를 Redis 에 저장하는 헥사고날 **아웃바운드 어댑터**다. (`cart.store=jpa` 기본 → PostgreSQL
`CartPersistenceAdapter`, `=redis` → 이 어댑터. `@ConditionalOnProperty` 로 빈 충돌 없이 토글.)

```java
@Component
@ConditionalOnProperty(name = "cart.store", havingValue = "redis")
public class RedisCartAdapter implements LoadCartPort, SaveCartPort {
    private static final String KEY_PREFIX = "cart:";
    private static final Duration TTL = Duration.ofDays(30);
    // ...
    public Cart save(Cart cart) {
        redis.opsForValue().set(key(cart.getUserId()), serialize(...), TTL);  // 30일 TTL
    }
}
```

설계 포인트:
- **키 설계**: `cart:{userId}` — "사용자당 장바구니 1개"라는 도메인 규칙과 `loadByUserId` 포트에
  정확히 대응. 집합체 전체를 **JSON 한 덩어리(String)** 로 저장(원자적 read/write, 단순).
- **TTL 30일**: 저장할 때마다 30일 만료를 갱신 → 도메인의 *"lastActiveAt 30일 정리"* 정책을
  **Redis 만료로 대체**. 별도 정리 cron 배치 불필요(Redis 가 알아서 만료).
- **직렬화**: `RedisCart` DTO ↔ 도메인 `Cart` 변환 후 Jackson JSON. 도메인 모델을 직접 직렬화하지
  않아 영속 포맷과 도메인을 분리.
- **헥사고날 정합**: 포트(`LoadCartPort`/`SaveCartPort`)만 구현 → 애플리케이션·도메인은 Redis 를
  전혀 모른다. 저장소를 jpa↔redis 로 갈아끼워도 도메인 무변경.

> 왜 장바구니에 Redis 가 잘 맞나: 장바구니는 **자주 바뀌고, 휘발성이 허용되며(만료 OK), 키 단위
> 단순 read/write** 인 전형적 Redis 워크로드. RDB 의 트랜잭션·정합성 보장이 굳이 필요 없다.

## 2-2. ② 2-tier 캐시 (L1 Caffeine + L2 Redis)

수평 확장(다중 인스턴스) 시 **로컬 캐시의 한계를 Redis 공유 계층으로 보완**하는 구조다.

### 배경 — 왜 2-tier 인가

기존 Caffeine 은 **인스턴스 로컬**이라, 인스턴스가 여러 개면 (1) 각자 중복 미스를 내고 (2) 한
인스턴스가 갱신해도 다른 인스턴스의 캐시는 **stale** 하다. → 공유 L2(Redis)와 **Pub/Sub 무효화**로
해소.

### 조회/쓰기 경로 (`TwoTierCache`)

```
조회: L1(Caffeine) hit → 반환
      L1 miss → L2(Redis) 조회 → 적중 시 L1 으로 승격(near-cache) 후 반환
      둘 다 miss → null → 로더(DB) 실행 후 양 계층 기록
쓰기: L1·L2 동시 기록 + Pub/Sub 로 "他 인스턴스 L1 무효화" 발행
```

### 핵심 메커니즘 4가지

1. **near-cache 승격**: L2 적중분을 L1 에 채워(`lookup()`) 다음 조회는 로컬에서 즉답.
2. **Pub/Sub 무효화**: `put/evict/clear` 시 `CacheInvalidationPublisher` 가
   `lemuel:cache:invalidation` 채널로 `originId|cacheName|key` 발행 →
   `CacheInvalidationListener` 가 각 인스턴스에서 **자기 L1 만** 무효화(L2 미터치·재발행 없음).
   - **루프 방지**: `originId` 가 자기 메시지면 무시.
   - **L2 는 발행자가 이미 갱신** 했으므로 수신자는 L1 만 비움 → 다음 조회 때 L2 에서 최신값.
3. **graceful degrade(fail-soft)**: L2 호출은 전부 try/catch — **Redis 가 죽어도** 경고 로그만 남기고
   L1/DB 로 폴백해 앱은 계속 동작(`l2Get/l2Put/l2Delete`). 무효화 발행 실패도 쓰기 흐름을 막지 않음.
4. **키 문자열 정규화**: Spring 이 넘기는 키(Long/String)를 `String.valueOf` 로 통일 → 분산 캐시·
   Pub/Sub 에서 키 매칭 일관성 보장(예: `products` 캐시의 Long `#productId` 와 String `'all'` 혼재 대응).

### 게이팅(비침습 설계) — `TwoTierCacheConfig`

- `@ConditionalOnProperty(app.cache.two-tier.enabled=true)` + `@ConditionalOnClass(RedisConnectionFactory)`
  → 꺼져 있거나 Redis 없으면 활성화 안 됨. 켜지면 로컬 전용 `CacheConfig`(Caffeine)는 back off.
- shared-common 은 Redis 를 `compileOnly` 로만 가져 settlement-service 런타임에 Redis 를 강제하지 않음.
- **직렬화 보안**: 필드 기반 Jackson + `DefaultTyping.EVERYTHING`(타입 충실성) + `BasicPolymorphic
  TypeValidator` 로 **안전 패키지만 역직렬화 허용**(gadget 공격 방어). 상세 [tps.md](tps.md) ⑦.

## 2-3. 인프라 — `docker-compose.yml`

```yaml
redis:
  image: redis:7-alpine
  command: ["redis-server", "--appendonly", "yes"]   # AOF 영속성 켬
  volumes: [redis-data:/data]
  healthcheck: ["CMD", "redis-cli", "ping"]
```

- **AOF 영속성**(`--appendonly yes`): 장바구니가 Redis 에 저장되므로(② 캐시는 재생성 가능하지만
  ① 장바구니는 사용자 데이터) 재시작 시 복구되도록 append-only 로그 활성.
- order-service 만 Redis 에 연결(`SPRING_DATA_REDIS_HOST: redis`), 헬스 토글
  (`MANAGEMENT_HEALTH_REDIS_ENABLED: true`) — jpa 모드에선 Redis 없이도 readiness 가 DOWN 안 되게.

## 2-4. 설계적으로 잘된 점

- **opt-in 토글**: 두 용도 모두 끌 수 있어, Redis 없는 환경(dev/test/단일 인스턴스)에서 기본 동작
  (PostgreSQL 장바구니 + 로컬 Caffeine)이 보장됨. 점진 도입 가능.
- **fail-soft**: 캐시 L2 는 Redis 장애에 폴백. 단일 장애점(SPOF)이 되지 않게 방어.
- **헥사고날 정합**: 장바구니 Redis 는 어댑터 교체로만 노출 → 도메인 오염 0.
- **TTL 활용**: 장바구니 30일 만료로 정리 배치 제거 — Redis 기능을 도메인 정책에 잘 매핑.

## 2-5. 한계·주의 (정직히)

- **장바구니 영속성 등급**: AOF 로 복구되지만 RDB(PostgreSQL)만큼의 내구성·백업 체계는 아님.
  중요 사용자 데이터라면 영속 보장 수준을 의식적으로 결정해야 함.
- **단일 인스턴스**: compose·k8s 모두 Redis 1개(HA 없음) → 운영은 Sentinel/Cluster 검토.
- **2-tier 정합성**: Pub/Sub 무효화는 **최선 노력(at-most-once)** — 메시지 유실 시 해당 인스턴스
  L1 은 TTL 만료까지 잠시 stale 할 수 있음(설계상 TTL 로 수렴, 강한 정합성 아님).
- **용도 한정**: 정합성이 자주 깨지는 데이터(`CouponService` used_count 등)는 의도적으로 캐시
  제외([tps.md](tps.md) 보강 참고).

## 정리

- Redis 는 이 프로젝트에서 **① 장바구니 저장소(`cart:{userId}` JSON + 30일 TTL)** 와 **② 2-tier
  캐시 L2(Caffeine L1 + Redis L2 + Pub/Sub 무효화)** 두 용도로 쓰이며 **둘 다 opt-in**.
- **장바구니**: 자주 바뀌고 만료 허용되는 키-단위 데이터라 Redis 에 적합. 헥사고날 어댑터로 jpa↔redis 토글.
- **2-tier 캐시**: 수평 확장 시 로컬 캐시 stale 문제를 공유 L2 + Pub/Sub 로 해소, **fail-soft** 로 Redis 장애에도 DB 폴백.
- **설계 강점**: opt-in 토글, 장애 폴백, 헥사고날 정합, TTL 로 정리 배치 제거.
- **한계**: Redis 단일 인스턴스(HA 없음), 무효화는 최선노력(TTL 수렴), 장바구니 내구성 등급 의식 필요.
- 관련 문서: [tps.md](tps.md)(⑦ 2-tier 캐시), [cpu.md](cpu.md)(⑨ 직렬화 비용), [ssd_hdd.md](ssd_hdd.md)(캐시로 DB 디스크 부담 완화).
