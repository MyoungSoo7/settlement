# Pass by Value vs Pass by Reference — 비교표

## 의사코드 — 왜 깊이가 달라지는가

```text
Pass by Value:
  agent.investigate() → return summary_string
  // 메인 RAM 부족이 sub의 산출 깊이를 제약한다.
  // sub는 "메인이 가져올 수 있는 분량"으로 미리 자기검열한다.

Pass by Reference:
  agent.investigate() → write(file); return path
  // sub의 산출 깊이가 메인 컨텍스트와 분리된다.
  // 메인이 가져오는 것은 경로 한 줄뿐이므로 sub는 제한 없이 적는다.
```

## 4축 비교

| 축 | Pass by Value | Pass by Reference |
|---|---|---|
| 에이전트 자기검열 | "메인이 읽을 수 있는 분량으로 정리" | "파일 하나에 다 담아" |
| 메인 컨텍스트 | 요약 5개로 부풀어 오름 (~200k 잠식) | 경로 3줄 (~수백 토큰) |
| 재활용 가능성 | 세션 종료 시 휘발 | 디스크에 영속 → 다음 세션도 읽음 |
| Disposable 효과 | 부분적 (요약은 메인에 남음) | 완전 (sub 컨텍스트 전부 폐기 가능) |

## 그림으로 보면 (강의자료 다이어그램 텍스트화)

### Pass by Value

```
   Codebase ←── Sub Agent
                  │ (정리된 요약 텍스트 반환)
                  ▼
              ┌────────────────────┐
              │  Main              │
              │  강의자료 관리자    │
              │                    │
              │  정리한 보고서를    │
              │  그대로 가져와서 읽음│
              │                    │
              │  모든 텍스트를 출력 │
              └────────────────────┘
```

메인이 sub의 산출을 *값*으로 가져온다. sub가 정리한 분량 전체가 메인 컨텍스트로 들어오고, 메인은 그걸 다시 사용자에게 풀어서 출력한다. 컨텍스트는 sub 본문 + 메인의 가공된 출력으로 두 배가 된다.

### Pass by Reference

```
                                Sub Agent ──→ Codebase
                                  │ ▲          ▲
                                  │ │ (Explore)│
   ┌────────────────────┐         ▼ │          │
   │  Main              │ ←── 보고서 파일 주소를
   │  강의자료 관리자    │     주소만 들어옴 (경로 한 줄)
   │                    │
   │  보고서 파일 주소만 │     File Read ──→ Explore (별도 sub)
   │  들어옴            │
   │                    │
   │  다음 해야 할 것    │
   │  3가지             │
   └────────────────────┘
```

메인은 sub가 작성한 파일의 *주소(reference)*만 가져온다. 본문은 디스크에 남아 있고, 메인은 경로 + "다음 해야 할 것 3가지" 같은 결정 재료만 들고 종료한다. 다음 세션이 같은 파일을 그대로 읽어 이어 쓴다.

## 패턴 선택 원칙

> **"sub-agent의 ROI는 결과를 어디에 두느냐로 결정된다."**

- **Value 패턴**: sub-agent를 *검색 함수*로 쓴다. 메인이 종합하고 판단한다. 한 번 보고 버릴 정보, 메인이 직접 가공해 사용자에게 보여야 하는 경우.
- **Reference 패턴**: sub-agent를 *저자*로 쓴다. 메인은 *인덱서/오케스트레이터*가 된다. 산출물의 수명이 세션 수명을 넘어야 하는 경우, 또는 sub의 산출 깊이가 메인 컨텍스트 한도보다 커야 하는 경우.

## 언제 어느 쪽을 고르는가

| 신호 | 권장 패턴 |
|---|---|
| 산출물을 한 번만 본다 | Value |
| 산출물을 다음 세션이 다시 읽는다 | Reference |
| sub의 답변이 메인 한도보다 길어야 한다 | Reference |
| 메인이 sub 결과를 그대로 사용자에게 보여야 한다 | Value |
| N번 재사용한다 (상각 곡선) | Reference |
| 큰 작업의 *가능 vs 불가능*을 가르는 결정이 필요하다 | Reference (enabling pattern) |

## 가드레일 — 깨끗한 비교를 위해

A안과 B안을 시연할 때 다음을 절대 어기지 말 것.

**A안(Value)에서 금지**
- sub가 파일을 작성하지 말 것.
- 메인이 리포지토리를 직접 Read하지 말 것. 모든 정보는 sub를 거쳐야 한다.

**B안(Reference)에서 금지**
- 메인이 sub가 작성한 파일을 Read하지 말 것.
- 메인이 리포지토리를 직접 탐색하지 말 것.
- sub가 요약을 반환하지 말 것. 경로 한 줄만 반환한다.

이 가드레일이 무너지면 두 실험의 차이가 흐려진다.
