# What vs How 체크리스트

작성한 PRD / Seed / AC 묶음에 How가 새어 들어왔는지 잡는 점검표다. 한 항목이라도 걸리면 그 문장을 실행 칸으로 빼낸다.

---

## How 누수 신호 (PRD·AC에 있으면 안 되는 것)

- [ ] "먼저 ~하고 그다음 ~한다" 같은 **단계 순서**가 적혀 있다.
- [ ] 특정 **도구·라이브러리·파일 구조**가 지정돼 있다.
- [ ] "이런 식으로 / 이렇게" 같은 **구현 방식**이 적혀 있다.
- [ ] 산출물의 **내부 구성 순서**(서론→본론)가 완료조건처럼 적혀 있다.

→ 위 항목은 전부 실행 단계의 자유도다. PRD·AC에서 빼면 AI가 더 나은 경로를 고를 수 있다.

## What 누락 신호 (있어야 하는데 빈 것)

- [ ] **독자**가 누구인지 PRD에 없다.
- [ ] **"끝났다"의 기준**(acceptance_criteria)이 비어 있거나 "잘 되면"으로 끝난다.
- [ ] constraints에 **넘지 말아야 할 선**(분량·인용 금지·범위)이 하나도 없다.
- [ ] goal 한 문장이 **두 가지로 해석**된다.

→ 빈 칸은 추측으로 메우지 말고 "확인 필요"로 남긴다.

## 판정 가능성 신호 (AC가 결정론으로 판정되는가)

- [ ] 각 완료조건이 **통과 / 실패**로 답할 수 있다.
- [ ] "좋다 · 깔끔하다 · 충분하다" 같은 **주관어**가 없다.
- [ ] 같은 결과를 **두 번 판정**했을 때 같은 답이 나온다.
- [ ] 다른 사람이 같은 AC로 판정해도 같은 답이 나온다.

→ 여기서 걸리면 가이드 06 [Evaluator]가 통과를 결정론으로 가를 수 없다. AC를 먼저 좁힌다.

---

## 한 줄 자가 검증

작성한 Seed를 덮어 두고, "이 묶음으로 무엇을 만들지" 한 문장으로 다시 말해 본다. 처음 적은 goal과 같은 What이 나오면 잠긴 것이다. 다르게 나오면 goal 문장이 아직 두 해석을 허용한다는 뜻이다.
