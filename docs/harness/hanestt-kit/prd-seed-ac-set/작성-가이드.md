# PRD / Seed / AC 작성 가이드

세 칸을 차례로 채운다. 위 칸이 잠겨야 아래 칸이 흐트러지지 않는다.

---

## 0. 먼저 — What과 How를 가른다

| 구분 | 들어가는 곳 | 예 |
|---|---|---|
| **What** (무엇을 / 무엇이 끝) | PRD, AC | "임원 두 명이 같은 결정 항목에 합의한다" |
| **How** (어떻게 만들지) | 여기 아닌 곳 (실행 단계) | "표를 먼저 만들고 그다음 요약을 쓴다" |

How를 PRD나 AC에 적으면, AI가 그 방법에 묶여 더 나은 경로를 못 고른다. 이 문서 세트는 What만 잠근다.

---

## 1. PRD — 무엇을, 왜

적는 것:

- **만들 대상**: 산출물이 무엇인가 (한 페이지 요약 / 답변 초안 / 기능 후보 표).
- **이유**: 이 산출물이 어떤 결정이나 행동을 가능하게 하는가.
- **독자**: 누가 읽고 무엇을 하는가.

적지 않는 것:

- 구현 단계, 도구 선택, 문장 순서 (전부 How).
- "잘 / 깔끔하게 / 보기 좋게" 같은 판정 불가능한 형용사.

한 줄 기준: PRD를 읽은 사람이 **"무엇을 왜 만드는지"**는 알지만 **"어떻게 만들지"**는 아직 모르는 상태면 잘 잠긴 것이다.

---

## 2. Seed — 고정 계약 (3슬롯)

Chapter 2에서 만든 Seed 형식을 그대로 쓴다. 세 슬롯이 다 차야 다음 단계로 넘어간다.

```yaml
goal: <한 문장. 누가, 무엇을, 누구를 위해 만드는가>
constraints:
  - <넘지 말아야 할 경계 — 분량, 인용 금지, 톤 등>
acceptance_criteria:
  - <충족하면 끝나는 조건 — 판정 가능한 문장으로>
```

- **goal**: 한 문장. 주어(누가)와 대상(무엇)과 독자(누구를 위해)가 보여야 한다.
- **constraints**: 하지 말아야 할 것, 넘지 말아야 할 선. 분량·인용 금지·톤·범위.
- **acceptance_criteria**: 이게 곧 AC Tree의 뿌리다. 다음 칸에서 쪼갠다.

슬롯이 비면 그 자리를 "확인 필요"로 남긴다. 추측으로 채우지 않는다.

---

## 3. AC Tree — 완료조건 쪼개기

acceptance_criteria 한 줄을, 사람이 매번 안 봐도 판정 가능한 작은 조건들로 쪼갠다.

원칙:

- 각 잎(leaf)은 **통과 / 실패**로 답할 수 있어야 한다.
- "임원이 만족한다"는 판정 불가 → "임원 두 명이 같은 결정 항목에 합의한다"로 좁힌다.
- 조건이 다른 사람·다른 세션에서도 같은 판정을 내게 적는다. 같은 입력에 판정이 달라지면 그 조건이 모호하다는 신호다.

이렇게 쪼갠 AC Tree가 가이드 06 [Evaluator]의 판정 입력이 된다. AC가 또렷하지 않으면 Evaluator도 결정론으로 통과를 가를 수 없다.

---

## 흔한 실패

| 증상 | 원인 | 교정 |
|---|---|---|
| AC가 "좋은 / 깔끔한"으로 끝난다 | 판정 기준이 주관어 | 통과/실패로 답할 수 있는 문장으로 |
| PRD에 단계가 적혀 있다 | How가 새어 들어옴 | 단계는 실행 칸으로 빼고 PRD엔 대상·이유만 |
| 같은 Seed로 다른 What이 나온다 | goal 문장이 두 해석을 허용 | goal을 한 해석만 남게 다시 적는다 |
| 슬롯을 추측으로 채웠다 | 입력이 부족한데 메웠다 | "확인 필요"로 남기고 사람에게 묻는다 |
