# 예시 — 마케팅 캠페인 ROI 요약 (PRD → Seed → AC Tree)

모호한 요청 한 줄을 What 묶음으로 잠근 사례다. 본인 템플릿을 채울 때 모양 참고로 쓴다.

> 처음 요청: "다음 분기 예산 정하게 캠페인 ROI 요약 좀 만들어줘."

이 한 줄에는 독자도, 분량도, "끝났다"의 기준도 없다. 아래처럼 잠근다.

---

## 1. PRD (무엇을 / 왜)

- 만들 대상: 캠페인 ROI 한 페이지 요약.
- 이유: 다음 분기 예산 배분 결정을 회의에서 바로 내리기 위해.
- 독자: 임원 두 명(결정자) + 후속 캠페인 실무자(근거 추적).
- 범위 밖: 분기별 시나리오 비교, 디자인 정리, 외부 공유용 버전.

## 2. Seed (고정 계약)

```yaml
goal: 마케팅 PM이 임원 두 명과 후속 캠페인 실무자가 함께 볼 수 있는 캠페인 ROI 한 페이지 요약을 만든다
constraints:
  - 한 페이지 분량을 넘기지 않는다
  - 미공개 마진 등 외부 사실을 인용하지 않는다
acceptance_criteria:
  - 임원 두 명이 같은 결정 항목에 합의한다
  - ROI 산출 근거가 후속 실무자에게 추적 가능하다
```

## 3. AC Tree (완료조건 쪼개기)

| 완료조건 (잎) | 통과 판정 질문 | 통과 / 실패 | 사유 |
|---|---|---|---|
| 분량 한 페이지 | A4 한 장에 들어가는가 | 통과 | 표 + 요약 12줄 |
| 외부 사실 미인용 | 미공개 마진·외부 수치가 없는가 | 통과 | 사내 집계만 사용 |
| 결정 항목 명시 | 임원이 고를 예산 항목이 표에 있는가 | 통과 | 3개 항목 + 권고안 |
| 근거 추적 가능 | 각 ROI 수치에 산출식·출처가 붙어 있는가 | 실패 | 채널 B 수치 출처 누락 |

판정: **재작업** — 채널 B 근거 한 줄만 보완하면 통과.

## 4. 잠금 확인

- 같은 Seed를 다시 진술 → "임원·실무자가 함께 보는 한 페이지 ROI 요약" 동일하게 나옴.
- How 누수: 없음 (PRD에 "표 먼저, 요약 나중" 같은 단계 없음).
- 주관어: 없음 ("보기 좋게" 대신 "한 페이지 / 추적 가능").

---

## 이 예시가 보여주는 것

처음 요청에는 독자 범위와 완료 신호가 비어 있었다. PRD에서 독자를 두 종류로 가르고, Seed의 acceptance_criteria를 "추가 질문 없이"에서 "두 명이 같은 항목에 합의한다"로 또렷하게 좁혔다. AC Tree로 쪼개고 나니 "근거 추적" 잎이 실패로 잡혔고, 막연히 다시 만드는 대신 채널 B 한 줄만 보완하면 된다는 판정이 나왔다. What을 먼저 잠갔기 때문에 재작업 범위도 한 줄로 좁혀진 것이다.
