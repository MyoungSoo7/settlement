# 결정론 게이트의 원리

Evaluator는 결과를 통과/실패로 가르는 검증기다. 핵심은 "더 엄격한 검사"가 아니라 **같은 입력에 같은 판정**이라는 점이다.

---

## 1. 왜 결정론인가

사람이 결과를 매번 눈으로 확인해야 한다면, 그건 위임이 아니라 대리 작업이다. 검증이 다음 두 조건을 만족할 때 비로소 손을 뗄 수 있다.

- **같은 입력 → 같은 결과**: 호출자·세션·시각이 달라도 PASS/FAIL이 안 바뀐다.
- **규칙 기반**: 판정이 모델의 그날 기분이 아니라 고정된 기준에서 나온다.

이 둘이 갖춰지면 사람은 검증기의 PASS만 믿고 넘어가면 된다. 이것이 위임의 전제다.

## 2. 구조 — claim과 evidence를 따로 읽는다

```text
[claim]    완료 주장 (AC 용어로 "끝냈다")
                │
[evidence] 실제 산출물 (파일·명령·출력 → 정규화된 manifest)
                │
                ▼
        규칙 기반 검증기  ── claim과 evidence를 대조
                │
                ▼
          PASS / FAIL  (+ 실패 사유)
```

핵심은 **claim과 evidence를 같은 채널로 받지 않는 것**이다. AI가 "끝냈다"고 말한 문장(claim)과, 실제로 남긴 증거(evidence)를 따로 읽은 뒤 대조한다. 증거가 있어도 그 증거가 AC 용어와 어긋나면 통과가 아니다.

## 3. 완료의 정의를 바꾼다

| | Before | 결정론 게이트 |
|---|---|---|
| 완료 신호 | AI의 "다 했습니다" 텍스트 | 검증기 PASS + 형식 있는 증거 |
| 막힘 상태 | 자유 서술 설명 | 형식 있는 blocked 증거 |
| 기준 | 사람이 매번 해석 | 고정 규칙으로 판정 |

"완료 텍스트"를 믿는 대신 "증거 → 검증기 → 판정" 순서로 완료를 확정한다.

## 4. 도메인마다 증거와 검증기가 다르다

같은 AC라도 도메인이 다르면 무엇을 증거로 보고 무엇으로 검증할지가 달라진다.

| 도메인 | 증거 | 검증기 |
|---|---|---|
| 코드 작업 | 파일 변경 + 테스트 명령 | 구조 검사 + 명령 성공 |
| 리서치 작업 | 인용된 주장 + 출처 추적 | 인용 일관성 |
| 문서 검수 | 산출물 한 장 + 금지어 목록 | 형식·의미·목적 3단 검수 |

본인 업무의 증거가 무엇인지부터 정해야 검증기를 만들 수 있다.

## 5. 통과 못하면 — 되돌린다

검증을 통과하지 못한 결과는 버리지 않는다. 수정 단계(Control)로 되돌려 다시 돌린다. 단, 무한히 되돌리지 않도록 **종료 신호**를 함께 정해 둔다.

- 재작업이 N회 같은 사유로 돌아오면 → 사람에게 보고하고 종료.
- 입력 자체가 모호해 판정이 안 되면 → 재질문(명세 갱신).
- "더 좋아질 수 있음"만으로는 재작업하지 않는다 → 충분하면 멈춘다.

---

## 한 줄

결정론 게이트는 "결과를 더 잘 만들라"가 아니라 **"이 결과를 믿고 넘어가도 되는가"**를 같은 기준으로 매번 같게 답하는 장치다. 그래서 사람이 손을 뗄 수 있다.
