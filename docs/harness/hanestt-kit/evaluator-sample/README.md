# Evaluator 예시 자료 — deterministic gate

## 한 줄

쓸 수 있는 결과만 통과시키고, 언제 멈출지 미리 정한다. 같은 입력에 같은 판정이 나오는 결정론 검증(deterministic gate)이 있어야, 사람이 매번 들여다보지 않고 AI에 맡길 수 있다.

## 이 폴더의 자료

- `결정론-게이트-원리.md` — Evaluator가 무엇이고 왜 결정론이어야 하는가. 증거 → 규칙 기반 검증기 → PASS/FAIL의 구조.
- `사례-agentos-done-report.md` — AgentOS "Context to Contract" 사례. "완료 보고는 증거가 아니다" — claim을 입력으로만 보고 실행 증거 · verifier 판정 · replay를 따로 다루는 deterministic gate.
- `evaluator.template.md` — 본인 업무용 빈 평가 기준표. 3단(형식·의미·목적) 검수 + 통과/재작업/재질문/종료 라우팅 + 종료·재질문 신호.
- `예시-평가기준표.md` — 다 채워진 한 장 예시. 고객 문의 답변 검수를 결정론 게이트로 돌린 사례.

## 어디서 쓰는가

Chapter 4 [종료 기준과 재질문 기준]과 Chapter 5 [Satisfice 멈춤 기준]의 보조 자료다. 가이드 05 [PRD / Seed / AC]에서 잠근 AC가 이 Evaluator의 판정 입력이 된다.

curriculum.md 기준 핵심 메시지: **쓸 수 있는 결과만 통과시키고, 언제 끝낼지 정한다.**

## 핵심 원리 — 왜 결정론이어야 하는가

```text
사람이 매번 눈으로 확인해야 한다  → 위임이 아니다
같은 입력 → 같은 PASS/FAIL 판정   → 비로소 위임이 된다
```

검증이 호출자나 세션마다 다른 답을 내면, 결과를 믿을 수 없어 사람이 계속 들여다봐야 한다. Evaluator의 판정이 결정론이라야 — 같은 결과를 두 번 넣어 같은 단어가 나와야 — AI에 맡기고 손을 뗄 수 있다. 통과하지 못한 결과는 버리지 않고 수정 단계로 되돌린다.

## 세 가지 판정과 멈춤

| 판정 | 언제 | 다음 행동 |
|---|---|---|
| **통과** | AC를 충족하고 남은 개선이 업무 가치에 비해 작다 | 결과·근거를 남기고 종료 |
| **재작업** | 입력은 충분한데 형식·검증·실행 중 하나가 실패 | 같은 입력으로 다시 (체크포인트로 복귀) |
| **재질문** | 입력이나 성공기준이 부족해 반복해도 수렴하지 않는다 | 사람에게 물어 명세를 갱신 |

"더 좋아질 수 있음"은 재작업 사유가 아니다. 충분하면 멈춘다(Satisfice).

## 사용 순서

1. `결정론-게이트-원리.md`로 "왜 결정론 판정이라야 위임이 되는가"를 머릿속에 정렬한다.
2. `사례-agentos-done-report.md`로 실제 deterministic gate가 완료 보고를 어떻게 다루는지 본다.
3. `예시-평가기준표.md`로 채워진 한 장의 모양을 본다.
4. `evaluator.template.md`를 본인 업무 하나로 옮겨 채운다. 입력은 가이드 05의 AC Tree를 가져온다.
5. **같은 결과를 새 세션에 두 번 넣어 판정 단어가 같은지 확인한다**(자가 검증). 다르게 나오면 판정 질문이 모호하다는 신호다.

## 완료 기준

- 3단 검수(형식·의미·목적)에 판정 질문이 한 줄씩 있다.
- 한 줄 판정이 통과 / 재작업 / 재질문 / 종료 중 한 단어로 정해진다.
- 종료 신호와 재질문 신호가 구체 조건("3회 / N건 / 외부 답 대기")으로 한 줄씩 있다.
- 같은 결과를 두 번 판정했을 때 같은 단어가 나온다(재현성).

## 원칙 — 한 줄

**검증이 결정론이라야 위임이 된다.** Evaluator는 "더 잘 만들라"고 다그치는 장치가 아니라, 쓸 수 있는 결과와 못 쓰는 결과를 같은 기준으로 가르고 언제 멈출지 정하는 장치다.
