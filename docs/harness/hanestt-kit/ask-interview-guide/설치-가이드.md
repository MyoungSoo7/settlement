# 설치 가이드

## 전제

- Claude Code (CLI 또는 IDE 확장)가 설치되어 있다.
- 본인 작업 저장소 한 개가 있다 (단일 리포지토리).
- 본 키트 폴더(`04-요구사항-인터뷰-키트/`)에 접근 가능하다.

## 절차

### 1) 작업 저장소에 `../../../../.claude/skills` 디렉토리 준비

```bash
cd <본인-저장소>
mkdir -p .claude/skills
mkdir -p .claude/scratch
```

`../../../../.claude/skills`는 Claude Code가 슬래시 명령으로 부를 스킬 정의를 두는 곳이다.
`../../../../.claude/scratch`는 키트 8개 스킬이 사이클 상태를 누적하는 곳이다.

### 2) 8개 SKILL.md를 옮긴다

이 키트의 `skills` 디렉토리를 본인 저장소의 `../../../../.claude/skills` 아래로 그대로 복사한다.

```bash
# 이 키트가 있는 경로에서
cp -r skills/* <본인-저장소>/.claude/skills/
```

옮긴 뒤 구조가 이렇게 되어야 한다:

```
<본인-저장소>/
├── .claude/
│   ├── skills/
│   │   ├── interview-harness/SKILL.md
│   │   ├── socrates/SKILL.md
│   │   ├── wonder/SKILL.md
│   │   ├── reflect/SKILL.md
│   │   ├── refine/SKILL.md
│   │   ├── restate/SKILL.md
│   │   ├── evolve-step/SKILL.md
│   │   └── ontology/SKILL.md
│   └── scratch/
│       └── (비어 있음, 첫 호출 때 채워짐)
└── ...
```

### 3) 스크래치 디렉토리 권한 확인

스킬들이 `../../../../.claude/scratch` 아래에 마크다운 파일을 쓰고 읽는다. 디렉토리 쓰기 권한이 필요하다.

```bash
ls -ld .claude/scratch
# drwxr-xr-x ... 가 보이면 정상
```

### 4) 첫 호출로 확인

새 Claude Code 세션을 열고 다음 한 줄을 입력한다.

```
/interview-harness
```

키트가 정상이라면:
- 스킬을 인식해 입력 요청이 뜬다 ("막연한 요청 한 줄을 입력하세요").
- 입력 후 `/socrates`가 호출되고 `### Wonder` 단계가 시작된다.
- `../../../../.claude/scratch/socrates.md`가 생성된다.

### 5) sub 스킬 단독 호출 회귀 점검

키트의 핵심 설계는 *분리는 그대로, 사용은 통합*이다. 마스터 호출이 동작해도 sub가 단독 호출이 안 되면 키트가 깨진 것이다.

새 세션을 각각 열어 다음을 확인:

```
/socrates       → Seed v1까지만 만들고 종료해야 한다
/wonder         → 의미 발산 한 단계만 하고 종료해야 한다
/evolve-step    → 기존 Seed 위에 사각지대 점검 한 사이클만 돌아야 한다
/ontology       → 기존 Seed 위에 의미 경계만 정해야 한다
```

각 단독 호출이 본인 스킬 범위를 *넘지 않는지* 본다 (예: `/wonder`가 reflect까지 진행하면 안 된다).

## 문제 해결

### `/interview-harness`가 인식되지 않는다

- `../../../../.claude/skills/interview-harness/SKILL.md`가 본인 저장소 안에 있는지 확인.
- Claude Code를 한 번 재시작하면 스킬 목록을 다시 읽는다.

### 사이클이 5번 도는데도 수렴 안 함

- `../../../../.claude/scratch/evolve-step.md`를 열어 사용자 채택률을 본다. 매 사이클 거의 다 거절했다면 Seed가 이미 닫혀 있는 것에 가깝다. 첫 사이클에서 적은 만큼만 채택하는 것이 정상.
- `.claude/scratch/interview-harness.md`의 similarity 로그를 본다. boundary 점수가 계속 0이면 boundary 표현이 매 사이클 바뀌고 있을 수 있다. ontology 호출 시 boundary 한 문장을 더 좁게 쓴다.

### `../../../../.claude/scratch/socrates.md`에 Seed 블록이 여러 개

- canonical 정책 위반. 최상위 `## Seed`는 항상 한 벌만 있어야 한다.
- 수동으로 가장 최신 한 벌만 남기고 나머지 `## Seed` 블록을 지운다.
- `evolve-step` 또는 `interview-harness`가 이 정책을 어겼다는 신호이므로 SKILL.md를 다시 확인.

## 키트 업데이트

이 키트의 8개 SKILL.md가 갱신되면 위 절차 2번을 다시 실행한다(덮어쓰기). 본인이 SKILL.md를 직접 수정했다면 그 변경분은 사라지므로, 본인 변경분은 별도 브랜치/포크로 관리한다.
