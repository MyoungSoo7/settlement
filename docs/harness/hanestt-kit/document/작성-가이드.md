# 작성 가이드 — 7개 칸을 어떻게 채우는가

`skill.template.md`의 빈 칸을 본인 업무에 맞게 채울 때 참고한다.

---

## 1) frontmatter — `name`

kebab-case. 슬래시 명령으로 부를 이름이다. 짧고 동사적이면 좋다.

- ✅ `pr-review`, `commit-message`, `interview-harness`
- ❌ `MyPRReviewer`, `pr_review_v3`, `the-best-skill`

## 2) frontmatter — `description`

세 부분으로 나눠 적는다.

**(a) 한 문장 정의** — 무엇을 입력으로 무엇을 만드는지.
> 예: "PR diff를 검수해 코멘트 후보 리스트와 머지 가능 여부 한 줄을 만든다."

**(b) Triggers (KO/EN)** — 본인이 이 스킬을 부르고 싶을 때 자연스럽게 입에 붙는 표현을 3~5개씩.
> 예 (KO): "PR 리뷰해줘", "diff 검수", "머지해도 돼?"
> 예 (EN): "review this PR", "audit diff"

**(c) Do NOT use when** — 이 스킬이 *아닌* 경우. 인접 스킬과의 경계.
> 예: "단순 commit message 생성은 `/commit` 스킬을 쓴다."

## 3) 역할

"당신은 X 역할을 한다"가 아니라 **"이 스킬은 X를 담당한다"** 형태. 인격을 부여하지 않는다(메인 세션이 이 문서를 *불러서 쓰는* 거지, 이 문서가 *말하는* 게 아니다).

- ✅ "이 스킬은 PR diff를 검수하고 코멘트 후보를 만든다."
- ❌ "당신은 시니어 리뷰어다. 당신의 임무는…"

## 4) 입력

입력 한 가지를 정확히 적는다. 두 가지 이상이면 스킬을 둘로 쪼개는 게 맞을 가능성이 높다.

- ✅ "PR 번호 한 개. `gh pr view <num>`으로 diff를 가져온다."
- ❌ "PR 정보 또는 commit hash 또는 파일 경로 또는…"

## 5) 출력 — 가장 중요한 칸

`md형` 하네스의 핵심이 여기다. **출력 스키마를 코드 블록으로 명시한다.** 같은 입력에 같은 형식이 나와야 재현성이 생긴다.

- 슬롯 이름은 정해진 영문 키로 (오타 방지).
- 빈 슬롯을 어떻게 표기하는지 규칙을 정한다 (`empty`, `null`, `___` 중 하나).
- list는 최소 개수를 명시한다 (예: "1개 이상" / "정확히 3개").

> 예시:
> ```yaml
> verdict: pass | fail | hold
> reasons:
>   - <한 줄 사유 1>
>   - <한 줄 사유 2>
> comments:
>   - file: <path>
>     line: <int>
>     body: <한 줄>
> ```

## 6) 원칙 — *항상 한다*

3~5개. 한 줄씩. 추상어 금지(`정확하게`, `사용자 중심으로` 같은 말은 검증 불가).

- ✅ "원칙: 코멘트 한 건당 한 파일만 가리킨다."
- ❌ "원칙: 사용자 친화적으로 작성한다."

## 7) 금지사항 — *절대 하지 않는다*

원칙의 거울. 흔히 실패하는 패턴을 미리 막는다.

- ✅ "검수 대상이 아닌 파일을 임의로 수정하지 않는다."
- ✅ "diff에 없는 라인 번호를 만들지 않는다."
- ❌ "잘못된 답을 하지 않는다." (너무 추상)

## 8) 예시 입출력

**한 쌍만** 둔다. 현실적인 입력 + 그에 대한 정확한 출력. 출력은 위 *출력 스키마*와 한 글자도 어긋나지 않아야 한다.

예시가 여러 개 필요하다면 스킬 범위가 너무 넓다는 신호다. 쪼갠다.

## 9) 인접 스킬과의 관계

본인 작업에 다른 스킬이 있다면 경계를 한 줄로 명시한다. 없다면 비워둔다.

> 예: "vs `/commit-message`: 이 스킬은 PR 단위 검수, `/commit-message`는 commit 단위 메시지 생성."

---

## 자가 검증 — md형이 작동하는지 확인하는 한 가지 시험

같은 요청을 **두 번 입력해 출력 형식이 일치하는가**.

- 일치 → md형 하네스가 작동한다. 끝.
- 일치하지 않음 → *출력* 절 스키마를 더 좁히거나, *예시*를 더 정확히 한다. 본문 *원칙·금지*에 더 적는 것은 마지막 수단이다(문서가 길어질수록 효과는 약해진다).

---

## 흔한 실수 5개

1. **출력 스키마를 산문으로만 적는다** — 코드 블록으로 정확히 적어야 재현성이 생긴다.
2. **원칙 칸에 추상어를 쓴다** — `정확하게` / `자연스럽게` 같은 말은 검증 불가.
3. **인격을 부여한다** — "당신은 시니어 X" 식 페르소나가 길어지면 본문이 길어지기만 한다.
4. **예시를 너무 많이 둔다** — 한 쌍이면 충분. 여러 쌍이 필요하면 스킬을 쪼갠다.
5. **금지사항이 비어 있다** — 흔히 실패하는 패턴이 명시되지 않으면 같은 실수가 반복된다.
