# AI Native 마인드셋 체크리스트

> 점검 대상: `practice-materials/chapter-05/2005-4-3-workspace` 에서 실행한 meeting-to-1page 하네스 v2.
> 각 칸은 워크스페이스의 실제 contract·handoff·run-log 증거로 채웠다. 검증되지 않은 수치는 넣지 않았고, 본인 실제 환경에 맞춰야 하는 칸만 `확인 필요`로 남겼다.

## 1. 업무와 결과

| 항목 | 내용 |
|---|---|
| 점검할 업무 | 회의록 한 건을 1page 기획 초안(7칸)으로 변환. capability graph 13 node 하네스 v2로 실행 |
| 최종 결과 | review_gate 결정론 검사 + L5(인용)∥L6(사실·범위) 검수를 통과한 7칸 1page 초안. leaf별 verdict·route, append-only run-log, checkpoint, 확인 필요 항목 분리본이 함께 남는다 |
| 비즈니스 임팩트 | 의사결정자가 추가 질문 없이 "사내 교육 신청 페이지 개편 방향(강조 화면 + 자동 메일)" 채택 여부를 결정한다. 같은 회의록을 매번 같은 형식·기준으로 변환해 결정 지연과 미확인 정보 혼입을 줄인다 (검증된 수치 없음 — CS 문의 60%는 추정·미확인으로 본문에서 분리) |
| 현재 AI 사용 방식 | 분업형 하네스. ask/build/review leaf를 runtime adapter(replay·Claude Agent SDK·codex exec)로 분리 실행하고, MCP로 review_parallel 한 node 또는 graph 전체를 호출한다. 순서·state·route는 모델이 아니라 코드가 정한다(결정론) |

## 2. 코드가 아닌 파이프라인

| 단계 | 현재 방식 | AI가 맡을 일 | 사람이 판단할 일 | 산출물 |
|---|---|---|---|---|
| 입력 | 회의록 원문(meeting.md) + 2장 requirements-contract + 3장 spec-contract를 frozen 계약으로 통합 | 의도·제약 분리, evidence slice(meeting.md:NN) 추출 | 어떤 회의를 다룰지, 성공 기준을 목표 문장으로 둘지 측정 지표로 둘지 | frozen requirements/spec contract + structured-contract.json (SHA-256 잠금) |
| 실행 | fill_handoff → dispatch_plan → invoke_runtime_adapter (replay / claude / codex) | ask/build leaf 초안 생성, handoff 정규화(비계약 상태 폐기) | 방향 결정, ask-back 사각지대 질문 채택(☑/☒) | handoff-L1-ask, handoff-L3L4-build, _brief |
| 검수 | review_gate 결정론 precheck → review_parallel(L5∥L6) → merge_verdict(퇴행 검사) | 결정론 5종 검사 + 인용 보존(L5)·사실/범위/누락(L6) 의미 검수 | 최종 수용, exit 선언 | handoff-L5-review, handoff-L6-review, run-log |
| 공유 | append-only run-log(.md/.jsonl) + checkpoints + pending-external + README 운영 가이드 | 실행 기록 요약·문서화 | 팀 공유 기준 확정 | final-1page-draft + 운영 가이드(README, demo.md) |

## 3. 검수와 운영 기준

| 기준 | 내용 |
|---|---|
| 성공기준 | 7칸이 meeting.md:NN 근거와 함께 채워지고, 결정 항목과 확인 질문이 분리되며, 의사결정자가 추가 질문 없이 결정으로 넘어간다. review_gate 통과 + L5 인용 통과 + L6 사실·범위 통과 + 이전 통과 퇴행 없음 |
| 실패 신호 | 회의록 밖 사실을 결론으로 단정 / 미확인 수치(CS 60%)를 본문에 단정 / 미결정을 결론 톤("~로 한다")으로 작성 / meeting.md:NN 줄번호 불일치 / 보존 표현 3줄 누락 |
| 중단 기준 | 같은 사유로 retry 2회면 blocked exit. last_good_checkpoint + pending을 보존하고 무한 재작업하지 않는다 |
| 재질문 기준 | acceptance.metric 등 회의록 밖 결정이 필요한 칸은 ask-back(requirements_update)으로 돌린다. 2장 사각지대 두 질문 + 사람 채택 구조로만 갱신하고 추측으로 채우지 않는다 |
| 사람이 최종 책임질 판단 | 회의 대상 선택 / 성공 기준의 목표·지표 결정 / 1page 최종 채택 / exit 선언 |

## 4. 팀으로 넘길 항목

- 공유할 하네스/규칙: frozen requirements/spec contract + 7칸 output contract + handoff 스키마 + review_gate profile(5종 검사)
- 공유할 실행 기록: append-only run-log.jsonl + checkpoints + L5/L6 verdict + pending-external
- 팀과 맞출 검수 기준: review_gate 5종 precheck 합격선 + L5 인용 / L6 사실·범위 판정 기준 + 보존 표현 큰따옴표 규칙
- 다음 실습으로 넘길 요구사항: (1) run-log·verdict를 팀이 읽을 수 있는 관측 포맷으로 정리 (2) 공유 contract 재사용 구조 (3) 리뷰 합격선 팀 합의

## 5. 다음 행동

- 다음 업무에서 바로 바꿀 행동: 다음 실행부터 append-only run-log와 L5/L6 verdict·route·checkpoint를 팀이 그대로 읽을 수 있는 관측 포맷으로 정리해 남긴다. 한 사이클 = 한 행(leaf · verdict · route · 재질문 사유 · 남은 외부 회신)으로 고정해, 누가 어디서 막혔는지 팀이 보이게 한다 (6장 동기2: 잘 못 쓰는 사람이 안 보임)
- 바꾸지 않을 것: frozen contract + review_gate → review_parallel 순서(결정론 guard)는 유지한다. 미확정 metric은 계속 ask-back으로만 채운다
- 확인할 사람 또는 리뷰 방식: 1page를 받는 의사결정자가 exit·채택을 닫는다. 관측 포맷의 컬럼 정의는 팀 리뷰로 한 번 합의 — `확인 필요`(실제 팀 리뷰어·관측 도구는 본인 환경에 맞춰 지정)
