# 팀 AI 인프라 요구사항 명세

> 입력: `2006-1-1 AI Native 마인드셋 체크리스트`(회의록→1page 하네스 v2)
> Zeude 관찰은 공유·동기화 / 모니터링·케어 / 빌더·랭킹 / 운영 기준 네 관점으로만 사용했다. 도입 효과·스케일·런타임 동작은 추측하지 않았고, 팀 현실에 맞춰야 하는 칸만 `확인 필요`로 남겼다.

## 1. Scope

| 항목 | 내용 |
|---|---|
| 대상 팀 | 부서 혼합(크로스펑셔널) 팀. 기획·개발·콘텐츠가 섞여 각자 `claude` 또는 `codex`로 회의록→1page 작업을 돌린다. (정확한 팀명·인원은 `확인 필요`) |
| 공유할 하네스/스킬 | 하네스 v2 자산 묶음 — frozen requirements/spec contract + 7칸 output contract + handoff 스키마 + review_gate profile(결정론 5종) + append-only run-log/verdict 포맷 |
| 해결할 운영 문제 | 네 가지가 동시에 막힌다. ① 공유해도 각자 안 깔고 버전이 제각각 ② 누가 어디서 막히는지 안 보인다 ③ 비개발자가 만든 스킬이 안 퍼진다 ④ 검수 합격선이 사람마다 다르다 |
| 최종 사용자 | 회의록→1page를 실행하는 팀원, 결과를 검수하는 검수자, 셋업·발급을 맡는 팀 리드(운영자) |
| 성공기준 | 한 줄 설치로 모두가 같은 버전의 하네스·스킬·review_gate를 쓰고, 실행이 사용자 단위로 관측되며, 비개발자 스킬이 사용량으로 드러나고, 검수 합격선이 팀 공통으로 통일된다 |

## 4. Constraints and Boundaries

- **접근 권한/보안**: 운영자(팀 리드/인프라 담당)가 `agent_key`를 발급·회수한다. 자격증명(설정·분석 백엔드, credentials)은 개인 머신에 평문으로 두지 않는다. 부서별로 어떤 스킬을 열람·수정할 수 있는지 권한 분리는 `확인 필요`. 회의록 원문에 민감 정보가 있으면 공유 contract 예시에 넣지 않는다.
- **팀에서 먼저 합의할 규칙**: 스킬·contract 명명 규칙, 버전 태깅 기준, review_gate 5종 합격선과 L5(인용)/L6(사실·범위) 판정 기준. 회의록→1page의 **최종 채택·exit 선언 책임은 의사결정자에게** 둔다.
- **지금 만들지 않을 것**: 모든 업무를 한 번에 자동화하는 통합 시스템. 미확정 metric을 추측으로 채우는 자동화. (미확정 칸은 계속 ask-back으로만 채운다 — 2006-1-1 유지 항목)
- **검증 없이 주장하지 않을 것**: 생산성 향상 폭, 비용 절감, 도입 효과 수치, 외부 사례 비교. Zeude의 설치·런타임·채택·스케일 동작은 관찰 범위 밖이라 근거로 쓰지 않는다.

## 5. Roadmap Handoff

> 동기 네 가지를 모두 짚었지만, 의존 순서가 있다. 공유·관측·리뷰가 깔린 뒤라야 비개발자 스킬이 같은 기준에서 측정되므로, 빌더·랭킹은 4순위로 다음 차수에 넘긴다.

| 우선순위 | 로드맵으로 넘길 요구사항 | 이유 |
|---|---|---|
| 1 | 공유·동기화 0터치 배포 (첫 대상: frozen contract + review_gate profile + 7칸 output contract) | 안 깔리고 버전이 제각각이면 나머지 세 축이 전부 의미가 없다. 크로스펑셔널이라 도구가 섞여 있어 한 줄 설치 효과가 가장 크다 |
| 2 | 관측 포맷: 한 사이클 = 한 행 (leaf · verdict · route · 재질문 사유 · 남은 외부 회신) | 2006-1-1이 이미 다음 행동으로 지목한 항목. "잘 못 쓰는 사람이 안 보임"을 사용자 단위 상태판으로 푼다 |
| 3 | 리뷰 합격선 팀 합의 (review_gate 5종 + L5/L6 판정 기준 통일) | 검수 기준이 사람마다 다른 문제를 닫는다. 공유·관측이 깔린 위에서 합의해야 같은 실행 기록으로 검증된다 |

### 확인 필요 (본인 환경)

- 대상 팀의 정확한 명칭·인원·부서 구성
- 대시보드 호스팅 위치(self-host URL)와 설정·분석 백엔드 자격증명 보관 정책
- `agent_key` 발급·회수 정책 세부 (운영자는 팀 리드/인프라 담당으로 확정)
- 부서별 스킬 열람·수정 권한 분리 기준
- 실제 팀 리뷰어와 관측 도구 지정 (2006-1-1에서 이월된 `확인 필요`)
