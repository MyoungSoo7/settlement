# 우리 팀 플레이그라운드 로드맵

> 입력: `2006-1-1 마인드셋 체크리스트` + `2006-1-2 팀 AI 인프라 요구사항 명세`
> 우선순위 3개(공유→관측→리뷰 합격선)는 2006-1-2에서 좁힌 그대로 가져왔다. "리뷰 합격선 통일"은 별도 단계가 아니라 Operating Board의 매주 리뷰에 녹였고, 빌더·랭킹이 3단계로 올라왔다. 첫 적용은 소수 파일럿 3~5명으로 작게 시작한다.

## 1. Playground Scope

| 항목 | 내용 |
|---|---|
| 대상 팀 | 부서 혼합(크로스펑셔널) 팀. 첫 파일럿은 3~5명 (실제 멤버는 `확인 필요`) |
| 첫 적용 업무 | 회의록 한 건 → 1page 기획 초안(7칸). capability graph 13 node 하네스 v2 |
| 참여자 | 운영자 = 팀 리드/인프라 담당, 사용자 = 파일럿 3~5명, 리뷰어 = 1page를 검토하는 의사결정자 |
| 공유할 하네스/스킬 | frozen requirements/spec contract + review_gate profile(결정론 5종) + 7칸 output contract + run-log/verdict 포맷 |
| 최종 목표 | 한 줄 설치로 모두 같은 버전을 쓰고, 실행이 사용자 단위로 관측되며, review_gate 합격선이 팀 공통으로 통일된 상태 |

## 2. Roadmap

| 단계 | 목표 | 행동 | 산출물 | 담당 | 완료 기준 |
|---|---|---|---|---|---|
| 1 · 공유→설치 보장 | 공유한 하네스가 파일럿 머신에 같은 버전으로 깔린다 | 첫 공유 자산(frozen contract + review_gate profile + 7칸 output contract)을 운영자가 0터치 설치 경로 한 줄로 배포하고 agent_key를 발급한다 | 공유 자산 목록 + 설치 경로 + agent_key 발급 대장 | 팀 리드(운영자) | 파일럿 3~5명이 같은 버전의 contract·review_gate를 쓴다 |
| 2 · 모니터링→케어 | 헤매는 사람을 찾아 돕는다 (감시가 아니다) | 한 사이클 = 한 행(leaf · verdict · route · 재질문 사유 · 남은 외부 회신)으로 운영 보드를 쓰고, 채택률·미사용자를 매주 본다 | 운영 보드 + 케어 기록 | 케어 담당 (`확인 필요` — 운영자 겸임 가능) | 미사용자·반복 실패 지점을 찾아 그 주에 돕는다 |
| 3 · 빌더→랭킹→전환 | 비개발자(기획·콘텐츠)가 만든 스킬이 퍼진다 | 비개발자 빌드 통로를 열고, 사용량 랭킹으로 많이 쓰인 스킬을 드러내 다음 파일럿 범위로 넓힌다 | 스킬 랭킹/리더보드 + 빌더 가이드 | 팀 리드 + 큐레이션 담당 | 비개발자 스킬이 랭킹에 오르고 빌더가 는다 |

## 3. Operating Board

| 운영 항목 | 기준 |
|---|---|
| 관측할 신호 | leaf별 verdict·route, 재질문 사유, 남은 외부 회신, 채택률, 미사용자, 같은 사유 retry 횟수 |
| 리뷰 주기 | 매주 한 번 — 최근 실행 몇 건과 운영 보드를 검토한다 |
| 실패 사유 기록 방식 | append-only run-log(.md/.jsonl) — 실패 유형 · 복귀 지점(last_good_checkpoint) · 다음 수정 항목을 남긴다 |
| 버전 변경 방식 | 운영자 승인. 변경 이유와 적용일을 버전 기록에 남긴다 |
| 검수 합격선 (2006-1-2 우선순위 3) | review_gate 5종 합격선 + L5(인용)/L6(사실·범위) 판정 기준을 매주 리뷰에서 합의·조정해 팀 공통으로 통일한다 |
| 중단 또는 조정 기준 | 같은 사유로 retry 2회면 blocked exit. 효과 없음·오용·관리 부담이 크면 파일럿 범위를 넓히지 않고 줄이거나 조정한다 |
| 사람이 최종 판단할 항목 | 회의 대상 선택 / 성공 기준의 목표·지표 결정 / 1page 최종 채택·exit 선언 / agent_key 발급·회수 승인 |

## 4. Review Script

- 이번 주 실행 결과: `확인 필요` — 첫 사이클을 돌린 뒤 채운다
- 반복된 실패: `확인 필요`
- 다음에 고칠 하네스/규칙: `확인 필요`
- 팀에 공유할 결정: review_gate 5종 합격선과 L5/L6 판정 기준 1차 합의
- 다음 리뷰 날짜: `확인 필요` — 매주 같은 요일로 고정

## 5. Guardrails

- 하지 않을 확산: 파일럿이 정착하기 전의 한 직군 전체·전사 확산
- 검증 없이 주장하지 않을 것: 생산성 향상 폭, 비용 절감, 도입 효과 수치, 외부 사례 비교, Zeude의 채택·스케일·런타임 동작
- 사람 책임으로 남길 것: 문제 정의, 성공 기준 결정, 1page 최종 채택·exit 선언, agent_key 발급·회수 승인
