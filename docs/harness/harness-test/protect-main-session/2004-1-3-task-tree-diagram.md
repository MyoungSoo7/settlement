# 2004-1-3. 1-shot SKILL의 AC tree 분해 — 분업 가능성 진단

## 이 산출물
chapter 3에서 만든 1-shot SKILL의 AC들을 ask / build / review 3분할 AC tree로 재정렬한 한 장짜리 다이어그램이며, `2004-2-3` handoff 양식 실습의 입력이 된다.

## 목적

chapter 3에서는 한 SKILL.md 안에 역할·입력 명세·도메인 제약·처리 순서·출력 검사를 통째로 적었다. 한 세션이 그 묶음을 한꺼번에 떠안고 돌렸다. 본 실습은 그 묶음을 ask / build / review 세 책임으로 다시 갈라 본다. "이 SKILL이 어디까지 분업할 가치가 있는가"를 진단하는 첫 단계다. 새 업무를 분해하는 일이 아니라, **이미 있는 1-shot SKILL을 sub agent 후보 트리로 다시 본다**.

## 권장 시간

30분

## 준비물

- 분석 대상 SKILL — 다음 중 하나:
  - **(권장)** chapter 3 프로젝트 1 산출물: `meeting-to-1page` SKILL.md (또는 본인이 chapter 3에서 만든 도메인 SKILL.md)
  - **(대안)** 본인이 이미 운영 중인 1-shot SKILL.md 또는 자주 쓰는 프롬프트 묶음 한 건
- 그 SKILL의 frontmatter description 한 줄 (Parent 한 줄로 사용)
- 그 SKILL 산출물의 검수자

## 입력

아래 입력을 먼저 채운다.

| 항목 | 내용 |
|---|---|
| 대상 SKILL 이름 |  |
| SKILL frontmatter description (Parent 한 줄) |  |
| 검수자 |  |
| SKILL.md 본문 5자리에서 뽑은 AC raw 목록 |  |
| 4장에 안 가져올 AC (1-shot으로 충분한 것) |  |

## AI에게 시키기

위 입력표를 채워 그대로 붙인 뒤, 한 줄을 덧붙인다.

> 이 SKILL.md의 AC를 ask / build / review 3분할 AC tree로 정렬해줘. 아래 작성 템플릿 구조를 따르고, 모르는 AC는 추측하지 말고 "확인 질문"으로 남겨. SKILL.md에 없는 AC를 새로 만들지 마. 각 리프 옆에는 SKILL.md의 어느 섹션에서 왔는지 출처를 적어.

## 단계별 진행

1. **AC raw 추출** — SKILL.md 5자리에서 한 줄씩 옮긴다:
   - 역할 / 책임 범위 → 책임 AC
   - 입력 명세의 "비어 있으면 질문할 항목" → ask 후보 AC
   - 도메인 제약 → build 가드 AC
   - 처리 순서 N단계 → build 행동 AC
   - 출력 형식 + 보존 표현 검사 → review AC
2. raw AC를 **ask / build / review** 세 묶음으로 분류한다.
3. 각 묶음 안에서 한 보조 세션이 받을 수 있는 리프로 쪼갠다.
4. 각 리프에 네 칸(목적·입력·범위·완료조건)을 한 줄씩 적는다.
5. 세 검사를 돌린다 — Parent 합 / 겹침 / Orphan.
6. **분업 가치 판정** — 리프 하나하나에 "1-shot으로 충분 vs sub agent로 분리할 가치 있음"을 한 줄씩.
7. 4-2 실습으로 가져갈 리프 1개를 표시한다 (build 묶음에서 고르는 게 권장).

## 작성 템플릿

````markdown
# AC tree — <대상 SKILL 이름>

## 1. Parent (SKILL 한 줄)
- SKILL description:
- 검수자:
- 4장에 안 가져올 AC:

## 2. AC tree
```text
<SKILL frontmatter description>
├─ ask (질문 책임)
│  ├─ L1 ...
│  └─ L2 ...
├─ build (생성 책임)
│  ├─ L3 ...
│  └─ L4 ...
└─ review (검수 책임)
   ├─ L5 ...
   └─ L6 ...
```

## 3. 리프 카드 표
| 리프 ID | 묶음 | 목적 | 입력 | 범위 (하지 않을 일 포함) | 완료조건 | SKILL.md 출처 |
|---|---|---|---|---|---|---|
| L1 | ask |  |  |  |  | 섹션명 / 줄번호 |
| L2 | ask |  |  |  |  | 섹션명 / 줄번호 |
| L3 | build |  |  |  |  | 섹션명 / 줄번호 |
| L4 | build |  |  |  |  | 섹션명 / 줄번호 |
| L5 | review |  |  |  |  | 섹션명 / 줄번호 |
| L6 | review |  |  |  |  | 섹션명 / 줄번호 |

## 4. 세 검사
- Parent 합: ask + build + review 합으로 SKILL description 한 줄이 닫히는가
- 겹침: 두 리프가 같은 산출물을 만들지 않는가
- Orphan: ask/build/review 셋 중 하나에 배정 안 된 리프가 없는가

## 5. 분업 가치 판정
- 분업 가치 있음 (sub agent로 분리할 가치):
- 분업 가치 낮음 (1-shot으로 충분 — 4장에 안 가져옴):

## 6. 4-2로 넘길 리프
- 선택한 리프 ID:
- 선택 이유:
````

## 예시 — meeting-to-1page SKILL 기준

````markdown
# AC tree — meeting-to-1page

## 1. Parent
- SKILL description: 회의록을 7칸 1page 기획안 초안으로 변환
- 검수자: 1page를 받는 의사결정자
- 4장에 안 가져올 AC: 처리 순서 1·2단계(원문 보존 · 확정/미결정 분리) — 1-shot 한 패스에 끝남

## 2. AC tree
```text
회의록 → 1page 7칸 변환
├─ ask (질문 책임)
│  ├─ L1 결정 항목 추출 질문
│  └─ L2 보존 표현 후보 추출
├─ build (생성 책임)
│  ├─ L3 6칸 맵핑
│  └─ L4 확인 필요 항목 한 칸 모음
└─ review (검수 책임)
   ├─ L5 보존 표현 인용 검사
   └─ L6 도메인 제약 위반 검사
```

## 3. 리프 카드 표
| 리프 ID | 묶음 | 목적 | 입력 | 범위 | 완료조건 | SKILL.md 출처 |
|---|---|---|---|---|---|---|
| L1 | ask | 비어 있는 결정 항목을 의사결정자 질문 표로 | 회의록 원문 | 결정권자·일정·수치 출처·환불 기준 4개. 답 작성 금지 | 질문 4개 + 의사결정자 한 줄씩 | 입력 명세 "비어 있으면 질문할 항목" |
| L2 | ask | 보존 표현 후보 추출 | 회의록 원문 | 큰따옴표 인용 3줄. 의역 금지 | 보존 표현 3줄 + 출처 인용 줄번호 | 도메인 제약 "보존 표현" |
| L3 | build | 회의록 확정 내용을 6칸으로 맵핑 | 회의록 확정 + L1 결과 | 7칸 중 "확인 필요" 칸 제외 6칸. 외부 사실 추가 금지 | 6칸 모두 한 줄 이상 | 처리 순서 3단계 |
| L4 | build | 미결정 항목을 "확인 필요" 한 칸으로 모음 | 회의록 미결정 + L1 결과 | 7칸 중 마지막 칸 | 미결정 모두 한 칸으로 묶임 | 처리 순서 4단계 |
| L5 | review | 보존 표현 인용 검사 | L3·L4 결과 + L2의 3줄 | 3줄 인용 일치 여부만. 본문 재작성 금지 | 통과/실패 + 위반 줄번호 | 처리 순서 5단계 + check-1page-output.sh |
| L6 | review | 도메인 제약 위반 검사 | L3·L4 결과 | 외부 사실·미확인 수치 단정·범위 밖 항목 | 통과/실패 + 위반 항목 한 줄씩 | 도메인 제약 |

## 4. 세 검사
- Parent 합: ask 2 + build 2 + review 2 = 7칸 채운 1page. OK
- 겹침: L3·L4는 다른 칸을 채움. L5·L6은 다른 가드. OK
- Orphan: 모든 리프가 ask/build/review 중 하나에 배정. OK

## 5. 분업 가치 판정
- 분업 가치 있음:
  - L1 ask — 회차별 의사결정자 답이 누적돼야 함. 1-shot 안에선 추적이 안 됨.
  - L5·L6 review — 1-shot 안에선 같은 세션의 관성 때문에 자기 결과를 자기가 통과시키는 경향.
- 분업 가치 낮음:
  - L3·L4 build — 처리 순서가 짧음. 1-shot으로 충분. 단 회의록이 길거나 검수자가 별도 build 세션을 원하면 가져옴.
  - L2 ask — 인용 추출은 build에 흡수해도 됨. 분업 가치 낮음.

## 6. 4-2로 넘길 리프
- 선택한 리프 ID: L3
- 선택 이유: build 묶음의 가장 큰 단위. handoff 양식 9칸을 채워 보기에 적합하다.
````

## 완료 기준

- AC raw 목록이 SKILL.md 출처(섹션명 / 줄번호)와 함께 적혀 있다.
- 모든 리프가 ask / build / review 셋 중 하나에 배정되어 있다.
- 각 리프의 네 칸이 채워져 있다.
- 분업 가치 판정 한 줄이 모든 리프에 적혀 있다 (있음 / 낮음).
- 4-2로 가져갈 리프 1개가 선택되어 있다.

## 자기 검수 체크리스트

- [ ] 입력 SKILL의 frontmatter description이 Parent 한 줄로 그대로 옮겨졌다.
- [ ] 모든 리프 AC에 SKILL.md 출처(섹션명 또는 줄번호)가 표시되어 있다.
- [ ] ask / build / review 한 묶음 안에 두 책임이 섞여 있지 않다.
- [ ] SKILL.md에 없는 AC를 추측으로 추가하지 않았다.
- [ ] 분업 가치 낮은 리프는 4장에 안 가져온다고 명시되어 있다.
- [ ] 같은 SKILL을 두 번 분석했을 때 같은 묶음으로 떨어진다 (재현 가능성 자가 시험 — 4장 척추).
