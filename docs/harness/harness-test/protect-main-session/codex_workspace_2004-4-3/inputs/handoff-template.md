## 2. 예시 — L3 build 세션 → 메인 (채운 handoff)

````markdown
# Handoff — L3 · build 세션 → 메인

## From task
- 목적: 회의록 확정 내용을 제목·배경·문제·사용자·해결 방향·성공 기준 6칸으로 맵핑
- 입력: 회의록 확정 묶음 + L1 답(결정권자·일정·수치 출처·환불 기준) + L2 보존 인용 3줄
- 범위: 7칸 중 "확인 필요 항목"을 제외한 6칸만. 그 칸은 L4 build 몫이라 손대지 않는다
- 금지: 회의록에 없는 수치·일정·담당자·환불 기준을 결론처럼 쓰기. 단서 붙은 수치를 본문 6칸에 넣기
- 완료조건: 6칸이 각각 한 줄 이상씩 채워진다

## Result
- 결과 요약:
  | 칸 | 채운 내용 |
  |---|---|
  | 제목 | 사내 교육 신청 페이지 개편 v1 |
  | 배경 | 분기 사내 교육 운영 중 신청 단계 이탈·문의가 늘어 회의 소집 |
  | 문제 | 일정·환불 안내가 신청 흐름에서 안 읽혀 신청 후 CS 문의 발생 |
  | 사용자 | 사내 교육 신청자 + 운영/CS팀 |
  | 해결 방향 | 신청 폼 진입 전에 일정·환불 안내를 단계별로 노출 |
  | 성공 기준 | 신청 후 일정·환불 관련 CS 문의를 줄인다 (← 측정 단위 미정, 모호) |
- 판정: 통과 (자체 검수 기준 — 6칸 모두 한 줄 이상)
- 남은 질문: 성공 기준을 측정 가능한 한 줄로 더 좁혀야 하나? (CS 문의 비중 수치는 L1 미회신이라 본문 단정 보류)
- 다음 세션의 첫 행동: review 세션에 6칸 초안 + L2 보존 인용 3줄 + L3 완료조건을 함께 보낸다 (L5 인용 일치 / L6 도메인 제약 검사)
````

## 3. 자기 검수 체크리스트

- [x] handoff가 한 화면에 다 들어온다.
- [x] "잘 됐다 / 안 됐다" 같은 모호한 판정 표현이 없다. (통과 한 단어로 정해짐)
- [x] 결과 요약에 보조 세션의 대화 흔적(인용·메타 설명)이 남아 있지 않다. (6칸 표만)
- [x] 다음 세션의 첫 행동이 "검토 부탁" 같은 모호 표현이 아니다. ("~를 보낸다" 동사 + 목적어)
- [x] 같은 양식을 ask/build/review 세 세션이 함께 쓸 수 있다. (§1 빈 골격 공용)
