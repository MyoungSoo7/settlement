# 2004-3-3 입력 — deterministic review gate

> 실습 절차: `practice-materials/chapter-04/2004-3-3-stop-and-requestion-criteria.md`
> 검토 대상 handoff: `handoff-L3-result.md`
> 근거 대조 파일: `meeting.md` (같은 워크스페이스)

## 입력표 (채움)

| 항목 | 내용 |
|---|---|
| 리프 ID | L3 |
| 리프 완료조건 | 6칸(제목·배경·문제·사용자·해결 방향·성공 기준)이 각각 한 줄 이상씩 채워진다 |
| handoff 결과 요약 | L3 build 세션이 회의록 확정 묶음을 6칸으로 맵핑 (`handoff-L3-result.md` §Result 6칸 표 + 근거 줄). 성공 기준 한 칸에 "측정 단위·기간 미정" 단서가 붙어 있다 |
| 근거로 확인할 파일·줄 | `meeting.md` — 배경 ← :5 / 문제 ← :12 / 해결 방향 ← :22, :23 / 범위 밖 ← :15, :27 / 성공 기준 ← :10 |
| 값으로 반드시 맞아야 할 항목 | 회의 일시 2026-05-22(:3) · 합의 범위는 두 갈래뿐(:22, :23) · "CS 문의 60%"는 한CS 추정이고 출처 미확인(:11, :34)이라 본문 6칸에 수치로 단정하면 안 됨 |
| ask-back이 필요한 모호함 | 성공 기준 "신청 직후 환불·일정 문의를 줄인다"가 목표 문장인지 측정 지표인지 — 측정 단위·기간이 정해져 있지 않다(:10 + handoff 남은 질문) |
| exit로 멈출 조건 | 환불 기준 명문화 가능 여부(운영팀 회신 대기 :35)·CS 문의 수치 출처(:34)처럼 외부 결정자 답이 있어야 풀리는 항목이 verdict를 막을 때, 또는 같은 rework 사유가 2회 반복될 때 |

## 그대로 붙일 프롬프트

위 입력표를 그대로 붙인 뒤, 한 줄을 덧붙인다.

> 이 입력으로 deterministic review gate를 만들어줘. 형식 -> 증거 -> false-negative 보정 순서로 검수하고, 결과는 pass / rework / ask-back / exit 중 하나로 라우팅해. oh-my-claudecode는 검증 gate 구현 사례로 말하지 말고, 팀 에이전트 흐름에서 gate 필요성을 읽는 read-only 관찰 근거로만 써. 설치·실행·성능 주장은 하지 마.

## 두 번 돌리기 (자가 검증)

같은 `handoff-L3-result.md`를 새 세션에 한 번 더 넣어 같은 verdict·route가 나오는지 본다 (완료 기준 마지막 항목).
