# Handoff — L3 · build 세션 → 메인

> 4-2 handoff로 돌아온 가상 결과 한 장. 2004-3-3 deterministic review gate의 입력.
> 양식 출처: `practice-materials/chapter-04/2004-2-3-workspace/handoff-L3-build-to-main.md` §1 빈 골격
> 근거(출처) 줄은 같은 워크스페이스의 `meeting.md` 줄번호를 가리킨다 — evidence stage가 이 줄을 직접 대조한다.

## From task (4-1 리프)
- 목적: 회의록 확정 내용을 제목·배경·문제·사용자·해결 방향·성공 기준 6칸으로 맵핑
- 입력: 회의록 확정 묶음(meeting.md) + L1 답(결정권자·일정·수치 출처·환불 기준) + L2 보존 인용 3줄
- 범위 (하지 않을 일 포함): 7칸 중 "확인 필요 항목"을 제외한 6칸만. 그 칸은 L4 build 몫이라 손대지 않는다
- 금지: 회의록에 없는 수치·일정·담당자·환불 기준을 결론처럼 쓰기. 단서 붙은 수치(CS 문의 60%)를 본문 6칸에 넣기
- 완료조건: 6칸이 각각 한 줄 이상씩 채워진다

## Result (이 세션이 채운 부분)
- 결과 요약:
  | 칸 | 채운 내용 |
  |---|---|
  | 제목 | 사내 교육 신청 페이지 개편 v1 |
  | 배경 | 분기 사내 교육 운영 중 신청 직후 환불·일정 변경 문의가 CS에 다수 유입해 회의 소집 |
  | 문제 | 일정·환불 안내가 페이지 맨 아래 작은 글씨라 사실상 안 읽혀 신청 후 CS 문의 발생 |
  | 사용자 | 사내 교육 신청자 + 운영/CS팀 |
  | 해결 방향 | 신청 직전 핵심 정보 강조 화면 + 자동 안내 메일에 환불·일정 기준 추가 (두 갈래) |
  | 성공 기준 | 신청 직후 환불·일정 문의를 줄인다 (← 측정 단위·기간 미정, 모호) |
- 근거(출처):
  - 배경       ← meeting.md:5
  - 문제       ← meeting.md:12
  - 해결 방향  ← meeting.md:22, meeting.md:23
  - 범위 밖(전체 리뉴얼 제외) ← meeting.md:15, meeting.md:27
  - 성공 기준  ← meeting.md:10
- 판정: 통과 (자체 검수 기준 — 6칸 모두 한 줄 이상)
- 남은 질문: 성공 기준을 측정 가능한 한 줄로 더 좁혀야 하나? (CS 문의 60% 수치는 출처 미확인 meeting.md:11·34이라 본문 단정 보류)
- 다음 세션의 첫 행동: review 세션(= 이 deterministic review gate)에 6칸 초안 + 근거(출처) + L3 완료조건을 함께 보낸다
