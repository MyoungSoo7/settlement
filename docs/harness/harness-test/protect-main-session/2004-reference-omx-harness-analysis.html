OMX HARNESS ANALYSIS POSITIONING

OmX를 무엇으로 볼 것인가

IS

런타임이 상태를 소유하는 사례

  • Codex CLI 위에 얹는 workflow layer이면서, dispatch·worker·authority를 event로 소유하는 Rust 코어가 있다.
  • delivered·integrated 같은 operation claim은 LLM이 아니라 런타임 상태가 판정한다.
  • goal 완료 claim을 hash로 frozen 상태에 묶는 append-only ledger가 있다.
IS NOT

복제할 도구도, 빈 껍데기도 아님

  • 특정 런타임을 정답으로 두지 않고, 설치·실행·성능을 주장하지 않는다.
  • "OmX엔 event store가 없다"는 틀렸다 — 두 고도에서 이미 event를 소유한다.
  • 다만 log를 replay해 claim을 재도출하는 일은 OmX 밖이다. 그건 5장 Ouroboros.
READ-ONLY LENS02 / 10
● SPEAKER NOTESSLIDE 02 / 10

정정 포인트를 분명히 한다. 예전 슬라이드는 event-store·replay를 5장 전용으로 미뤘는데, OmX는 operational 상태기계와 goal ledger 두 군데에서 이미 event를 소유한다. 5장이 더하는 건 "log를 source of truth로 두고 replay로 재도출"하는 부분이다.

OMX HARNESS ANALYSIS WHAT OMX ACTUALLY IS

OmX의 실제 구조 — 세 층

L1 · WORKFLOW

Node.js 패키지

npm i -g oh-my-codex + omx setup. role keyword·skills·$deep-interview → $ralplan → $ultragoal 흐름을 얹는다. CLI/JSON이 canonical control plane, MCP는 옵션.

L2 · RUNTIME

Rust crates

omx-runtime-core가 dispatch·authority·worker 상태를 event로 소유한다. omx-mux는 tmux 전송. 상태 권위는 여기 있다.

L3 · ENGINE

Codex CLI

실제 실행은 Codex가 한다 — OmX가 대체하지 않는다. 별도 prerequisite. OmX는 그 위에서 역할·상태·검수를 조직한다.

workflow는 Node, 상태 권위는 Rust, 실행 엔진은 Codex — 세 층이 나뉘어 있다.

THREE LAYERS03 / 10
● SPEAKER NOTESSLIDE 03 / 10

중요한 사실 정정 — OmX 본체는 Rust 앱이 아니라 Codex CLI 위의 Node 패키지다. 다만 상태 권위(dispatch·worker·authority)는 Rust 코어 omx-runtime-core가 event-sourcing으로 소유한다. "폴더를 베끼자"가 아니라 "어느 층이 무엇을 진실로 소유하나"를 본다.

OMX HARNESS ANALYSIS RUNTIME CORE

omx-runtime-core — command이 event가 되고, event가 진실이 된다

command은 상태기계가 합법 전이로 받아들일 때만 event가 된다. 그래서 결과를 지어낼 수 없다 — event가 있다는 건 전이가 실제로 일어났다는 뜻이다. snapshot은 event_log를 replay해 재구성하므로 같은 log면 같은 상태다.

claim-vs-log: delivered는 의견이 아니라 합법 전이가 남긴 event다.

EVENT-SOURCED RUNTIME04 / 10
● SPEAKER NOTESSLIDE 04 / 10

이 한 장이 정정의 핵심이다. RuntimeEngine::process(command)는 단일 변경 지점이고, 상태기계가 전이를 받아들여야만 event 하나가 event_log에 들어간다. load()는 event를 다시 replay해 동일 snapshot을 만든다(round-trip 테스트가 보장). 그래서 "delivered"는 LLM이 아니라 상태기계가 판정한다. 단 events.json은 통째 rewrite이고 compact()가 종결 event를 지우므로, 영구 불변 ledger는 아니다 — 그 부분이 5장이 더할 자리.

OMX HARNESS ANALYSIS VERIFY = EVIDENCE

검수는 LLM 판단이 아니라 evidence에 묶인다

PLANNER · ask

성공 기준을 먼저

outcome-first 계획 — success criteria, validation path, stop condition을 정한다.

EXECUTOR · build

증거를 남긴다

edit-test-verify 후 무엇이 바뀌었는지 + validation evidence를 보고한다. chat-only 상태는 인정하지 않는다.

VERIFIER · review

claim을 증거로 검사

claim → 가장 작은 검증 → 출력 확인 → 증거. 각 claim이 검증 command나 명시적 proof gap으로 추적된다.

quality-gate-g001: APPROVE + blockers 0 + npm run build · node --test 844/844 · 'integrated'는 git reachability(leader HEAD 도달)로 판정 — 의견 아님.

PLANNER · EXECUTOR · VERIFIER06 / 10
● SPEAKER NOTESSLIDE 06 / 10

OmX 검수는 산문 판단이 아니다. verifier는 claim을 명시하고 그것을 증명하는 최소 검증을 돌린 뒤 출력을 읽고 증거로 보고하며, 각 claim을 command나 proof gap으로 추적한다. team 통합 성공('integrated')은 git reachability라는 결정적 조건으로 판정한다. 얇은 LLM-기색 층은 리뷰의 APPROVE/CLEAR 권고 자체인데, 그것도 증거 인용과 빈 blocker 목록을 요구한다. ask/build/review 분업이 그대로 매핑된다.

OMX HARNESS ANALYSIS C3 CONTROL · STATUS

제어와 상태 표면 — 그리고 그 한계

HOOKS

native + fallback

Codex native hook이 있으면 쓰고, 없으면 tmux/runtime fallback. .omx/hooks/*.mjs 플러그인은 정규화된 envelope(source: native|derived)를 쓴다.

DELIVERY

전송은 확인된다

omx-mux가 pane tail을 잡아 DeliveryConfirmation(Confirmed / Unconfirmed)을 낸다. 키 입력이 닿았는지를 확인하지, 작업이 끝났는지는 아니다.

HUD STATE

가변 JSON의 projection

HUD는 session-scoped .omx/state/*.json을 읽는다. 상태마다 권위 owner가 하나, 나머지는 derivative evidence.

상태가 가변·세션 한정이라 split-brain을 막는 reconciliation 계획이 따로 있다 — 즉 불변 event log의 projection은 아니다.

STATUS → ROUTE + STATE07 / 10
● SPEAKER NOTESSLIDE 07 / 10

"status → route + state log" 줄을 구체화한다. keyword가 route하고, hook이 상태를 갱신하고, HUD가 그 상태를 보여준다. 단 그 상태는 가변 JSON이라, stale·root-fallback을 막는 reconciliation 계획이 존재한다 — 이게 OmX가 HUD 수준에서는 event-sourced가 아니라는 증거다. delivered/integrated 같은 전송·라우팅 claim은 결정적으로 검증되지만, "작업이 목표를 만족했나"는 이 층의 범위 밖이다.

OMX HARNESS ANALYSIS BRIDGE TO CHAPTER 05

OmX는 claim을 묶고, Ouroboros는 claim을 재도출한다

OMX · 4장

claim을 binding

  • operation claim(delivered·integrated)은 합법 전이·git reachability로 결정적 검증.
  • goal 완료 claim은 hash-bound ledger가 frozen 상태에 묶는다.
  • 한계: 테스트를 재실행하지 않고, log를 replay해 상태를 재도출하지 않는다.
OUROBOROS · 5장

claim을 re-derive

  • EventStore → EvidenceManifest → TraceGuard(rlm_forge) deliver gate.
  • AC claim이 admissible typed evidence를 인용했는지 구조적으로 검사 — self-report 대체.
  • log가 source of truth라, replay로 결과를 다시 도출한다 + capability graph.
BIND vs RE-DERIVE08 / 10
● SPEAKER NOTESSLIDE 08 / 10

정확한 연결 문장 — "append-only hash-bound ledger that binds claims(OmX, 4장) vs replayable deterministic event store that re-derives claims(Ouroboros, 5장)". 둘 다 'claim-vs-log, not claim-vs-judge'라는 같은 본능인데, OmX는 operation·binding 고도까지고, 5장이 그 규율을 goal/AC 고도로 끌어올려 log를 replay해 재도출한다.
