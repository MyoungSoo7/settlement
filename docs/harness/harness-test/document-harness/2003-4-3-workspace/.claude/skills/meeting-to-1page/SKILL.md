# Meeting → 1page Domain Skill

회의록 원문을 1page 기획안 초안으로 옮기는 도메인 작업 규칙. 본 도메인 스킬은 메타 스킬 `using-1page-harness/SKILL.md`의 협업 규약 아래에서 동작한다.

- 메타 스킬: 어떤 도메인이든 공통으로 따르는 협업 규약(hook 협업·재작성 루프·대상 파일 필터).
- 도메인 스킬 (이 파일): 회의록 → 1page 변환에서만 적용되는 보존 표현·출력 7칸 스키마·예시.

## 역할

- 너의 역할: 회의록 원문과 확인된 요구사항 안에서만 1page 기획안 초안을 7칸으로 구조화하는 보조자.
- 책임 범위: 제목·배경·문제·사용자·해결 방향·성공 기준·확인 필요 항목 7칸을 모두 채운다.
- 책임지지 않을 범위: 최종 일정·담당자 확정, 없는 사실 보강, 환불 기준 명문화 같은 외부 결정.

## 입력 명세

- 회의록 원문 (필수, 인용 블록 또는 첨부).
- 확인된 요구사항 (필수).
- 반드시 보존할 표현 n줄 (필수, 출력에서 큰따옴표 인용으로 유지).
- 비어 있으면 질문할 항목 — 결정권자·일정·수치 출처·외부 확인 사안 등.

## 도메인 제약

- 회의록에 없는 수치·일정·담당자·외부 사례·환불 기준은 결론처럼 쓰지 않는다.
- "정확한 출처는 다시 뽑아 와야 한다" 같은 단서가 붙은 수치(예: CS 문의 60%)는 본문 결론이 아니라 확인 필요 항목 칸으로만 옮긴다.
- 회의에서 범위 밖으로 합의된 항목(예: 페이지 전체 리뉴얼, 결제 모듈 모달 위치 변경, 환불 기준 자체 명문화)은 해결 방향 칸에 넣지 않는다.
- 미결정 항목을 "~로 한다 / ~할 예정이다" 같은 결론 톤으로 단정하지 않는다.

## 처리 순서

1. 회의록 원문 보존 — 인용 블록 또는 첨부로 그대로 둔다.
2. 확정된 내용과 결정되지 않은 내용을 두 묶음으로 분리.
3. 확정 내용을 제목·배경·문제·사용자·해결 방향·성공 기준 6칸으로 옮긴다.
4. 결정되지 않은 내용은 확인 필요 항목 한 칸으로 모은다.
5. 보존 표현 n줄이 큰따옴표 인용으로 살아 있는지 마지막에 한 번 더 확인.
6. 한 페이지 안에 7칸이 모두 들어가는지 확인하고 마감.

## 출력 형식 — 1page 7칸

```
[제목]              [배경]              [문제]
[사용자 또는 대상]  [해결 방향]         [성공 기준]
[확인 필요 항목]
```

각 칸 작성 기준:

| 칸 | 작성 기준 |
|---|---|
| 제목 | 한 줄로 압축한 목표. 보존 표현이 있으면 인용해 포함. |
| 배경 | 회의록의 인상·관찰만 3줄 이내. 미확인 수치는 본문에서 빼고 확인 필요 항목으로. |
| 문제 | 회의 인용을 그대로 옮긴다. "일정·환불 안내가 페이지 맨 아래 작은 글씨"처럼. |
| 사용자 또는 대상 | 회의에서 언급된 그룹만. 가공·확장 금지. |
| 해결 방향 | 확정된 요구사항만. 범위 밖 항목은 인용으로 부기. |
| 성공 기준 | 보존 표현(목표 한 줄)을 큰따옴표로 유지. |
| 확인 필요 항목 | 미결정 항목 전부. 결론 톤("~로 한다") 금지. |

## 도메인별 보존 표현 (사내 교육 신청 페이지 개편 예시)

- "신청 직후 환불·일정 문의를 줄인다"
- "페이지 전체 리뉴얼은 이번 범위가 아니다"
- "환불 기준 자체가 명문화돼 있지 않다"

본인 회의록에서 보존 표현을 새로 뽑을 때 기준:

- 의사결정자가 직접 발언한 목표 문장.
- 회의에서 범위를 좁힌 문장 ("이번 범위가 아니다" 같은 표현).
- 운영 제약을 명시한 문장 ("아직 ~이 없다" 같은 표현).

## 메타 스킬과의 협업

본 도메인 스킬은 `using-1page-harness` 메타 스킬 아래에서 동작한다.

- 출력 파일명에 `1page` / `one-page` / `2003-4-3` 중 하나를 포함 → PostToolUse hook이 자동 검사.
- hook이 exit 2를 반환하면 위반된 칸만 정확히 보정해 같은 파일을 다시 Edit.
- 세션 시작 시 SessionStart hook이 핵심 규약을 컨텍스트 상단에 주입.

## 예시 입력 / 출력

### 입력
2026-05-22 사내 교육 신청 페이지 개편 회의록 (`examples/sample-meeting.md`).

### 출력 (1page 7칸 채움)
- 제목: 사내 교육 신청 페이지 — "신청 직후 환불·일정 문의를 줄인다" v1
- 배경: 지난 분기 신청 후 환불·일정 변경 문의가 CS로 다수 유입. 일정·환불 안내는 페이지 맨 아래 작은 글씨로만 존재.
- 문제: "일정·환불 안내가 페이지 맨 아래 작은 글씨로 있어서 사실상 안 읽힌다" (김디자이너 발언 인용).
- 사용자 또는 대상: 사내 교육 신청자, CS팀.
- 해결 방향: (1) 신청 직전 핵심 정보 강조 화면, (2) 자동 안내 메일에 환불·일정 기준 추가. "페이지 전체 리뉴얼은 이번 범위가 아니다".
- 성공 기준: "신청 직후 환불·일정 문의를 줄인다".
- 확인 필요 항목: 최종 담당자·일정 (월요일 정기 미팅), CS 문의 60% 수치 출처, 환불 기준 명문화 여부 (운영팀 별도 확인), "환불 기준 자체가 명문화돼 있지 않다".

## 적용 범위 밖 (4장 분업형 / MCP형 후보)

- 확인 필요 항목 회차 추적 (월요일 미팅 → 운영팀 회신 → CS 수치 회신).
- 보존 표현 누락 시 자동 재시도 횟수 제어 (hook은 신호만 줌).
- 시안·메일 템플릿 작업 일정의 외부 회신 단계 순서 강제.
- 입력별 회귀 케이스.

## 시운전 메모

- 첫 시운전 입력: 2026-05-22 사내 교육 신청 페이지 개편 회의록.
- 첫 시운전 결과: 7칸 모두 채움. PostToolUse hook 1회 통과.
- 두 번째 시운전: 같은 회의록 재투입 → 같은 칸 구성 (deterministic 한 줄 시험 통과).
- v2에서 손볼 항목: 시안·메일 일정 외부 회신을 어떻게 추적할지.
