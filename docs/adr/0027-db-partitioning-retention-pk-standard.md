# ADR 0027 — DB 파티셔닝·리텐션·PK 전략 표준

- 상태: Accepted
- 일자: 2026-07-15
- 관련: ADR 0020(이벤트 드리븐 프로젝션), ADR 0003(Outbox), ADR 0007(원장 불변식), account-domain-rules · ledger-invariants 스킬
- 배경: 크로스컷 DB 설계 리뷰(R3)에서 "파티션 유지보수 함수는 전 서비스에 있으나 호출 배선이 없어 런웨이(2028-12) 소진 후 DEFAULT 파티션 유입 트랩" 지적. 12개 DB-per-service 에 흩어진 파티셔닝·리텐션·PK 관례를 하나의 표준으로 고정한다.

## 컨텍스트

각 서비스가 자체 DB 를 소유하는 DB-per-service(ADR 0020) 구조라, 시계열·append-only 테이블의 파티셔닝·리텐션·PK
관례가 서비스별로 제각각 진화했다. 세 축의 결정을 표준으로 묶지 않으면 (1) 파티션 런웨이 소진, (2) 멱등 방어선
조기 삭제로 인한 이중 처리, (3) 파티션드 테이블 PK 제약 위반이 서로 다른 서비스에서 반복 재발한다.

## 결정

### 1. 파티션 대상·키·런웨이

시간축으로 무한 append 되고 시간 기준 조회·리텐션이 지배적인 테이블만 **RANGE 파티션**한다.

| 테이블 | 서비스 | 파티션 키 | 주기 | ensure 함수 |
|--------|--------|-----------|------|-------------|
| `audit_logs` | order·settlement·loan·investment·account·operation·company·ai·organization | `created_at` | 월별 | `ensure_audit_log_partition(months_ahead int)` |
| `ops_metric_bucket` | operation | `bucket_start` | 월별 | `ensure_ops_metric_bucket_partition(months_ahead int)` |
| `chat_messages` | ai | `created_at` | 월별 | `ensure_chat_message_partition(months_ahead int)` |
| `stock_quotes` | market | 거래일 | **연별** | `ensure_stock_quote_partition(years_ahead int)` |
| `indicator_values` | economics | 기준일 | **연별** | `ensure_indicator_value_partition(years_ahead int)` |

- **DEFAULT 파티션 트랩**: 모든 파티션드 부모에 DEFAULT 파티션을 두어 범위 밖 삽입 실패를 막되, DEFAULT 에 데이터가
  쌓이면 프루닝 이점이 사라지고 리텐션(DETACH+DROP)이 무력화된다. 마이그레이션은 유한 구간(월별=2028-12,
  연별=2028)까지만 선생성한 **일회성 런웨이**이므로, 그 이후는 반드시 ensure 함수를 주기 호출해 굴려야 한다.
- 특정 월/년 데이터가 이미 DEFAULT 에 들어간 뒤에는 그 구간 파티션을 `CREATE ... PARTITION OF` 로 만들 수 없다
  (범위 행 충돌). 즉 ensure 는 **데이터가 도착하기 전에** 미래 파티션을 확보해야 한다.

### 2. 비파티션 결정

멱등·원장 무결성 테이블은 **전역 UNIQUE 가 파티션 키보다 우선**하므로 파티션하지 않는다. PostgreSQL 파티션드
테이블은 UNIQUE/PK 에 파티션 키를 포함해야 해서, 전역 유일성이 깨지기 때문이다.

- `outbox_events`(`event_id UUID UNIQUE`), `processed_events`(`(consumer_group, event_id)` PK) — 멱등 3단 방어선.
- `ledger_entries` · `account_entries` — 복식부기 원장. 전표는 전역 참조·역분개 대상이라 시간 파티션과 상충.

이들은 파티션 대신 **리텐션 함수**로만 관리한다(아래 4).

### 3. 유지보수 자동화 (스케줄러 + 부팅 러너)

ensure 함수는 존재만으로는 런웨이를 굴리지 못한다 — 호출 배선이 이 ADR 의 핵심이다.

- **order·settlement**: `PartitionMaintenanceScheduler`(`{service}.config`) — `@EventListener(ApplicationReadyEvent)`
  로 부팅 시 1회 + `@Scheduled(월 1회)` + ShedLock `@SchedulerLock` 으로 replicas 중 1개만 롤.
- **나머지 9개 서비스**(loan·investment·account·operation·company·ai·market·economics·organization):
  `PartitionMaintenanceRunner implements ApplicationRunner` — 부팅 시 1회 ensure(재배포·재기동 주기로 런웨이 갱신).
- **fail-open 원칙**: 테스트 DB(create-drop)엔 유지보수 함수가 없어 호출이 실패한다. 유지보수는 보조 작업이므로
  모든 호출을 try-catch(warn 로그)로 감싸 **부팅·스케줄이 절대 실패로 이어지지 않게** 한다.
- **prune(파기)은 자동 호출하지 않는다**: 금융 감사·원장·멱등 데이터 삭제는 규제·재전송 창을 고려한 운영 판단이라
  자동화 대상이 아니다. 자동화는 선생성(ensure)만 담당한다.
- **네이티브 SQL 스키마 한정**: JdbcTemplate 은 `hibernate.default_schema` 를 무시하므로, 함수 호출을
  `default_schema`(opslab/public) 로 명시 한정한다(SchedulingLockConfig 와 동일 관례).

### 4. 리텐션 함수 시그니처 규약

| 유형 | 시그니처 | 반환 | 하한 가드 |
|------|----------|------|-----------|
| 이벤트 큐(outbox) | `prune_outbox_published(p_retention INTERVAL DEFAULT '7 days')` | `BIGINT`(삭제 건수) | ≥7일(재전송 창) |
| 멱등 추적 | `prune_processed_events(p_retention INTERVAL DEFAULT '30 days')` | `BIGINT` | **≥7일**(조기 삭제 시 이중 처리) |
| 파티션드 감사 | `prune_audit_logs(retain_months int)` | `int`(DROP 파티션 수) | ≥1개월, DEFAULT 파티션 보호 |
| 파티션드 시계열 | `prune_{stock_quotes,indicator_values}(retain_years int)` 등 | `int` | 파티션 DETACH+DROP |

- INTERVAL 계열은 **행 DELETE**, 파티션 계열은 **DETACH+DROP**(메타데이터 연산 O(1)). audit 는 월 단위, 시세·지표는
  연 단위, 큐·멱등은 INTERVAL.
- prune 계열은 하한 가드로 성급한 삭제를 DB 레벨에서 차단한다(예: `processed_events` ≥7일 미만 요청 시 EXCEPTION).

### 5. PK 표준

- 신규 테이블 PK 는 `bigint GENERATED BY DEFAULT AS IDENTITY` (BIGSERIAL 대체, `INSERT ... RETURNING id` 보존,
  `@GeneratedValue(IDENTITY)` 호환).
- **파티션드 테이블은 PK 에 파티션 키를 포함**해야 하므로 `(id, {partition_key})` 복합 PK 로 둔다(예: audit_logs
  `PRIMARY KEY (id, created_at)`). id 시퀀스 연속성·전역 유일성은 그대로 유지된다.

### 6. PII 암호화 스킴 (enc:v1)

민감 PII(지급계좌·예금주명·챗 본문 등)는 앱단 **AES-GCM(256)** 으로 암호화하고, 저장 형식은
`enc:v1:` || Base64(IV(12B) || ciphertext+tag(128bit)) 로 고정한다.

- 스킴 버전 접두(`enc:v1:`)로 미래 키 로테이션·알고리즘 교체(v2)를 식별 가능하게 한다.
- **레거시 평문 lazy migration**: 복호화 시 `enc:v1:` 접두가 없으면 도입 이전 평문으로 간주해 그대로 반환하고,
  다음 저장 시점(또는 append-only 는 신규 행부터) 암호문으로 전환한다.

## 결과

- **긍정**: 런웨이 소진 트랩이 배선으로 닫힌다(재기동·월배치가 파티션을 계속 굴림). 파티셔닝/비파티셔닝·리텐션·PK
  결정이 단일 출처로 고정돼 신규 시계열 테이블 추가 시 판단 기준이 명확하다.
- **부정/한계**: 부팅 러너 방식(9개 서비스)은 배포 주기가 런웨이보다 길어지면 이론상 소진 가능 —
  `app.partition.{months,years}-ahead`(기본 월 3 / 연 1)를 넉넉히 잡아 완충한다. 진짜 장기 무중단 운영에서는
  이들도 스케줄러로 승격하는 것이 다음 단계다.
- **미적용(후속)**: financial-statements·common-data 서비스도 `ensure_audit_log_partition` 을 보유하나 이번 배선
  대상에서 제외됐다 — 동일 패턴으로 러너 추가 필요(별도 트랩).
```
설정 키: app.partition.months-ahead(기본 3) · app.partition.years-ahead(기본 1) · app.partition.ensure-cron(order/settlement)
```
