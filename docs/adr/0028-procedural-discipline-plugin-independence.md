# ADR 0028 — 절차 규율층 플러그인 독립 내재화와 이중 라우팅 경계

- 상태: Accepted (구현 완료)
- 일자: 2026-07-22

## 컨텍스트

이 저장소 하네스는 "플러그인 독립 이중 경로" 원칙(`HARNESS.md`)을 가지고 있다 — 실시간
PreToolUse 가드(`guard.mjs`)·git pre-commit·CI(`harness-guard.yml`) 3중으로 **결정적 불변식**은
플러그인 유무와 무관하게 강제된다. 그런데 범용 **절차 규율**(TDD 착수 순서, 버그 조사 시 원인
규명 우선, 완료 판정 시 게이트 확인)은 이 원칙의 사각지대에 있었다 — 외부 플러그인
superpowers(전역 설치, SessionStart 주입)와 OMC 전역 지침(`~/.claude/CLAUDE.md`)에 전적으로
위임되어 있었다. 플러그인이 제거되거나 미설치된 환경(CI, 새 클론, Codex 등 타 에이전트 런타임)
에서는 절차 규율층이 통째로 사라지는 숨은 의존이었다.

추가로 두 가지 사실이 이 의존을 실증했다:

1. superpowers 는 SessionStart 주입 + 스킬 강제 문구로 동작하는 **확률적(문서 규율) 하네스**다.
   실측 텔레메트리 상 이 저장소에서 superpowers 스킬 로드가 **0회**였다 — 전역 설치가 곧 실제
   적용을 보장하지 않았다.
2. 하네스 런타임 상태·로그가 `.omc/`(OMC 플러그인 소유, 플러그인 제거 시 정리 대상)에 있어 같은
   "플러그인 소유 디렉토리 의존" 문제가 규율층 밖에서도 반복되고 있었다.

## 결정

절차 규율 3종을 저장소 스킬로 **내재화**하고, 하네스 런타임을 플러그인 비소유 경로로 이전하며,
OMC 전역 워크플로와의 이중 라우팅 경계를 명문화한다.

### 1. 절차 규율 3종의 저장소 스킬화

`.claude/skills/{tdd-discipline, debugging-discipline, verify-before-done}` 을 신설:

| 스킬 | 강제하는 절차 |
|---|---|
| `tdd-discipline` | 구현 코드 작성 **전** 실패 테스트를 먼저 목격 → 최소 구현. 완료 판정은 게이트(JaCoCo LINE 90%)가 정답 |
| `debugging-discipline` | 재현 → 증거 수집 → 가설 검증 순서. 원인 규명 전 수정 금지, 가설이 3연속 기각되면 중단하고 재평가 |
| `verify-before-done` | DoD 게이트를 실제로 실행하고 증거를 병기 — 완료는 주장이 아니라 게이트 출력. 자기 승인 금지 |

`CLAUDE.md` 작업 프로토콜에 "외부 플러그인 스킬 위임 금지" 를 명문화해, 이 3종이 정본이고
동등 기능의 외부 플러그인 스킬(superpowers 등)에 위임하지 않음을 고정했다. (커밋 `ef21852e3`)

### 2. 권장의 기계화 확장 — 라우터 트리거

`scripts/harness/skill-router.mjs` 가 세션 중 **첫 소스 편집**(`/src/(main|test)/` 경로 매치, 42행)
에 `tdd-discipline` 로드 리마인더를 1회 주입한다 — 라우트 최후순위(도메인 규칙 라우팅이 항상
우선, cap 3 내). superpowers 방식(확률적 자기판단·SessionStart 상시 주입)과 달리, **경로 매칭
기반 결정적 트리거**로 "제안이 실제로 표시됐는가" 를 텔레메트리로 관측 가능하게 만든다.

### 3. 하네스 런타임 이전 — `.omc` → `.claude/harness`

하네스 로그·상태를 `.claude/harness/{logs,state}` 로 이전해, OMC 플러그인 소유 디렉토리
(`.omc/`, 플러그인 제거 시 정리 대상)에 대한 의존을 규율층뿐 아니라 런타임 계층에서도 제거했다.
(커밋 `3f0b8d0af`)

### 4. 이중 라우팅 경계 명문화 (`HARNESS.md`)

OMC keyword-detector 의 워크플로 모드(ralph·autopilot·ulw·team 등)는 **OMC 소유**, 도메인 규칙과
절차 규율 스킬은 **본 하네스가 정본** — 이라는 소유권 분리를 `HARNESS.md` 에 명문화했다. 지침
우선순위는 프로젝트 `CLAUDE.md`/`HARNESS.md` > 플러그인 주입 순으로 고정한다. (커밋 `c1cc5dd4c`)

### 5. 효과 측정 체계

측정 계층 L1(스킬 호출 횟수)~L4(인과 귀속: 스킬 로드가 실제 결함 감소·재작업 감소로 이어졌는가)
를 정의하고, superpowers on/off day-parity A/B 준실험을 2026-07-22 부터 시작한다. 정본은
`docs/superpowers-harness.md`. (커밋 `3f3e09a94`)

## 검토한 대안

| 옵션 | 채택? | 이유 |
|---|---|---|
| **A. superpowers 위임 유지** | ✗ | 세션 주입이 공짜로 오지만, 플러그인 제거·미설치 환경(CI·새 클론·Codex)에서 규율이 통째로 소실된다. 실증: superpowers v5→v6 업데이트에서 스킬 20종·에이전트가 저장소 통제 밖에서 일방적으로 제거된 사례가 있었다 — 외부 통제 리스크가 규율의 정본성과 충돌 |
| **B. 절차 규율을 `guard.mjs` 기계 차단으로 강제** | ✗ | 절차(사고 순서: 재현→증거→가설, 실패 테스트 먼저 등)는 툴 호출 차단으로 표현할 수 없다 — 결정적인 것(setter 금지, MSA 경계 등)만 게이트로 강제하고, 판단·절차는 스킬이라는 기존 원칙(HARNESS.md)을 유지 |
| **C. 전 스킬을 SessionStart 훅으로 상시 주입 (superpowers 방식 복제)** | ✗ | 토큰 고정비용 대비 효과가 확률적이고, `CLAUDE.md` 네이티브 로드가 이미 스킬 디스패처 역할을 하고 있어 중복. 온디맨드 로드 + 라우터 리마인더(결정 2) 채택 |
| **D. 절차 규율 3종 저장소 스킬화 + 라우터 트리거 + 런타임 이전 + 경계 명문화 + 측정 체계 (본 결정)** | ✓ | 플러그인 독립 이중 경로 원칙을 규율층까지 확장하면서, 결정적 게이트와 확률적 절차 규율의 역할 분리를 그대로 유지 |

## 결과

### 좋아지는 점

- superpowers/OMC 미설치 환경에서도 절차 규율이 동작한다 — "플러그인 독립 이중 경로" 원칙이
  가드(결정적 강제)뿐 아니라 규율층(절차 권고)까지 확장됐다.
- 라우터 텔레메트리로 "스킬 제안 대비 실제 로드율" 이 측정 가능해졌다 — 종전에는 문서로만
  존재하던 규율 준수가 관측 가능한 지표로 전환됐다.
- 외부 플러그인과 기능이 정면 중복되는 3건(TDD·디버깅·완료검증)이 발생하지만, 지침 우선순위
  조항(결정 4)으로 "어느 쪽이 정본인가" 의 모호성을 해소했다.

### 트레이드오프 / 리스크

- 스킬 3종(`tdd-discipline`, `debugging-discipline`, `verify-before-done`)의 유지보수 책임이
  저장소로 이동했다 — 외부 플러그인 업데이트를 그냥 받아쓸 수 없다.
- 절차 준수 자체는 여전히 **확률적**(주입·리마인더 기반)이다 — 기계적 차단이 아니며, LLM 이
  리마인더를 무시할 가능성은 남는다. 이 한계는 대안 B 검토에서 의도적으로 감수한 것이며,
  L1~L4 측정 체계와 A/B 준실험(결정 5)으로 관리한다.
