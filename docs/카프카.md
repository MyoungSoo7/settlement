# 1부. Kafka 소개

## 1-1. Kafka 란

**Apache Kafka** 는 분산 **이벤트 스트리밍 플랫폼**. 메시지를 디스크에 영속 저장하는 분산 커밋 로그(commit log)로, 높은 처리량·내구성·수평 확장을 제공한다.

핵심 개념:

| 개념 | 설명 |
|------|------|
| **Topic** | 메시지의 논리적 분류(스트림). 예: `lemuel.payment.captured` |
| **Partition** | 토픽을 나눈 단위. 병렬 소비·순서 보장의 기본 단위. 파티션 **내부**에서만 순서 보장 |
| **Offset** | 파티션 내 메시지의 순번. 컨슈머가 어디까지 읽었는지 추적 |
| **Producer** | 메시지 발행. 키(key) 해시로 파티션 결정 |
| **Consumer / Consumer Group** | 메시지 소비. 그룹 내 컨슈머들이 파티션을 나눠 가짐(병렬). 같은 그룹은 메시지를 한 번만 처리 |
| **Broker** | Kafka 서버 노드. 여러 브로커가 클러스터 구성 |
| **Replication** | 파티션 복제본(replica)으로 내구성 확보 |

## 1-2. 왜 쓰는가

- **비동기 디커플링**: 생산자/소비자가 서로의 가용성에 묶이지 않음(시간적 디커플링).
- **버퍼링/백프레셔**: 소비자가 느려도 로그에 쌓아두고 따라잡음.
- **재생(replay)**: offset 을 되돌려 과거 메시지 재처리 가능(retention 기간 내).
- **확장성**: 파티션 늘려 컨슈머 병렬도 확장.

## 1-3. 전달 보장 (Delivery Semantics)

- **at-most-once**: 유실 가능, 중복 없음
- **at-least-once**: 유실 없음, **중복 가능** → 컨슈머 멱등성 필요 (가장 흔함)
- **exactly-once**: 트랜잭션/멱등 프로듀서로 근접 달성

> 실무 정석: **at-least-once + 컨슈머 멱등 처리**. 이 프로젝트도 이 전략.

## 1-4. 흔한 문제와 해법

- **순서**: 파티션 내에서만 보장 → 순서가 중요한 이벤트는 같은 key 로 같은 파티션에.
- **이중 쓰기(dual write)**: "DB 커밋 + 메시지 발행"이 원자적이지 않은 문제 → **Transactional Outbox** 패턴.
- **독성 메시지(poison pill)**: 영원히 실패하는 메시지가 파티션을 막음 → **DLT(Dead Letter Topic)** 격리.

# 2부. 이 프로젝트의 Kafka 적용 분석

## 2-1. 역할 — 서비스 간 비동기 백본

이 프로젝트에서 Kafka 는 **order-service ↔ settlement-service 의 유일한 동적 연결**이다(동기 HTTP 호출 0). 거래 컨텍스트의 상태 변화를 정산 컨텍스트로 비동기 전파한다.

- 브로커: **Redpanda**(Kafka 호환, `docker-compose.yml`). 내부 `redpanda:29092` / 외부 `localhost:9092`.
- 스위치: `app.kafka.enabled` — 로컬 기본 OFF, 프로덕션 compose/K8s 에서 ON(`@ConditionalOnProperty`).

```mermaid
flowchart LR
    P["order-service<br/>Payment.capture()"] -->|outbox INSERT (1 tx)| OB[("outbox_events")]
    OB -->|폴러 claim (SKIP LOCKED)| PUB["KafkaOutboxPublisher"]
    PUB -->|publish key=payment_id| T["topic: lemuel.payment.captured<br/>(3 partitions)"]
    T -->|consume group=lemuel-settlement| C["PaymentEventKafkaConsumer"]
    C -->|멱등 체크 + 정산 생성| S[("settlements")]
    C -.재시도 끝 실패.-> DLT["lemuel.payment.captured.DLT"]
```

## 2-2. 토픽 설계 — `KafkaConfig`

| 토픽 | 파티션 | retention | 비고 |
|------|--------|-----------|------|
| `lemuel.payment.captured` | 3 | 7일 | 결제완료 → 정산 생성 트리거 |
| `lemuel.payment.refunded` | 3 | 7일 | 환불 → 역정산 |
| `lemuel.payment.captured.DLT` | 3 | **30일** | 처리 실패 격리(원본보다 길게 보관) |
| `lemuel.payment.refunded.DLT` | 3 | 30일 | |

- **파티션 = 컨슈머 병렬 소비 상한**(동시성 ≤ 파티션 수). 처리량 증대 시 파티션과 컨슈머 concurrency 를 함께 상향(파티션은 줄일 수 없어 늘리는 방향만).
- **키 = `payment_id`** → 같은 결제의 이벤트는 같은 파티션 → **순서 보장**, 다른 결제는 분산 → 병렬.
- DLT 는 원본과 같은 파티션 수(recoverer 가 원본 partition 번호로 라우팅).

## 2-3. 프로듀서 — Transactional Outbox 경유

**핵심**: 도메인은 Kafka 로 직접 쏘지 않는다. **DB 커밋과 발행의 원자성**을 위해 Outbox 를 경유.

```
Payment.capture() → payments UPDATE + outbox_events INSERT (같은 트랜잭션)
   → 폴러가 FOR UPDATE SKIP LOCKED 로 claim → KafkaOutboxPublisher.publish
```

프로듀서 설정(`application.yml`)과 `KafkaOutboxPublisher`:
- `acks: all` + `enable.idempotence: true` — 모든 ISR 확인 + **프로듀서 중복 발행 방지**.
- `retries: 5`, `max.in.flight: 5`, `compression: lz4`.
- **토픽 명명 규칙**: `aggregateType` + `eventType` → `lemuel.<aggregate>.<event_snake>` (예: Payment+PaymentCaptured → `lemuel.payment.captured`).
- **헤더 부여**: `event_id`(멱등), `event_type`, `aggregate_type`, **`traceparent`**(W3C trace context → 컨슈머가 같은 분산 trace 합류).
- **동기 `publish`**(폴러 트랜잭션 내 즉시 실패 감지 → markFailed) + **비동기 `publishAsync`**(배치 폴러가 future 모아 대기 → 라운드트립 병렬화, 처리량↑).

## 2-4. 컨슈머 — `PaymentEventKafkaConsumer`

```java
@KafkaListener(topics="${app.kafka.topic.payment-captured}", groupId="lemuel-settlement",
               containerFactory="kafkaListenerContainerFactory")
@Transactional
public void onPaymentCaptured(ConsumerRecord, Acknowledgment ack)
```

- **수동 커밋**(`enable-auto-commit: false`, `MANUAL_IMMEDIATE`): 멱등 처리 + 정산 생성이 끝난 뒤에만 `ack.acknowledge()`.
- `isolation.level: read_committed`: 커밋된 메시지만 소비.
- `auto-offset-reset: earliest`: 새 컨슈머는 처음부터 재생.
- **병렬 소비**: `concurrency`(기본 3)개 스레드가 파티션을 disjoint 분할해 정산을 병렬 생성.

### at-least-once + 멱등 3단 방어
Kafka 는 재전달로 **중복 소비가 가능**하므로 멱등이 필수. 실제 돈(정산)이라 더 엄격:
1. `outbox_events.event_id` UUID UNIQUE — 프로듀서 중복 발행 방지
2. `processed_events (consumer_group, event_id)` PK — 컨슈머 재수신 스킵
3. `settlements.payment_id` UNIQUE — 스키마 최종 방어

→ 파티션 간 병렬 처리를 해도 중복 정산이 생기지 않음.

## 2-5. 장애 처리 — DLT + 재시도 (`KafkaErrorHandlerConfig`)

Spring Kafka 기본(`FixedBackOff(0,9)` 후 조용히 skip → 사실상 유실)을 **DLT 격리**로 교체.

| 예외 유형 | 처리 |
|-----------|------|
| 일시적(DB 락, IO) | `FixedBackOff(2s × 3회)` 재시도 후 실패 시 DLT |
| 독성(JSON 파싱·`IllegalArgumentException`·`IllegalStateException`) | **재시도 없이 즉시 DLT**(`addNotRetryableExceptions`) — 같은 파티션 후속 메시지 stall 방지 |

- **DeadLetterPublishingRecoverer**: 재시도 끝나면 원본을 `<topic>.DLT` 로 복사. `kafka_dlt-*` 헤더(원본 토픽/파티션/오프셋/예외 FQCN/스택트레이스) 자동 부여 + 원본 `event_id`/`traceparent` 패스스루 → 사후 추적·**멱등 replay** 가능.
- **DLT 프로듀서**: `acks=all` + `idempotence` + `retries=5` (DLT 손실 방지).
- **메트릭**: `settlement.kafka.dlt.published`(DLT 라우팅), `settlement.kafka.retry`(재시도) → Prometheus 알림 연계([프로메테우스.md](프로메테우스.md)).

## 2-6. DLT 재처리 — `DlqReplayService` / `DlqAdminController`

- 운영자가 DLT 메시지를 점검·재처리: `GET /admin/dlq/inspect` → 페이로드·예외 확인 → `POST /admin/dlq/replay` 로 원본 토픽 재발행.
- **replay 폭주 방지**: `x-replay-count` 헤더로 재처리 횟수 cap(무한루프 차단). 메트릭 `settlement.kafka.dlt.replayed`.
- 흐름: 스키마 drift/버그로 DLT 적재 → 패치/핫픽스 → replay (ADR 0017).

## 2-7. Outbox 와의 관계 (정리)

이 프로젝트의 "Kafka 사용"은 곧 **Outbox + Kafka 조합**이다.

| 구간 | 메커니즘 | 보장 |
|------|----------|------|
| 도메인 → outbox | 같은 DB 트랜잭션 INSERT | 원자성(이중 쓰기 해결) |
| outbox → Kafka | 멀티워커 폴러(`FOR UPDATE SKIP LOCKED`, lease) | at-least-once, FIFO, 중복발행 방지 |
| Kafka → 컨슈머 | 수동 ack + 멱등 3단 | at-least-once + exactly-once 효과 |
| 실패 격리 | DLT + 재시도 + replay | 유실 방지, 운영 복구 |

- 별도로 **Ledger Outbox**(`ledger_outbox`)·**ES 색인 큐**(`settlement_index_queue`)도 같은 DB-backed 큐 패턴(이쪽은 Kafka 가 아닌 로컬 폴러/이벤트).

## 정리

- **위치**: order ↔ settlement 비동기 백본(동기 호출 0). 브로커는 Redpanda(Kafka 호환).
- **토픽**: `lemuel.payment.captured/refunded`(3파티션, 7일) + `.DLT`(30일). 키=`payment_id` 로 순서 보장 + 병렬.
- **프로듀서**: 직접 발행 X → **Transactional Outbox** 경유, `acks=all`+idempotence, traceparent 헤더 전파.
- **컨슈머**: 수동 ack + `read_committed` + concurrency 병렬, **멱등 3단 방어**로 at-least-once 안전 처리.
- **장애**: 재시도(2s×3) + 독성 메시지 즉시 **DLT** 격리 + 운영자 **replay**(cap), 전 과정 메트릭화.
