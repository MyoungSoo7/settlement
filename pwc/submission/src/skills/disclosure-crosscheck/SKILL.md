# 공시 대사 — 내부 장부와 외부 공시는 같은 회사를 말하고 있는가

내부 장부는 회사의 주장이고, DART 공시는 외부에 서명해 제출한 기록이다.
둘을 대사해 어긋나는 지점을 찾고, 공시 이력 자체의 행간(정정 반복, 의견 변형,
자금 조달 공시)에서 리스크 신호를 읽는다.

> **공시 데이터는 읽기 전용 분석 입력이다.** 조회만 하고 수정·제출 행위는 하지 않는다.

## Step 1: 대상 확정

- `dart_corp_search` 로 기업명/종목코드에서 `corp_code` 를 확정한다. 동명 기업이 여러 개면
  `dart_company`(기업개황)로 업종·소재지를 대조해 특정한다.
- 키가 없으면 `dart_status` 로 상태를 확인하고, 라이브 대사가 불가함을 산출물에 명시한다.

## Step 2: 공시 수치 확보

- `dart_financial_summary` 로 주요 계정(매출, 영업이익, 자산, 부채)의 당기/전기/전전기를 확보한다.
- 세부 계정(매출채권, 차입금 등)이 필요하면 `dart_financial_full` 로 내려간다.
- **기준 정렬을 먼저 한다**: 연결(CFS)/별도(OFS) 여부, 회계연도 구간, 단위(원/천원/백만원)를
  내부 장부와 맞춘 뒤에만 비교한다. 기준이 다른 비교는 break 가 아니라 노이즈다.

## Step 3: 내부 장부 대사

`accounting-anomaly` 의 break 버킷 규칙을 재사용한다 — 키는 `계정 × 기간`:

| 버킷 | 공시 대사에서의 의미 |
|---|---|
| 금액 break | 내부 장부와 공시 수치가 유의미하게 상이 — 집계 범위·기준 차이부터 확인 |
| 추세 break | 내부는 성장인데 공시는 정체(또는 반대) — 인식 시점·범위 차이 가설 |
| 한쪽만 존재 | 내부에만 있는 계정(부외 가능성) 또는 공시에만 있는 계정(누락 가능성) |

허용오차는 단위 반올림을 감안해 계정 잔액의 ±1% 를 기본으로 하되, 초과분은 전부 기록한다.

## Step 4: 공시 이력의 행간 (`dart_disclosures`)

수치가 아니라 **공시 행위 자체**가 신호가 되는 패턴:

| 패턴 | 의심 신호 |
|---|---|
| 같은 보고서의 정정공시 반복 | 내부 결산 통제 취약 — 무엇이 몇 번 정정됐는지 목록화 |
| 감사보고서 제출 지연·의견 변형(한정/부적정/의견거절) | 최우선 보고 — 단독으로도 CEO 브리핑 대상 |
| 유상증자·CB/BW 발행 공시 빈발 | 영업현금으로 부족한 자금을 외부 조달 — cashflow-bottleneck/macro-exposure 발견과 교차 검증 |
| 최대주주 변경, 임원 대량 변경 | 지배구조 이벤트 — 시점이 실적 악화와 겹치는지 확인 |

## Step 5: 산출물

1. **대사 리포트** — 계정별: 내부 수치, 공시 수치(출처: 접수번호·보고서명), 버킷, 원인 가설,
   판별 테스트 (예: "연결 범위 차이 확인 — 종속회사 목록 대조").
2. **공시 행간 리스트** — 신호별: 공시 일자·제목, 패턴 분류, 왜 신호인지 한 문장.

단정하지 않는다 — "공시와 내부 장부가 다르다" 는 오류가 아니라 **기준 차이라는 가설과
분식이라는 가설이 모두 열려 있는 상태**이며, 판별 테스트를 통과하기 전까지 둘 다 가설로 둔다.
중요 발견은 `ceo-risk-recon` 의 교차 검증으로, 최종 보고는 `ceo-briefing` 으로 넘긴다.
