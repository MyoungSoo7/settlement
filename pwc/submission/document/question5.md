## Q1. 무엇을, 누가, 어떤 상황에서 쓰나요?

**무엇을** — 시산표·채권 aging·원가배분표처럼 **따로 보면 정상인 회계 데이터를 서로 교차 대조**해서, CEO가 아직 질문하지 않은 리스크(수익 조기 인식, 특정 거래처 신용 집중, 원가 배분 왜곡으로 인한 보이지 않는 손실, 차입 의존 성장·금리 노출)를 **역으로 추출**하고, "왜 문제인지"의 인과 사슬을 **서명용 브리핑(자연어)** 으로 전달하는 에이전트입니다. 대시보드가 아니라 "무엇을 먼저 확인해야 하는가"를 제기하는 도구입니다.

**누가 · 어떤 상황에서**

- **사용자는 기업 CEO 한 사람입니다.** 분기 마감 데이터에서 놓친 리스크를 결론·근거·확인절차·권고조치가 분리된 **서명용 브리핑**으로 직접 받습니다. 실무 워킹페이퍼나 컨설턴트용 중간 산출물은 만들지 않으며, 판별 테스트·권고 조치도 CEO가 담당 부서에 그대로 지시할 수 있는 단위로만 표현됩니다.
- (삼일PwC 딜리버리 관점에서 이 플러그인은 컨설턴트가 쓰는 내부 도구가 아니라, **고객사 CEO에게 배포되는 최종 딜리버러블**입니다 — 어떤 회사 데이터 폴더를 지정해도 같은 절차로 도는 재사용 가능한 진단 프레임워크이기 때문에 고객사별 재개발 없이 배포됩니다.)

**전형적 상황** — 분기·월 마감 직후, 손익계산서·재무상태표·현금흐름표가 각각은 "좋아 보이는" 데이터로 들어왔을 때. 사람이 보고서 3개를 눈으로 교차 대조하기 전에 에이전트가 먼저 행간의 이상 신호를 제기합니다.

## Q2. 왜 이 문제를 선택했나요?

세 가지 근거가 맞물립니다.

1. **대시보드로는 안 되는 '추론' 영역을 정조준했습니다.** 과제가 명시적으로 요구한 것은 "합성 데이터를 이쁘게 보여주는 대시보드"(AI가 아니어도 가능)가 아니라 **AI의 추론 능력으로 데이터의 행간에 숨은 문제를 포착하고 논리적 근거를 자연어로 제시**하는 것이었습니다. 이 문제는 그 요구를 정면으로 겨냥합니다 — 세 이상 신호 모두 **단일 파일만 봐서는 안 보이고, 파일 간 교차 대조에서만 드러나도록** 설계했습니다.
2. **정산 MSA 도메인 경험의 자연스러운 전이입니다.** 복식부기 원장·대사(reconciliation)·이벤트 정합성을 다뤄 온 경험을, "숫자 사이의 관계가 안 맞는 지점"을 찾는 회계 리스크 추론으로 옮겼습니다. GL Reconciler 패턴(break list → root-cause trace → 서명용 exception report)을 CEO 리스크 도메인으로 포팅한 것도 같은 맥락입니다.
3. **경영진 의사결정의 실제 pain point입니다.** 경영진이 놓치는 문제는 숫자가 없어서가 아니라 **숫자 사이의 관계를 늦게 읽기 때문**에 발생합니다. "매출이 얼마인가?"가 아니라 "그 매출이 실제 현금으로 전환되는가?"를 먼저 묻는 것이 실제 손실을 줄입니다.

## Q3. 플러그인은 어떻게 작동하나요?

불변식 게이트 + 7개 스킬로 구성된 오케스트레이션 구조입니다. (헥사고날의 책임 분리 원칙을 스킬 경계에 적용)

```
사용자 요청: "데이터에서 CEO가 놓친 리스크를 찾아 근거와 함께 설명해줘"
        │
        ▼
[불변식 게이트]  bin/verify-books.mjs
  - aging↔시산표 대사·원가 배부 검산 등 7종 기계 검증
  - GATE PASS 여야 추론 진입, FAIL 은 그 자체를 데이터 품질 리스크로 보고
        │
        ▼
[ceo-risk-recon]  오케스트레이터
  - 데이터 인벤토리·grain 파악, 전문 스킬 실행 순서 결정
        │
        ▼
전문 분석 스킬 (책임 분리)
  1. accounting-anomaly   회계처리 이상 (매출·채권·계약부채·현금흐름 정규화-매칭-버킷-가설)
  2. cashflow-bottleneck  현금흐름 병목 (DSO/CCC + 거래처 집중 판별 테스트)
  3. cost-allocation-audit 원가 배분 왜곡 (배부 기준 민감도 재계산)
  4. macro-exposure        거시 노출 (ECOS 금리·환율·물가 × 차입/원가 민감도)
  5. disclosure-crosscheck 공시 대사 (DART 공시 ↔ 내부 장부 + 공시 행간 신호)
        │
        ▼
교차 검증 Layer
  - 여러 데이터에서 같은 방향 신호가 수렴하는지 확인
  - 금액 영향·확신도·시급성으로 우선순위 산정
        │
        ▼
[ceo-briefing]  서명용 브리핑 (결론 → 근거 → 왜 문제인가 → 확신도 → 권고 조치)
```

**설계 4원칙 (anthropics/financial-services GL Reconciler 패턴 포팅, Apache 2.0)**

0. **불변식 게이트 선행** — 대사 일치·검산·비중 합 100% 처럼 기계로 확정 가능한 정합성을 `verify-books` 로 먼저 확정하고, LLM 추론은 GATE PASS 위에서만 시작. ("불변식으로 확정할 수 있는 것을 먼저 기계적으로 확정하고, 그 위에서만 AI 추론이 의미를 갖는다")
1. **산출물 3단 계약** — break list → root-cause trace → 서명용 exception report.
2. **가설-판별 테스트 구조** — 원인 분류를 "결론이 아니라 가설"로 태깅하고 판별 테스트를 명시. (예: "수익 조기 인식 **가능성**" + "기말 전후 전표·계약 진행률 확인 필요")
3. **가드레일** — 외부 문서는 신뢰하지 않는 데이터, 분석은 **읽기 전용**(`capabilities: ["Read"]`), 실행 조치는 사람 승인 밖. 향후 실데이터는 ERP/DW를 직접 수정하지 않고 읽기 전용 MCP로만 수집.

**입출력** — 입력은 `trial_balance.csv`(분기별 매출·채권·재고·계약부채·영업이익·영업현금흐름·변동금리 차입·이자비용), `ar_aging.csv`(분기×거래처별 채권 aging, 3개 분기 — 추세 판별 테스트 실행 가능), `cost_allocation.csv`(제품별 원가·배부기준·실제 기계시간). 출력은 CEO Briefing 마크다운. PoC는 CSV지만 구조는 ERP·DW·감사시스템·MCP 서버로 확장 가능하도록 설계했습니다.

## Q4. AI를 어떻게 활용했나요?

두 층위에서 활용했습니다.

**(1) 런타임 — 에이전트의 '추론' 능력이 핵심 엔진**

규칙 매칭이 아니라 LLM 추론으로 동작합니다.

- **행간 교차 대조**: 매출 +11%인데 매출채권 +47%, 계약부채 급감, 영업현금흐름 흑자→적자 — 세 신호를 한 파일이 아니라 **여러 파일에 걸쳐 하나의 가설("팔았다고 기록했지만 현금이 안 들어오는 매출")로 수렴**시키는 것은 추론입니다.
- **가설-판별 구조의 자연어 근거 생성**: 단정("분식이다")하지 않고 "가능성 + 확인해야 할 데이터"로 표현합니다. 확신도(가능성 높음/확인됨)를 분리해 제시하는 것이 대시보드와의 결정적 차이입니다.
- **결론 뒤집기**: 제품C가 현재 흑자(+20)로 보이지만 기계시간 기준 재배부 시 -880 적자로 뒤집힌다는 것을 **민감도 재계산 + 인과 설명**으로 도출합니다.

**(2) 빌드 — AI 코딩 도구로 제작**

Claude Code / Codex를 사용해 7개 스킬(SKILL.md), 불변식 게이트(`verify-books.mjs`)와 브리핑 자동 채점기(`briefing-eval.mjs`), 교차 검증되도록 설계된 샘플 데이터, 그리고 심사·검증용 **정답지(심어둔 이상 신호 + 기대 추론)** 를 설계·구현했습니다. 즉 AI를 "산출물의 추론 엔진"이자 "제작 도구"로 동시에 활용했습니다.

## Q5. 어떻게 검증했나요?

**정답지(ground truth)를 의도적으로 심어 두고, 에이전트가 그것을 재발견하는지로 검증하는 구조**를 설계했습니다. 단, 정답지는 채점기 코드에 하드코딩되어 있지 않습니다 — 신호 판정과 근거 수치는 파생 엔진(`src/common/signals.mjs`)이 채점 시점에 분석 대상 데이터에서 다시 계산하므로, 동봉 샘플이 아닌 임의 회사 데이터(`--data-dir`)에도 같은 검증이 그대로 적용됩니다.

**(1) 심어둔 이상 신호 = 정답지** (`README.md`의 검증 표)

| # | 신호 | 데이터 근거 | 기대 추론 |
|---|------|------------|----------|
| 1 | 수익 조기 인식 의심 | 매출 +11.3% vs 매출채권 +46.9%, 계약부채 340→180, 영업현금흐름 880→-310 | 세 신호가 한 가설로 수렴 — cut-off/계약 진행률 확인 권고 |
| 2 | 특정 거래처 신용 집중 | 90일+ 채권 1,920 중 에이프릴리테일 1,750 (91.1%) | 전사 문제 아닌 거래처 1곳 — 여신·담보 확인 |
| 3 | 원가 배분 왜곡 | 제품C 매출비중 15%로 공통원가 450 배부, 실제 기계시간 45% → 재배부 1,350 | 흑자 +20 → -880 적자 전환, 가격/철수 검토 |
| 4 | 차입 의존 성장·금리 노출 | 변동금리 차입 6,000→14,500, 이자비용 90→320, 이자보상배율 23.9→8.9배 | 영업CF 악화(신호 1)를 차입으로 메우는 구조 — 고정금리 전환·헤지 확인 (ECOS 금리 추세 결합) |

**(2) 검증 설계의 핵심 3가지**

- **교차 대조로만 드러나도록 설계**: 네 신호 모두 단일 파일(또는 단일 분기)만 봐서는 정상으로 보이고, 파일·기간 간 교차 대조(행간)에서만 드러납니다 → "추론 능력"을 실제로 요구하는지 판별합니다.
- **정량 대조 기준 명시 + 기계 검증**: `ar_aging.csv`의 분기별 합계가 `trial_balance.csv`의 매출채권과 **일치**하도록 맞췄고(대사 무결성), 이런 불변식 7종은 `verify-books.mjs` 게이트가 기계적으로 검증합니다. 91.1% 집중·-880 적자전환 같은 수치는 손계산으로 재검산 가능합니다.
- **판별 테스트 동반**: 각 신호에 대해 "가능성"과 "확인해야 할 추가 데이터"를 함께 요구해, 에이전트가 단정하지 않고 감사 trail로 쓸 수 있는 근거를 남기는지 확인합니다. 단정 표현("분식입니다", "확실합니다")은 `briefing-eval.mjs` 가 기계 검출합니다.

**(3) 재현(runnable)**

데모는 아래로 재현 가능합니다.

```powershell
powershell -ExecutionPolicy Bypass -File src/bin/run-sample.ps1
```

프롬프트: *"src/data/sample 의 데이터에서 CEO가 놓치고 있는 리스크를 추출하고, 근거를 설명한 뒤 서명용 브리핑으로 정리해줘."* — 기대 결과는 위 4개 신호를 모두 포착하고 정답지의 기대 추론과 일치하는 브리핑을 생성하는 것입니다.

생성된 브리핑은 자동 채점됩니다:

```powershell
node src/bin/verify-books.mjs                # 불변식 게이트 (GATE PASS 확인)
node src/test/briefing-eval.mjs briefing.md  # 재현율 4/4 + 단정 표현 + 필수 섹션
node src/test/briefing-eval.mjs --self-test  # 채점기 자체 회귀 테스트
```

> 검증 기준 = **데이터 정합성**(불변식 게이트 7종 GATE PASS) + **재현율**(심어둔 4개 신호를 몇 개 포착하는가 — 자동 채점) + **근거 정합성**(정량 수치가 손계산과 일치하는가) + **표현 안전성**(단정 대신 가설-확신도-확인절차로 분리하는가 — 단정 표현 기계 검출).
