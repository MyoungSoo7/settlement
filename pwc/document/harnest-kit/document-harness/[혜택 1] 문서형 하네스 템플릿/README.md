# 문서형 하네스 템플릿 (md형)

## 한 줄

같은 규칙·역할·금지사항·출력 형식을 매번 설명하는 일을 문서 한 장에 고정한다. 이게 md형 — 가장 작고 가장 먼저 닿는 하네스다.

## 이 폴더의 자료

- `skill.template.md` — 본인 작업용 SKILL.md / CLAUDE.md 섹션 빈 템플릿. 7개 칸으로 구성: frontmatter, 역할, 트리거, 원칙, 금지사항, 출력 형식, 예시.
- `작성-가이드.md` — 빈 템플릿 7개 칸을 채우는 방법. 각 칸에 무엇이 들어가고 무엇이 들어가면 안 되는지.
- `예시-interview-harness.md` — 다 채워진 SKILL.md 한 장 예시. 요구사항 인터뷰 자동화를 md형으로 고정한 사례. 본인 템플릿을 채울 때 참고.
- `예시-pr-검수.md` — 또 하나의 예시. PR 검수 규칙을 md형으로 고정한 사례.

## 어디서 쓰는가

Chapter 1-2 [문서형 하네스로 규칙 고정] 와 Chapter 1-4 [하네스 선택 기준]의 md형 분지. 본인 업무에서 "매번 같은 역할·금지·형식을 설명하고 있다"가 진단이라면 이 템플릿이 첫 산출물이다.

curriculum.md 기준 핵심: **반복되는 규칙은 대화에 쓰지 말고 문서에 둔다.**

## md형이 다루는 것 / 다루지 않는 것

| 다루는 것 | 다루지 않는 것 |
|---|---|
| 역할 정의 | 메인/보조 세션 분리 |
| 원칙·금지사항 | 탐색·검수의 외부화 |
| 출력 형식 (입출력 스키마) | 단계 순서·상태·재시도 |
| 도메인 용어, 톤, 예시 | 워크플로우 강제 |

뒤 두 칸이 필요해지면 분업형(가이드 02) / MCP형(가이드 03)으로 올린다. md형 한 장에 워크플로우까지 적으려고 하면 문서가 길어지기만 하고 효과는 약해진다.

## 사용 순서

1. `작성-가이드.md`로 7개 칸의 의미를 본인 머릿속에 정렬한다.
2. `예시-interview-harness.md`와 `예시-pr-검수.md`를 훑어 채워진 모양을 본다.
3. `skill.template.md`를 본인 작업용 한 장으로 옮겨 채운다.
4. 같은 요청을 두 번 넣어 출력 형식이 일치하는지 본인이 검증한다(자가 검증).
5. 일치하지 않는 칸이 있으면 그 칸을 다시 좁힌다.

## 완료 기준

- frontmatter에 name·description·트리거 세 칸이 모두 채워져 있다.
- 역할 / 원칙 / 금지사항 / 출력 형식 네 본문 칸이 한 문장 이상씩 채워져 있다.
- 예시 입출력 한 쌍이 들어 있다.
- 같은 입력을 두 번 넣어 같은 형식의 결과가 나온다 (재현성).
