# PRD / Seed / AC — 본인 업무용 빈 템플릿

본인 업무 하나를 골라 채운다. How는 적지 않는다. 한 묶음으로 끝낸다.

---

## 1. PRD (무엇을 / 왜)

- 만들 대상:
- 이유 (어떤 결정·행동을 가능하게 하나):
- 독자 (누가 읽고 무엇을 하나):
- 범위 밖 (이번에 만들지 않는 것):

## 2. Seed (고정 계약)

```yaml
goal:
constraints:
  -
  -
acceptance_criteria:
  -
```

채우지 못한 슬롯은 빈칸 대신 `확인 필요: <물어야 할 질문>`으로 남긴다.

## 3. AC Tree (완료조건 쪼개기)

acceptance_criteria를 통과/실패로 판정 가능한 잎으로 쪼갠다.

| 완료조건 (잎) | 통과 판정 질문 | 통과 / 실패 | 사유 |
|---|---|---|---|
|  |  |  |  |
|  |  |  |  |
|  |  |  |  |

## 4. 잠금 확인

- 같은 Seed를 다시 진술했을 때 같은 What이 나오는가? (예 / 아니오)
- How가 PRD나 AC에 새어 들어왔는가? (예 → 빼낸다 / 아니오)
- 판정 불가능한 주관어(좋다·깔끔하다)가 AC에 남아 있는가? (예 → 좁힌다 / 아니오)

---

## 제출 체크리스트

- [ ] PRD에 만들 대상·이유·독자가 적혀 있고, 구현 방법(How)이 섞이지 않았다.
- [ ] Seed 세 슬롯이 모두 채워졌거나, 빈 슬롯이 "확인 필요"로 명시됐다.
- [ ] AC Tree의 각 잎이 통과/실패로 판정 가능한 문장이다.
- [ ] 같은 Seed로 같은 What이 다시 나온다(재현성).
- [ ] 외부 사례, 검증되지 않은 수치, 가격 또는 마케팅성 정보를 넣지 않았다.
