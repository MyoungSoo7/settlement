# 사용 예시 — "task 관리 CLI 만들고 싶다" 한 줄을 끝까지

설치가 끝났다는 가정. `/interview-harness`를 한 번 호출해 사이클 3번에서 수렴하는 예시 워크스루.

---

## 입력

```
task 관리 CLI 만들고 싶다.
```

## Cycle 1 — broad 사이클

### Stage 1 점검

`.claude/scratch/socrates.md`에 `## Seed` 없음 → `/socrates` 호출.

### `/socrates` 내부

#### `/wonder`

> 이 요청에서 "task"가 무엇을 의미하는지 후보를 적어주세요.

사용자 답:
```markdown
- 의미 A — todo 리스트 한 줄짜리 단순 항목
- 의미 B — 프로젝트 관리 도구의 작업 단위
- 의미 C — 일과 일정이 섞인 시간 블록
```

#### `/reflect`

> 의미 A부터 봅시다. "todo 리스트 한 줄짜리 단순 항목" — 본인이 뜻한 게 이것인가요?

사용자: A가 가깝다. B는 너무 무겁다. C는 아니다.

```markdown
- 의미 A
  - 사용자 의도 nuance: 짧고 빠르게 적고 끝나는 일
  - 모델 해석 nuance: todo 리스트 한 줄짜리 단순 항목
  - 같음 / 다름: 같음
- 의미 B
  - 같음 / 다름: 다름 (프로젝트 관리는 과함)
- 의미 C
  - 같음 / 다름: 다름 (시간 블록은 다른 도구)
```

#### `/refine`

> 의미 A를 다듬어 shared meaning 한 줄로 만들면 어떻게 되나요?

사용자 승인: 
```markdown
shared meaning: 본인 혼자 짧게 적고 끝낼 수 있는 todo 단위
```

#### `/restate`

> 이 shared meaning을 다른 사람이 그대로 실행할 수 있는 goal 한 문장으로 옮기면?

사용자 승인:
```markdown
goal: 단일 사용자가 로컬에서 todo task를 추가하고 조회하는 CLI를 만든다
```

### Seed Gate 점검

- goal ✓
- constraints? 사용자 답: Python 3.14+, 로컬만.
- acceptance_criteria? 사용자 답: `task add "..."`로 생성됨, `task list`로 조회됨.

세 칸 모두 채워짐 → Cycle 1 종료.

### Seed v1

```yaml
goal: 단일 사용자가 로컬에서 todo task를 추가하고 조회하는 CLI를 만든다
constraints:
  - Python 3.14+
  - 로컬 파일 저장만 (외부 동기화 없음)
acceptance_criteria:
  - `task add "..."` 명령으로 task가 생성된다
  - `task list` 명령으로 task 목록이 출력된다
```

## Cycle 2 — narrow 사이클

### `/evolve-step`

#### Q1. 놓친 사용자 시각

```markdown
- accepted — 우선순위가 필요한가? (모바일에서 잘 안 보면 의미가 없는 task가 섞임)
- rejected — 협업자와 공유는?
- rejected — 알람/리마인더는?
```

#### Q2. 잘려 나간 가능성

```markdown
- rejected — 다른 도구 동기화 (Notion, Linear 등)
- accepted — 완료 처리 (단순 추가/조회만으로는 끝났는지 알 수 없음)
- rejected — 카테고리/태그
```

### Seed v2 (`socrates.md`의 `## Seed` 교체)

```yaml
goal: 단일 사용자가 로컬에서 todo task를 추가하고 조회하는 CLI를 만든다
constraints:
  - Python 3.14+
  - 로컬 파일 저장만 (외부 동기화 없음)
acceptance_criteria:
  - `task add "..."` 명령으로 task가 생성된다
  - `task list` 명령으로 task 목록이 출력된다
  - 각 task는 우선순위(높음/보통/낮음)를 가진다
  - `task done <id>` 명령으로 task가 완료 표시된다
```

### `/ontology`

> 이 idea("...todo task를 추가하고 조회하는 CLI")의 의미는 어디서 시작해 어디서 끝나는가?

사용자: 한 사용자의 todo 관리까지. 협업·동기화·시간 블록은 밖.

> properties 3개를 적어주세요.

```yaml
ontology:
  idea: 단일 사용자가 로컬에서 todo task를 추가하고 조회하는 CLI를 만든다
  boundary: 한 사용자의 로컬 todo 관리까지. 협업·동기화·시간 블록은 밖.
  properties:
    - task
    - 우선순위
    - 완료 상태
```

### similarity 비교

Cycle 1에는 직전 ontology가 없으니 비교 skip → Cycle 2에서 처음 ontology가 생김. 다음 사이클에서 비교 시작.

## Cycle 3 — 안정화

### `/evolve-step`

#### Q1. 놓친 사용자 시각

```markdown
- rejected — task에 마감일?
- rejected — 검색 기능?
- rejected — task 편집?
```

#### Q2. 잘려 나간 가능성

```markdown
- rejected — 통계
- rejected — export 기능
- rejected — 다중 워크스페이스
```

채택 0개 → Seed v3 = Seed v2. Transferred meaning = `No transferred meaning`.

### `/ontology`

사용자가 boundary와 properties를 똑같이 답함:

```yaml
ontology:
  idea: 단일 사용자가 로컬에서 todo task를 추가하고 조회하는 CLI를 만든다
  boundary: 한 사용자의 로컬 todo 관리까지. 협업·동기화·시간 블록은 밖.
  properties:
    - task
    - 우선순위
    - 완료 상태
```

### similarity 비교

`.claude/scratch/interview-harness.md`의 `## Previous Ontology` (Cycle 2)와 현재 (Cycle 3) 비교.

| 항목 | 비교 결과 | 점수 |
|---|---|---|
| idea | Seed.goal 그대로 동일 | 1.0 |
| boundary | 사용자에게 물음: "Cycle 2와 Cycle 3의 boundary가 같은 뜻인가요?" → 예 | 1.0 |
| properties | {task, 우선순위, 완료 상태} ∩ {task, 우선순위, 완료 상태} / ∪ = 3/3 | 1.0 |

**평균 = 1.00 ≥ 0.85 → 수렴, 종료.**

## 이 워크스루에서 봐야 할 것

1. **Cycle 1**의 `/socrates` 질문은 **broad**다 — "task가 무엇을 의미하나요?" 같은 발산 질문.
2. **Cycle 2**의 `/evolve-step`은 사각지대 6개 후보를 모델이 *제안*만 한다. 채택은 사용자가 한다. 이 예시에서는 6개 중 2개만 채택됐다.
3. **Cycle 3**에서 채택이 0개가 되고 boundary/properties가 안정되며 similarity가 1.00에 도달했다 — 자연 수렴.
4. **socrates.md의 `## Seed`는 매번 교체**됐다 — v1 → v2 → v2(채택 0). canonical 한 벌만 유지.
5. **evolve-step.md는 누적**됐다 — Cycle 1, 2, 3의 audit가 모두 남아 있어 어떤 의사결정이 어디서 일어났는지 추적 가능.
